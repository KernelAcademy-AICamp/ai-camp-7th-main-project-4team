엔진 강화 (진단 로그 → 개선 작업목록)

진단 로그를 엔진 강화 액션으로 되돌린다(docs/6 §1.1 성장 구조). 자동 학습이 아니라 사람이 실행할 우선순위를 뽑는다 — ① 미표기∩페인(병목부위 끼임)을 모아 브랜드 실측을 A축 역산 보정, ② 자유서술의 새 페인을 PAIN_FLAGS로 승격, ③ 확신했는데 틀린 응답으로 규칙 튜닝, ④ 수요 대비 데이터 부족 브랜드 수집.

① 미표기 ∩ 페인 → A축 역산 보정 후보 — 사이즈표에 없는 병목부위인데 자주 "끼임" = 그 브랜드 실측을 거꾸로 보정할 신호
② 새 페인 후보 (레지스트리 승격) — 자유서술에서 반복되는 불편 = PAIN_FLAGS 정식 부위로 승격 검토(엔진이 커짐)
③ 규칙 튜닝 우선순위 — 엔진이 확신(high)했는데 틀린 유형 = 캘리브레이션 위험, engine.js 밴드 튜닝 1순위
④ 데이터 확장 우선 — 사용자가 실제 입은 브랜드(수요) 대비 실측 보유 — 부족하면 수집
⑤ 역산 미사용 착용감 → 엔진 배선 후보 — "더 자세한 착용감"으로 모으는 배·소매·총장 등. 지금은 수집만 되고 역산(engine.js CAT_PARTS)에 안 씀. 응답이 쏠리면 역산 부위로 승격 검토
⑥ 기장 선호(lengthPrefs) 분포 — 사용자가 고른 기장 느낌(짧음/딱좋음/긺). 추천 기장 힌트로 승격 후보
