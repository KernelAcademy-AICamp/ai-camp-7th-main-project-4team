회원·거래 운영 (3.B)

회원(진단 사용자)·전문가 심사는 화면 구현. 화면 우선 — 샘플 데이터로 렌더하며 실회원 저장소·인증 배선은 마일스톤1(Supabase). 매칭·분쟁·정산·리뷰(3.B.3~6)는 실매칭 오픈(v2) 대상.

회원 — 진단 이력 기준. 상태 변경은 프로토타입(비영속)
전문가(스타일리스트) 심사 대기열 — pro-signup 신청 → 자격 게이트. 승인/반려는 프로토타입

페이크도어(스타일리스트찾기)+견적요청이 수집하는 매칭 수요 신호. 실매칭 오픈(v2) 전에 어떤 서비스·상황·예산의 수요가 있는지를 검증하는 게 목적. 소스: localStorage fitting.reqs(+샘플). 매칭 개입·분쟁·정산·리뷰(3.B.4~6)는 실매칭 오픈(v2) 대상.

서비스 유형 수요 — 온라인/동행/이미지 중 어디에 수요가 몰리나
상황(Occasion) 수요 — 어떤 상황에 스타일링을 찾나(전문분야 우선순위)
예산대 분포 — 오픈 견적(역경매) 기준. 가격 수용대 검증
요청 로그 — 최신순. 오픈=역경매·지명=특정스타일리스트·알림=오픈알림 신청(순수 수요)

에스크로 보관금 기준 중재 큐. 소비자 신고(fitting.reqs)·전문가 소명(fitting.pro.reqs)의 분쟁 건을 모아 릴리스/환불을 판정 · 신고(어뷰징)는 별도 큐. 실 에스크로·PG 연동은 v2 — 조치 시 해당 거래 상태를 되써요(데모).

분쟁 큐 — 에스크로 보관금 기준 중재(환불 승인/기각)
신고 큐 — 어뷰징·부적절 콘텐츠·노쇼 → 회원 제재(3.B.2) 연계

거래 진행 모니터링 — 미응답·유찰 개입, 완료 거래 에스크로 정산 집행, 신고 후기 모더레이션. 실매칭·정산 집행은 v2 · 조치는 데모. 소스: fitting.reqs·fitting.pro.reqs(+샘플).

매칭 모니터링·개입 — 미응답·유찰·어뷰징(3.B.3)
정산 집행 — 완료 거래 에스크로 릴리스·수수료(3.B.5)
리뷰 모더레이션 — 신고·부적절 후기 숨김(3.B.6)

고객 문의(1:1) 처리 — 소비자 고객센터(fitting.support) 티켓을 유형·상태로 관리, 답변 전송 + 거래 문의는 분쟁 큐(3.B.4) 전환. 반복 문의는 FAQ(G.2) 승격. 실 티켓·SLA는 v2.

문의 티켓 큐 — 유형·상태 · 답변/분쟁 전환
FAQ 관리 — 반복 문의를 도움말로 승격(자가해결율↑)
