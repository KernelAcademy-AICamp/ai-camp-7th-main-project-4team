# 핏팅 — 정책 · 데이터 정의서

> **이 문서는 무엇?** 동의·가입·시점 관리 같은 **정책**과, 서비스가 저장하는 **데이터 모델**을 정의합니다. 엔진이 다루는 스키마(A축·B축)는 [6_사이즈엔진.md](6_사이즈엔진.md), 매칭 도메인 데이터는 [7_전문가매칭.md](7_전문가매칭.md) §9. 용어는 [0_용어집.md](0_용어집.md).

---

## 1. 동의 · 수집 정책

> **한 줄 원칙:** 진단은 동의 없이, 엔진 개선만 동의(opt-in). 게이트로 막지 않는다.

| 항목 | 정책 | 근거 |
|---|---|---|
| **진단 이용** | **동의 게이트 없이 시작.** 진단(체형·사이즈 계산)은 서비스 제공에 필요한 최소 처리라 별도 동의 없이 제공. | 구 G.5 필수 진입 게이트 **폐기** ([의사결정기록.md](의사결정기록.md)) |
| **엔진 개선 활용** | 진단 데이터를 **엔진 학습·개선에 쓰는 것만** 결과화면(2-5)에서 **opt-in**으로 받음. 선택 안 해도 진단·결과는 그대로 이용 가능. | 서비스 제공 근거와 개선 목적을 분리 |
| **만 14세 확인** | 별도 게이트가 아니라 **엔진 개선 opt-in에 접목**해 확인. | 게이트 최소화 |
| **비회원 수집** | 비회원도 진단 제공에 **필요한 최소**를 수집(성별·키·몸무게·착용경험 등). 비로그인이 곧 익명은 아님. | 표본 극대화 |
| **민감정보 고지** | 체형·신체치수는 민감정보임을 개인정보 처리방침(G.4)에 명시. | 한국 서비스 법적 필수 |

> ⚠️ 결제·정산·법적 문구는 실제 정책/법률 검토가 선결. 위는 프로토타입 기준 설계.

## 2. 가입 시점 정책 (lazy registration)

본선(핵심 흐름)은 **비회원 그대로** 끝까지 갑니다. 가입은 아래 두 지점에서만 요청합니다.

| 가입 지점 | 트리거 | 비회원은 | 가입하면 |
|---|---|---|---|
| **① 결과 저장** (2-4) | "결과 저장하기" | 임시 링크로 잠깐 보관(만료) | 계정에 **영구 보관** |
| **② 전문가 매칭 실제 이용** (3-1) | "요청/알림신청" | 구경·유형 선택까지 | **가입 관문** 통과 후 진행 |

- 소셜(카카오·네이버·구글) + 이메일. 가입 시 약관·만14세 동의.
- 화면 흐름상 위치는 [3_사용자플로우.md](3_사용자플로우.md) §3-2.

## 3. 시점 버저닝 (재진단·체형 변화)

> **왜 필요한가요?** 사람 몸은 변합니다(체중 등). 재진단할 때 과거 기록을 덮어쓰면 "예전 결과"와 "정확도 학습용 데이터"가 사라집니다.

- 진단은 **덮어쓰지 않고 계속 쌓습니다**(append-only). 체형은 "언제 기준"인지 날짜를 붙여 버전으로 둡니다.
- 그래서 ① 과거 결과 보존 ② 정확도 학습 데이터 무오염 ③ 체형 변화 추이를 자산으로 축적 — 세 가지를 동시에 얻습니다.
- 결과 화면에 "어느 시점(몸무게·날짜) 기준"인지 표시하고, 체중이 바뀌면 "새로 진단(과거는 보존)" 경로를 엽니다. 옷장(카테고리) 재사용은 체형이 같을 때만 유효.
- 엔진 쪽 시점 처리는 [6_사이즈엔진.md](6_사이즈엔진.md) §3-[8].

## 4. 데이터 모델 (진단 코어)

> **왜 처음부터 깔끔히?** 여기 쌓이는 데이터가 곧 "브랜드 교차 사이즈 데이터셋"이고, 이것이 나중에 회사의 가장 큰 자산(B2B API, 어드민 D)이 됩니다.

```
user                누구 · 성별 · 나이 · 키 · 몸무게 · 선호핏(옷 종류별)

garment_experience  누구 · 브랜드 · 옷종류 · 입은사이즈 ·
                    느낌{ 부위(어깨·가슴·소매·배·총장 / 허리·엉덩이·허벅지·밑위·기장)
                          : 큼|여유|딱맞음|끼임 }           ← 핏팅만의 핵심 데이터

result_card         누구 · 체형(8유형 코드, 아직 미확정일 수 있음) ·
                    추천[{ 브랜드 · 옷종류 · 추천사이즈 · 신뢰도 · 근거 }]

validation          누구 · 브랜드 · 판정(맞음|애매|틀림) ·
                    실제사이즈 · 다른브랜드도전의향(예/아니오)   ← 킬 메트릭의 원천

fakedoor_event      누구 · 관심서비스 · 누른시각 ·
                    알림신청여부 · 연락처                       ← 전환신호의 원천
```

- `result_card`의 체형은 상의+하의를 다 하면 8유형으로 확정, 상의만 하면 미확정(상체 부분).
- `garment_experience`는 §3대로 덮어쓰지 않고 시점별로 쌓입니다.
- 데이터 구조의 1순위 축은 "옷 종류(category)". 엔진이 다루는 상세 스키마(sizespec·wearexperience 등)는 [6_사이즈엔진.md](6_사이즈엔진.md) §1.
- 매칭(전문가·요청·정산 등) 데이터 모델은 [7_전문가매칭.md](7_전문가매칭.md) §9.
