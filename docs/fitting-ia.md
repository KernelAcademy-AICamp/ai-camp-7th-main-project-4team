# 핏팅(Fitting) — 화면 리스트 & 정보구조(IA)

> ★ = MVP 실제 구현 / ○ = 목업
> 데이터 중심축(브랜드 실측 DB → 사이즈 번역 엔진 → 피드백 로그)이 세 영역을 모두 관통한다.

---

## 1. 전체 구조 (3 영역 + 데이터 축)

```
핏팅
├─ ① 사용자 앱 (User-facing)
│   ├─ 랜딩 ★
│   ├─ 진단 플로우 ★ ……………… MVP 핵심
│   └─ 전문가 매칭 (페이크도어 ★ → 목록·요청·결제 ○)
│
├─ ② 전문가 앱 (Expert-facing) ○ ……… 양면 플랫폼 증명
│   ├─ 지원·심사
│   ├─ 대시보드 · 매칭 요청 상세(차별점 핵심)
│   └─ 추천 작성 · 정산
│
├─ ③ 운영 어드민 (Admin) ○
│   ├─ 브랜드 데이터 관리
│   ├─ 메트릭 대시보드
│   └─ B2B 사이즈 API 소개
│
└─ ◆ 데이터 중심축 (백엔드, 모든 면 공유)
    브랜드 실측 DB → 사이즈 번역 엔진 ★ → 피드백 로그(킬 메트릭)
```

## 2. IA 트리 (뎁스 구조)

```
1. 사용자 앱
  1.0 랜딩 ★
      1.1   기본 정보 입력 ★
        1.1.1   착용 경험 보정 입력 ★  [진단 대상 카테고리 선택(상의★ / 하의·아우터·원피스·치마 ○) → 같은-종류 1~3벌 가변]
                                       [state: 진행표시·이전수정 / 옷없음(0벌)=건너뛰기 / 입력점검 분기(1벌 종료·더 넣기) / "옷 목록에 없음" 대안입력]
          1.1.1.1  진단 로딩 ★
          1.1.1.2  결과 카드 ★  [대상 카테고리 기준 · 다른 종류 진단 다리]
                                [state: 신뢰도 0벌 저신뢰/1벌 보통/2~3벌 정확 · 추정실패 / 재진단 / 결과저장→가입유도 ①]
          1.1.1.3  피드백 수집 ★
          1.1.1.4  결과 → 전문가 전환 ★   ← 진단(무료)→전문가(유료) 다리
          1.1.1.5  결과 공유 ○
      1.2   전문가 매칭 진입 (페이크도어) ★  [state: 요청·알림신청 시 가입 게이트 ②]
        1.2.1   매칭 알림 완료 / 준비중 ★   ← 페이크도어 착지
        1.2.2   전문가 목록 ○
          1.2.2.1  전문가 프로필 ○
          1.2.2.2  매칭 요청 ○
          1.2.2.3  추천 결과 ○
          1.2.2.4  결제 / 예약 ○
      1.3   로그인 / 회원가입 ○   ← lazy registration: 비회원 진단 후 ①②에서 연결
      1.4   마이페이지 ○   ← 가입 시 결과 영구 귀속

  공통 (Global)
      G.1  서비스 소개 / 작동 원리 ★
      G.2  FAQ ○
      G.3  서비스 이용약관 ★          ← 한국 법적 필수
      G.4  개인정보 처리방침 ★        ← 한국 법적 필수, 민감정보 명시
      G.5  개인정보 수집·이용 동의 ★  ← 1.1 입력 진입 게이트
      G.6  알림 ○                     ← 양면 공통(사용자·전문가)
      G.7  404 / 에러 ○

2. 전문가 앱  (대부분 목업, 서비스 상품 관리만 ★)
  2.0 전문가 안내 / 혜택 ○             ← 공급자 랜딩(수수료·수익 안내)
      2.0.1   전문가 지원 / 등록 ○
        2.0.2   심사 / 승인 대기 ○
        2.0.3   정산 계좌·세금 정보 등록 ○
  2.1 전문가 대시보드 ○                ← 가용상태 토글
      2.1.1   매칭 요청 상세 ○  ← 차별점 핵심 / 수락·거절
        2.1.1.1  고객 메시지 / 상담 ○   ← 스타일링의 본질
        2.1.1.2  추천 / 결과 작성 ○     ← 유형별 결과물
  2.2 서비스 상품 관리 ★               ← 3종 서비스 전시(메뉴판)+가격
  2.3 프로필 / 포트폴리오 관리 ○        ← 노출 통제
  2.4 리뷰 / 평점 관리 ○               ← 평판
  2.5 수익 / 정산 ○                    ← 정산 + 수익 리포트

   서비스 유형 3종: 온라인 스타일링 / 동행 쇼핑 / 이미지 컨설팅
   (전문가가 2.2에서 전시 → 사용자가 1.2 선택·1.2.2 탐색)

3. 운영 어드민  (목업)
  [A] 사이즈 엔진 / 데이터 ← 핏팅의 심장
      3.A.1  브랜드·상품 사이즈 DB 관리
      3.A.2  엔진 정확도 모니터링
      3.A.3  번역 규칙 / 파라미터 관리
      3.A.4  수집 파이프라인 상태
  [B] 회원·거래 운영
      3.B.1  전문가 승인 / 심사 큐
      3.B.2  회원 관리
      3.B.3  분쟁·환불·신고 처리
      3.B.4  정산 집행 · 수수료 정책
      3.B.5  리뷰 관리
  [C] 사업 현황
      3.C.1  운영 대시보드 (GMV·매출)
      3.C.2  제품 메트릭 대시보드 (킬 메트릭)
  [D] B2B API 사업 (v3) ← 데이터 사업화
      3.D.1  API 고객(쇼핑몰) 관리
      3.D.2  API 키 발급 / 관리
      3.D.3  API 사용량·과금·매출
      3.D.4  API 소개 (외부용)
```

## 3. 화면 리스트 (상세)

### ① 사용자 앱

| 번호 | 화면 | 구분 | 실행 / 구성요소 | 비고 |
|------|------|:----:|----------------|------|
| 1.0 | 랜딩 | ★ | 히어로(서비스 소개 + 진단 시작 CTA) · 상황별 전문가 리스트(모집/예고 모드) · 작동 방식 3스텝 · 헤더(로그인·전문가 지원) | 전문가 0명 시 '오픈예정/1기 모집' |
| 1.1 | 기본 정보 입력 | ★ | 성별·나이 · 키·몸무게 · 선호핏(스키니~오버) | 진단 1단계 |
| 1.1.1 | 착용 경험 보정 입력 | ★ | **진단 대상 카테고리 선택** → 같은-종류 옷 [브랜드+핏+사이즈] **1~3벌 가변** · **카테고리별 부위** 느낌(상의=어깨·가슴·소매·배·총장 / 하의=허리·엉덩이·허벅지·밑위·기장 … → 큼/여유/딱맞음/끼임) | 진단은 카테고리 단위, 역산이 부위를 못 넘음 |
| └ state | · 진단 대상 카테고리 선택 | ★ | 상의 작동 / 하의·아우터·원피스·치마 "곧"(부위 미리보기) | 카테고리 단위 진단의 진입점 |
| └ state | · 진행 표시 + 이전 단계 수정 | ★ | 스텝 인디케이터 · 뒤로/수정 | 입력 길어 이탈 방지 |
| └ state | · 입력 점검 분기(1벌 종료 / 더 넣기) | ★ | "1벌로 진단" vs "핏 다른 옷 추가" · 신뢰도 3단계 안내 | 한 벌만 기억나는 사용자 이탈 방지 |
| └ state | · 옷 목록에 없음(대안 입력) | ★ | "직접 입력" / "해당 없음" 분기 | MVP 브랜드 N개 한정이라 빈발 |
| 1.1.1.1 | 진단 로딩 | ★ | 엔진 처리 대기(체형 역산 + 매칭) | LLM 호출 시 수 초 |
| 1.1.1.2 | 결과 카드 | ★ | ① 체형 프로필(캐릭터) ② 핏 인사이트 ③ **(대상 카테고리) 브랜드별 추천 사이즈** · **진단 현황(옷장)→이어서 진단** | 제품의 얼굴 · 상의 기준 · 공유 가능 URL · 리텐션 루프 |
| └ state | · 신뢰도 단계(0벌 저신뢰 / 1벌 보통 / 2~3벌 정확) | ★ | 0벌=기본추정 warn / 1벌="한 벌 더" 안내 / 2벌+ 배너 없음 | 1벌 허용하되 정확도 숨기지 않음 |
| └ state | · 추정 실패 | ★ | "옷을 더 추가해주세요" 안내 | empty/error state |
| └ state | · 재진단 / 입력 수정 후 재시도 | ★ | 결과 불만족 시 되돌리기 | 이탈 회수 |
| └ state | · 결과 저장하기 → 가입 유도 | ★ | 비회원=URL 임시보존 / 가입 시 영구보존 | lazy registration 전환 트리거 ① |
| 1.1.1.3 | 피드백 수집 | ★ | '실제 내 사이즈와 맞나?' 동의/비동의 · '안 사본 브랜드도 사보겠다' 의향 | 킬 메트릭 측정 (≥65%) |
| 1.1.1.4 | 결과 → 전문가 전환 | ★ | "스타일까지 전문가에게 받기" CTA | 무료→유료 퍼널 핵심 다리 |
| 1.1.1.5 | 결과 공유 | ○ | 카드 이미지 / 링크 공유 | 바이럴 후크 |
| 1.2 | 전문가 매칭 진입 | ★ | '전문가에게 직접 받기' 버튼 · 관심 서비스 선택(온라인 스타일링/동행 쇼핑/이미지 컨설팅) · (초기) 오픈 알림 신청 + 1기 전문가 모집 | 페이크도어, 유형별 전환신호(≥25%) |
| └ state | · 요청/알림신청 시 가입 게이트 | ★ | 둘러보기는 비회원, 실제 이용 시 가입 | lazy registration 전환 트리거 ② |
| 1.2.1 | 매칭 알림 완료 / 준비중 | ★ | "오픈 시 알려드릴게요" 확인 화면 | 페이크도어 착지 |
| 1.2.2 | 전문가 목록 | ○ | 서비스 유형 탭(온라인 스타일링·동행 쇼핑·이미지 컨설팅) · 상황별 필터(소개팅·면접·결혼식·여행) · 전문가 카드(사진·전문분야·평점·가격) | v2 실동작 |
| 1.2.2.1 | 전문가 프로필 | ○ | 경력 · 전문분야 · 제공 서비스/가격 | |
| 1.2.2.2 | 매칭 요청 | ○ | 서비스 유형 선택 → 유형별 폼 분기(온라인: 취향·예산 / 오프라인: 일정·지역) | 유형별 요청 폼 |
| 1.2.2.3 | 추천 / 예약 결과 | ○ | 온라인: 스타일·상품·구매링크 / 오프라인: 예약 확정·세션 안내 | 유형별 결과물 |
| 1.2.2.4 | 결제 / 예약 | ○ | 매칭 결제 · 일정 | |
| 1.3 | 로그인 / 회원가입 | ○ | 계정 인증 · 진단 후 결과저장·매칭 시점에 진입 | lazy registration — 비회원 진단 후 전환 트리거에서 연결 |
| 1.4 | 마이페이지 | ○ | **진단 이력(시점별 불변 스냅샷)** · 체형 변화 추이(체중·버전) · 카테고리별 최신 결과 · 받은 추천 | v2, 가입 시 결과 영구 귀속 · 옷장의 영구 거처 |

## 4. 설계 포인트 (제출/면접용)

- **MVP 선별 기준** — 전체 ~26개 화면 중 진단 플로우 7개(★)를 우선 구현. 근거는 킬 메트릭(사이즈 번역 정확도). "전체를 설계하되, 가장 검증이 급한 곳을 먼저 만든다"는 우선순위 판단.
- **카테고리 단위 진단(설계 척추)** — 진단을 옷 종류(상의·하의·아우터·원피스·치마)별로 분리. 근거는 UI가 아니라 엔진: 역산이 신체 부위를 넘지 못해(상의→어깨·가슴, 하의→허리·엉덩이) 입력·부위·여유매핑·출력·킬 메트릭이 모두 카테고리 축을 가진다. 스키마엔 `category`를 1급 축으로 넣되 **작동은 상의 1종부터**(브랜드축 × 카테고리축을 동시에 벌리지 않음). "상의부터"가 임의 축소가 아니라 엔진에서 따라오는 스코핑임을 보여주는 카드.
- **입력 부담 차등(1~3벌 + 신뢰도 3단계)** — 착용 경험을 2~3벌로 강제하지 않고 0/1/2+벌을 모두 허용하되, 신뢰도를 *0벌=기본추정 · 1벌=보통 · 2~3벌=정확*으로 차등 표기하고 "한 벌 더" 경로를 항상 연다. 한 벌만 기억나는 사용자의 이탈과, 킬 메트릭(번역 정확도) 과신 방지를 동시에 잡는 절충.
- **멀티-카테고리 리텐션 루프(진단 현황·옷장)** — 카테고리 단위 진단의 약점("또 처음부터?")을 강점으로 뒤집는다. 체형 프로필(B)이 신체 단위라 카테고리 중립이므로 **기본 정보·체형 추정은 재사용** → 두 번째 종류부터 *옷만* 새로 넣으면 끝. 결과 카드에 종류별 완료 현황(옷장)을 두어 원탭 "이어서 진단"으로 잇고, 완료 표시로 "옷장 채우기" 동기를 만든다. 장기적으로 1.4 마이페이지가 이 옷장의 영구 거처. *한 번의 기본 정보 입력 → N개 카테고리 진단*으로 LTV를 키우는 구조.
- **시점 버저닝(재방문·체형 변화 대비)** — 진단을 **불변 스냅샷(append-only)**으로 적립하고 체형은 **효력일자 버전**으로 둔다. 체중이 바뀌어 재진단해도 과거 진단·피드백을 덮지 않고 새 스냅샷을 쌓아, ① 과거 결과 보존 ② 킬 메트릭/학습 데이터 무오염 ③ 체형 변화 추이(1.4) 자산화를 동시에 얻는다. 결과 화면은 *어느 체형 기준*인지(몸무게·시점)를 표기하고, 체중 변화 시 "새로 진단(과거는 이력 보존)" 경로를 연다. 카테고리 재사용도 체형 동일 시에만 유효. 시간이 곧 데이터 해자가 되는 설계(같은 사람의 체형 변화 × 브랜드 핏 이력).
- **2.1.1 매칭 요청 상세** — 사용자가 받은 사이즈 번역 결과가 전문가의 추천 입력으로 재활용되는 지점. 차별점("도구가 전문가 정확도를 높이는 인프라")이 화면으로 구현됨. 숨고·크몽식 단순 매칭과 갈리는 곳.
- **전문가 리스트 빈 상태(empty state)** — 초기 전문가 0명일 때 "오픈 예정/1기 모집"으로 노출. 같은 화면이 사용자에겐 수요 측정(알림 신청), 전문가에겐 모집 공고로 동시에 작동. 양면 콜드스타트 해법.
- **데이터 중심축** — 사이즈 번역 엔진 하나가 사용자(안심)·전문가(정확도)·B2B(API 상품)를 모두 떠받침. "데이터가 해자"라는 비즈니스 논리가 IA에 반영됨.
- **사용자 여정 완결성 검토(보강분)** — 기능 단위로 쌓은 화면을 사용자 목적 달성 여정으로 되짚어 빈틈을 메움: ① 입력 진행표시·이전수정 ② 옷 목록에 없을 때 대안입력 ③ 재진단 ④ 추정실패 상태 ⑤ 페이크도어 착지(1.2.1) ⑥ **결과→전문가 전환(1.1.1.4)** ⑦ 법적 고지(G.3 이용약관·G.4 개인정보 처리방침·G.5 수집·이용 동의) ⑧ 404. 상태성(①~④)은 기존 화면의 state로, 독립 화면(⑤~⑧)은 정식 번호로 추가. 특히 1.1.1.4는 무료 진단 → 유료 전문가 전환 퍼널의 핵심 다리이자 페이크도어 클릭률(≥25%) 측정의 정확도를 좌우. 한국 서비스 필수인 약관·처리방침은 고정 페이지로, 수집·이용 동의는 민감정보(체형·신체치수)를 받기 직전인 1.1 입력 진입에 동의 절차로 결합.
- **Lazy registration (지연 가입)** — 가입은 진단 *뒤로* 미룬다. 비회원도 랜딩~진단~결과~피드백을 모두 이용(진단 표본 손실 방지·킬 메트릭 측정 극대화). 가입이 필요해지는 두 전환 트리거에서만 회원가입으로 연결: ① 결과 카드에서 "결과 저장하기"(비회원=URL 임시보존 → 가입 시 영구보존), ② 전문가 매칭 실이용(요청·알림신청) 시 가입 게이트. 가치를 경험한 뒤 가입을 유도하므로 전환율↑. 가입 시 결과는 마이페이지(1.4)에 영구 귀속.
- **전문가 앱 사업자 관점 검토(보강분)** — 매칭·추천·정산의 핵심 워크플로우만 있던 전문가 앱을 "공급자도 사업자"라는 관점으로 보강: ⓐ 전문가 안내/혜택(합류 전 수수료·수익 확인) ⓑ 정산 계좌·세금 등록(돈 받을 준비) ⓒ 프로필/포트폴리오 관리(노출 통제) ⓓ 요청 수락/거절 + 가용상태(일감 취사선택) ⓔ 고객 메시지/상담(스타일링의 본질=대화) ⓕ 수익 통계/리포트(상황별 수익 전략) ⓖ 리뷰/평점 관리(평판=일감) ⓗ 알림(공통). 특히 ⓔ 메시지·ⓖ 리뷰는 빠지면 양면 서비스가 논리적으로 성립 안 하는 필수, ⓐ·ⓕ는 "공급자를 사업자로 이해한다"를 보여주는 카드.
- **서비스 유형 3종 (상품 모델)** — 전문가 서비스를 시장 표준 명칭으로 3종 정의: 온라인 스타일링(비대면 상품·코디 추천) / 동행 쇼핑(오프라인 동행) / 이미지 컨설팅(오프라인 체형·퍼스널컬러 세션). "퍼스널 쇼퍼"는 서비스명이 아닌 전문가 직함으로 사용. 전문가가 2.2(서비스 상품 관리)에서 메뉴판처럼 전시·가격 책정 → 사용자가 1.2 페이크도어에서 관심 유형 선택(유형별 전환신호 측정) → 1.2.2 목록에서 유형 탭으로 탐색 → 유형별 요청 폼·결과물 분기. "전시·선택"은 가벼워 MVP(★)에 일부 포함, "실제 거래 이행"(예약·결제)은 v2 목업. 단순 매칭 앱이 아니라 스타일링 서비스 마켓플레이스로 포지셔닝하는 골격.
- **운영 어드민 4영역 재구성** — "데이터 관리 도구"에 그쳤던 어드민을 양면 마켓플레이스 운영 도구로 확장, 4영역으로 구조화. [A] 사이즈 엔진/데이터 — 핏팅의 심장이므로 가장 깊게: DB 관리뿐 아니라 정확도 모니터링(어디서 틀리는지)·규칙 튜닝·수집 파이프라인. [B] 회원·거래 운영 — 전문가 승인·분쟁/환불·정산 집행·수수료 정책(마켓플레이스가 돌아가는 골격). [C] 사업 현황 — 운영 대시보드(GMV·수수료 매출)와 제품 메트릭(킬 메트릭) 분리. [D] B2B API 사업 — 소개 페이지를 넘어 고객·키·사용량·매출 추적까지(데이터 해자를 실제 매출로 전환). 특히 3.A.2(엔진 정확도 모니터링)는 "킬 메트릭을 측정만이 아니라 개선 지점까지 운영", 3.D.3(API 매출 추적)은 "데이터 해자를 매출로 전환하는 경로"를 보여주는 카드.
