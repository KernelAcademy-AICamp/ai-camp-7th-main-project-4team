# 핏팅(Fitting) — 빌드 플랜 & 구현 맥락

> **무엇부터, 왜 이렇게 만드는가.** 개발 순서(Phase 0~3) + 구현 중 판단이 흔들리지 않게 할 맥락.
> 원칙은 하나 — **이 서비스의 심장은 사이즈 번역 엔진이고, 그 정확도가 검증 안 되면 나머지는 무의미하다.**
> (구 `fitting-build-plan.md` + `claude-code-kickoff.md` 통합)

---

## 0. 검증된 것 / 검증 안 된 것 (구현 전 반드시 인지)

**검증된 가정 (이 위에서 구현)**
- 사람마다 "맞는 브랜드"가 따로 있고, 사용자들도 이를 경험적으로 안다.
- 막힘의 이유는 스타일 40% / 사이즈 38%(설문). 사이즈는 정답이 있어 도구화 가능.
- 기존 몰의 사이즈 추천은 *그 몰 안 + 구매 이력 있을 때만* 작동 → 몰 교차 구간이 빈자리.

**아직 검증 안 된 핵심 가정 (= 이 MVP가 답하려는 것)**
- ⚠️ **착용 경험으로 체형을 역산해 브랜드별 사이즈를 맞힐 수 있는가.** ← 서비스의 심장. 이게 안 되면 전부 무너짐.
- 사람들이 "안 사본 브랜드도 이 추천이면 사겠다"고 실제로 움직이는가.

> 이 서비스의 가치는 **번역 정확도 하나**에 걸려 있다. 화려한 UI보다 ③ 브랜드별 사이즈 출력의 정확도 검증 루프를 우선한다.

---

## 1. 의존성 원칙 (왜 이 순서인가)

```
데이터(A) → 엔진(B) → UI(C)
A 브랜드 실측 DB = 엔진의 연료
B 사이즈 번역 엔진 = 제품의 심장
C MVP 프론트 = 엔진을 검증하는 측정기
```

UI를 먼저 예쁘게 만들고 싶은 유혹을 누른다. 엔진이 틀리면 UI는 의미 없다. 풀스택을 다 만든 뒤 "엔진이 틀렸다"를 알면 매몰비용이 크므로, **코드 최소 상태에서 가설의 심장부터 친다.**

## Phase 0 — 엔진 타당성 실험 (가장 먼저, 코드 최소)

**목적:** "착용 경험으로 브랜드별 사이즈를 맞힐 수 있는가"를 제품 만들기 전에 확인.

- **카테고리 1종(상의)으로 고정**해 수직 관통 — 진단은 카테고리 단위라 섞지 않는다(하의는 부위가 달라 별도 진단).
- 브랜드 3개(예: 유니클로·무신사스탠다드·스파오)의 **상의** 사이즈표를 **수동으로** 스프레드시트/JSON에 입력.
- 본인+지인 5~10명의 "입어본 **상의** + 부위별 느낌" 수집.
- 역산 로직을 스크립트(단독 .ts/.py)로 돌려 추천 사이즈 vs 실제 사이즈 대조.
- **판정:** 정답률/오차(±1 사이즈)가 의미 있게 나오는가?
  - 나오면 → Phase 1로.
  - 안 나오면 → **제품 만들지 말고 엔진 로직부터 재설계** (여기서 거르는 게 가장 싼 실패).

> 산출물: `experiments/engine-feasibility/` — 시드 데이터 + 역산 스크립트 + 결과 리포트. Next.js 앱이 아니라 독립 스크립트로 충분.

## Phase 1 — 데이터 시드 & 스키마 (백엔드 기반)

**목적:** 엔진이 참조할 정규화된 사이즈 사전 + 데이터 모델 확정.

이 서비스는 두 데이터의 매칭이다. 가장 먼저 골격을 잡아야 할 부분.

- **A. 브랜드 실측 데이터 (시드로 준비):** `브랜드 × 카테고리 × 핏라인 × 사이즈 × 부위별 실측치(cm)`. 초기 N개 브랜드의 공개 사이즈표를 **통일 스키마로 정규화**. **부위 집합은 카테고리에 종속**(상의의 어깨 ≠ 하의의 밑위).
- **B. 사용자 체형 프로필 (역산 결과):** 착용 경험 입력 → A를 참조해 부위별 추정 치수·여유 선호를 역산. 체형은 신체 단위라 카테고리-중립이지만 **역산에 쓰인 입력은 카테고리에 묶인다**. 뒤집으면 **체형 프로필·기본 정보는 카테고리 간 재사용** → 두 번째 종류부턴 옷 경험만 새로 받으면 되고 입력 비용이 급감(멀티-카테고리 리텐션의 엔진적 근거). 단 **재사용은 체형이 그대로일 때만** — 체중이 바뀌면 끊는다.
- **매칭:** B를 A의 **같은 카테고리** 다른 브랜드에 투영 → "이 브랜드(이 종류)에선 당신 사이즈는 ○○".

> 양방향 학습: 사용자 착용 경험은 B(체형)뿐 아니라 A(브랜드가 실제로 어떻게 떨어지는가)도 교정한다. 데이터 구조를 단방향으로 잠그지 말 것.

### 1-1. 스키마 정의 (가장 먼저 — 나중에 못 바꾸는 뼈대)

**`category`를 1급 축으로 처음부터 넣는다.**

- `Category(상의/하의/아우터/원피스/치마)` + 카테고리별 **부위 집합 정의**(상의=어깨·가슴·소매·배·총장 / 하의=허리·엉덩이·허벅지·밑위·기장 / …).
- `Brand`, `Product(카테고리×핏라인)`, `SizeSpec(브랜드×카테고리×핏×사이즈×부위별 cm)`.
- `BodyProfileVersion(역산 결과: 부위별 추정치·여유 선호 — **효력일자(as_of) 단위 버전**)` ← 체중·체형 바뀌면 새 버전, 과거 진단은 옛 버전을 계속 참조.
- `Diagnosis(**불변 스냅샷·append-only**: as_of·그 시점 몸무게·대상 카테고리·쓴 착용경험[]·body_version_id·추천[])` ← 재진단은 새 레코드, 과거를 수정 안 함.
- `WearExperience(브랜드·핏·사이즈·부위별 느낌 + weight_context(시점·몸무게))` ← B 역산엔 현재 체형과 일치할 때만, A 보정엔 항상 유효.
- `FeedbackLog(맞음/틀림 + 의향, 카테고리·body_version 동반)` ← **피드백 로그 스키마를 지금 같이 설계** (나중에 못 끼움).

> 데이터·작동은 상의만 채우되, 스키마/부위 테이블은 5종을 모두 표현할 수 있게 잠그지 않는다. **시점 버저닝을 처음부터** — 재방문·체중 변화는 나중에 못 끼우는 뼈대(불변 스냅샷 + 체형 버전). 정확도 분석도 체형 버전별로 분절해 체중 변화 노이즈가 엔진 정확도 신호를 오염시키지 않게 한다.

### 1-2. 시드 데이터 + 변환 규칙

1. **시드 데이터** — Phase 0의 수동 데이터를 정식 스키마로. **상의** + 브랜드 3~5개로 시작.
2. **변환 규칙 모듈**(§4 참조) — 인체↔의류 치수 변환, 여유분 계산, 주관평가(큼/여유/딱맞음/불편)↔cm 매핑. **매핑 테이블 키 = `(category, part)`** — 부위·종류마다 cm 폭이 다르므로.

> 크롤링은 나중. 지금은 수동/반자동 입력. 유니클로 1개를 수직 관통해 끝까지 만든 뒤 확장.

## Phase 2 — 번역 엔진 (핵심 로직)

**목적:** Phase 0 로직을 서비스 코드(API)로. (판단 로직 상세는 `engine/reverse-inference.md`)

1. **역산** — 착용 경험 입력 → A(브랜드 실측) 참조 → 사용자 부위별 추정치·여유 선호 산출.
2. **매칭** — 산출된 체형을 A의 다른 브랜드에 투영 → "이 브랜드에선 당신 사이즈 ○○".
3. **혼합 구조** — 사이즈 계산은 **규칙 기반**(결정론·검증가능), 자연어 인사이트·체형 캐릭터 서술은 **LLM**(Claude API). 초기엔 사이즈코리아 평균을 디폴트로, 착용 경험이 들어오면 그 값으로 교정.
4. **단위 테스트** — Phase 0 케이스를 테스트로 고정해 정확도 회귀 방지.

> 출력: ① 체형 프로필(캐릭터) ② 핏 인사이트 ③ (대상 카테고리의) 브랜드별 추천 사이즈. 역산·매칭은 모두 **대상 카테고리 안에서만** 수행(상의 입력 → 상의 사이즈). 카테고리를 넘는 추론은 하지 않는다.

## Phase 3 — MVP 프론트 + 검증 루프 (측정기)

**목적:** 킬 메트릭을 실제로 잴 수 있는 최소 제품. 구현 화면은 IA의 ★(MVP)만 (화면·흐름 상세는 `screens.md`).

1. **진단 플로우** (단일 멀티스텝 · **카테고리 단위**)
   - 1.1 기본 정보 입력 → 1.1.1 착용 경험 보정: **① 진단 대상 카테고리 선택**(상의 작동, 하의·아우터·원피스·치마는 곧) → **② 같은-종류 옷 입력(1~3벌 가변)**.
   - 입력 state: 진행표시·이전수정 / 옷없음(0벌)=건너뛰기 / **입력 점검 분기**(1벌로 끝낼지·핏 다른 옷 더 넣을지).
   - 1.1.1.1 로딩 → 1.1.1.2 결과 카드(**대상 카테고리 기준** · "다른 종류 진단" 다리).
   - 결과 신뢰도 state: **0벌=기본 추정(저신뢰) / 1벌=보통 / 2~3벌=정확** · 추정실패.
   - **멀티-카테고리 리텐션:** 결과 화면 **진단 현황(옷장)** → 다른 종류 원탭 이어가기. **기본 정보·체형 프로필 재사용**(1.1 건너뜀) → 옷 경험만 새로 입력.
2. **검증 수집** — 1.1.1.3 피드백 수집 ← **킬 메트릭(≥65%) 측정. 이게 없으면 MVP가 아니다.**
3. **전환 다리 + 페이크도어** — 1.1.1.4 결과→전문가 전환 CTA → 1.2 페이크도어(유형 선택·클릭 로깅) → 1.2.1 알림 완료.
4. **랜딩 & 법적 고지** — 1.0 랜딩 · G.1 소개 · G.3~G.5 약관·방침·동의(민감정보).
5. **배포 → 소수 사용자 → 측정** — 킬 메트릭 ≥65% · 보조(의향 ≥50% · 유료 클릭 ≥25%). 통과 → 전문가 매칭 실구현 / 미달 → Phase 2 엔진으로 회귀.

## 4. 데이터 수집·정규화 파이프라인 (A축 = 브랜드 실측 채우기)

> 핵심 인식: **진짜 과제는 "수집"이 아니라 "제각각인 소스를 한 스키마로 정규화"하는 것.** 작업의 80%가 추출·정규화에 있다.

### 4-0. 선행 규칙 정의 (수집보다 먼저 — 이게 없으면 수집해도 비교 불가)

- **인체 치수 ↔ 의류 치수 변환:** 사이즈코리아=인체 치수(둘레/길이), 브랜드 실측=의류 치수(단면/평면). 가슴 등은 `의류 단면 ×2`로 둘레 환산.
- **여유분(ease) 계산:** `의류 둘레환산 − 인체 둘레 = 그 옷이 의도한 여유`. 핏라인(슬림/레귤러/루즈)의 정체가 곧 이 여유값.
- **주관 평가 ↔ cm 매핑:** 사용자 입력("딱맞음/여유/큼")을 cm 여유 구간으로 변환하는 테이블(초기엔 가설, 피드백으로 보정). ⚠️ **카테고리·부위별로 다르다** — 매핑 테이블 키에 `(category, part)`.

### 4-1. 수집 대상 (5단계)

1. 사이즈코리아 표준체형 데이터.
2. 시장 점유율 높은 브랜드 리스트(브랜드명·컨셉·타겟 등 추천용 메타 포함).
3. 브랜드별 핏라인 리스트(시즌·옷유형 구분).
4. [브랜드+핏라인+시즌+옷유형]별 대표상품 5개의 사이즈별 실측치. ⚠️ **판매량은 비공개** → 대리지표(리뷰 수·별점·"인기순" 상위·베스트 노출)로 프록시. 스키마에 "프록시"임을 명시.
5. 표준체형 ↔ 브랜드 실측 비교 → 경향성 분석("이 핏은 이 체형에 맞음").

### 4-2. 파이프라인 구조

```
collect/    브랜드별 어댑터. robots.txt 확인 → HTML·이미지 수집 → 원본 스냅샷 저장
extract/    텍스트 파싱 + 이미지 OCR → raw 추출치 (출처 URL·시점 메타 포함)
normalize/  브랜드별 명칭·단위·측정기준 매핑 → 통일 스키마 + 4-0 변환·여유 계산
validate/   범위 체크·이상치 탐지·사람 검수용 리포트 (특히 OCR 결과는 육안 검수)
store/      검증 통과분만 적재 (신뢰도·시점 메타 동반)
analyze/    5번 경향성 분석 (표준체형 ↔ 브랜드 실측)
```

> "완전 자동 크롤러"는 환상. 실제론 **브랜드별 어댑터 + 공통 정규화** 구조. 정규화 매핑은 브랜드마다 1회 수동 규칙 작성 필요.

### 4-3. 크롤링 주의 (시작 전 확인)

- **표적은 플랫폼이 아니라 브랜드 공식몰.** 무신사·지그재그 등은 약관/robots에서 크롤링 금지 + 봇 차단 강함. 자사몰(유니클로·스파오 등)이 안전하고 1차 출처라 정확.
- 각 사이트 robots.txt 확인 → 공식 API 우선 확인 → 요청 간격(rate limit) 준수.
- **사이즈표가 이미지(JPG)인 경우 다수** → OCR 필요.
- 모든 실측치에 **출처·수집시점·신뢰도** 메타 저장. 브랜드는 시즌마다 실측을 바꾸므로 시점 메타 없으면 낡은 데이터로 오추천.

### 4-4. 진행 순서 — 1개 브랜드 수직 관통 먼저

5개 브랜드 수집기를 동시에 짜면 정규화 지옥. **유니클로 하나만** collect→extract→normalize→analyze 끝까지 관통 → 진짜 문제(이미지 사이즈표, 여유 계산 오차 등) 발견 → 그 뒤 2번째 브랜드로 어댑터 구조 일반화. **카테고리축도 동시에 늘리지 않는다** — 상의 한 종류만 먼저 관통(브랜드축 × 카테고리축을 함께 벌리면 정규화·검증 폭증). 상의가 킬 메트릭을 통과한 뒤 하의로 일반화.

### 4-5. Out of Scope (인지만, 지금은 미룸)

- 원단 신축성/소재(정확도 향상되나 수집비용 큼) → v2.
- 브랜드 간 사이즈 명칭 교차 번역표(A의 M = B의 L) → 따로 수집 아님. 5번 경향성이 쌓이면 **분석 산출물로 자동 도출**.

## 5. 검증 루프 (이게 곧 성공 기준)

★ **킬 메트릭:** 사용자가 **이미 아는 브랜드**로 추천을 돌렸을 때 "내가 실제 입는 사이즈와 맞다" 동의율 **≥ 65%**. (실구매 없이 번역 엔진 정확도를 직접 측정)

> 측정 단위도 **카테고리별**이다. MVP는 **상의 기준**으로 ≥65%를 본다 — 카테고리마다 역산 난이도가 달라 한 종류씩 통과시킨다.

보조: 진단 완료율 ≥ 60% / "안 사본 브랜드도 사보겠다" 의향 ≥ 50% / 전문가 클릭률 ≥ 25%.

> 구현에 반드시 포함: 추천 결과에 "맞다/안 맞다"를 응답하는 피드백 수집 지점. 이게 없으면 킬 메트릭을 못 잰다.

## 6. 우선순위 요약

| Phase | 만드는 것 | 통과 조건 |
|-------|----------|----------|
| 0 | 엔진 타당성 실험(스크립트) | 역산 정확도 의미 있음 |
| 1 | 스키마 + 시드 + 변환규칙 | 브랜드 3~5개 정규화 완료 |
| 2 | 번역 엔진 API + 테스트 | Phase 0 케이스 회귀 통과 |
| 3 | MVP 프론트 + 검증 루프 | 킬 메트릭 측정 가능 |

> MVP 이후(목업 단계): 전문가 매칭 실구현 → 계정/마이페이지 → 서비스 유형별 거래 → 운영 어드민 → B2B API. IA의 ○ 화면들이 이 로드맵의 뒷 단계.
