## D-1. 서비스 피벗 — 쇼퍼 연결 → 사이즈 번역 *(초기)*

- **전:** 전문 쇼퍼를 연결해 주는 서비스(구 브랜드명 *핏메이트*).

- **이유:** 새 옷 구매가 막히는 이유가 스타일 40% / 사이즈 38%로 반반인데, **사이즈는 정답이 있어 도구로 풀 수 있다**(검증 가능). 스타일은 전문가(후속)로.

## D-7. 진입 동의 게이트(구 G.5) 폐기 → 진단은 동의 없이, 엔진 개선만 opt-in

- **전:** 진단 진입 전 개인정보 수집·이용 **필수 동의 게이트(G.5)**.
- **후:** 진단은 서비스 제공 근거로 **동의 없이** 시작. **엔진 개선 활용만** 결과화면에서 **opt-in**, 만 14세 확인을 여기에 접목.
- **이유:** 필수 게이트는 진입 이탈을 키우고 표본을 깎는다. 서비스 제공(진단)과 개선 목적(학습)을 분리해, 막지 않되 개선 활용만 동의받는다.
- **영향:** [2_정보구조.md](2_정보구조.md) §0·§G, [3_사용자플로우.md](3_사용자플로우.md) §1, [5_정책·데이터.md](5_정책·데이터.md) §1. **주의:** 구 2_화면과흐름 문서가 이 게이트를 필수로 서술했던 드리프트를 재편 때 정정함.

## D-14. "환상의 궁합" 산정 기준 = 정반대 실루엣(상보성) · 2026-07-13

- **배경:** 결과 카드 맨 아래의 `🩷 환상의 궁합`(소희 설계, 카드 구조 고정 요소)은 `web/data/bodytypes.json`의 `gender.*.match` 값으로 하드코딩돼 있었으나 **왜 그 짝인지 산정 근거가 어디에도 없었다.** 바이럴 요소라 근거가 필요.
- **결정:** **환상의 궁합 = 나와 실루엣이 가장 상반된(정반대) 체형.** 8유형을 두 축 ① **상하 균형**(어깨−힙 볼륨) ② **허리 굴곡·중앙 볼륨**에 놓고, 내 좌표에서 **가장 먼 유형**을 궁합으로 본다. 서사는 *"서로에게 없는 라인을 채워주는 정반대 체형 → 정반대라서 끌린다."*
- **현재 값이 이 기준과 일치**(4쌍 모두 대칭 반대):
  - STR(직선·굴곡0) ↔ HRG(잘록한 굴곡) — 허리 굴곡 축 양극단
  - TRI(하체 볼륨) ↔ INV(상체 볼륨) — 상하 균형 축 양극단
  - BAL(표준·슬림) ↔ RND(둥근 볼륨) — 전체 볼륨 축 양극단
  - DIA(중앙 볼륨) ↔ TUB(일자 컬럼) — 중앙 볼륨 축 양극단
- **이유:** ① 근거 없는 매칭은 "왜?"에 답 못 해 바이럴 신뢰도가 떨어짐 ② 상보성(정반대) 프레임은 재미·공유 동기가 크고 실루엣 정의와도 자연 정합 ③ 4쌍이 이미 대칭이라 데이터 변경 없이 기준만 명문화하면 됨.
- **범위:** **기준 명문화까지만.** 데이터(`bodytypes.json`)·카드 마크업은 미변경. 카드에 근거 한 줄 노출(미세카피)은 마크업이 디자이너 소유라 소희와 상의 후 별건으로.
- **영향:** [4_화면정의.md](4_화면정의.md) §7(8유형 데이터), `web/data/bodytypes.json`(`match` 필드 해석 근거), `web/js/card.js`(렌더 위치).

## D-15. MVP 구현 = web/ 단일 소스 → app/ 생성 산출물(데이터 어댑터 스왑) · 2026-07-15

- **배경:** MVP는 **킬 메트릭 검증용 구현체**라 디자인·화면이 아직 미확정 → 수정이 계속 발생. 이 상태에서 `web/`(정적 프로토타입)를 `app/`(Next)로 **수기 이식하면 dual-source 드리프트**(D-11이 경계) + 흔들릴 때마다 재이식.
- **결정:** `web/`를 **단일 소스로 유지**, `app/`는 **생성 산출물(손 안 댐)**. 단 UI를 React로 **트랜스파일하지 않고**, **데이터 접근 층(어댑터)만 스왑**한다 — 프로토타입=`localStorage`/로컬 엔진 / 프로덕션=`/api/*`(서버 엔진·Supabase). 즉 `app/` = `web/` + prod 어댑터 + Next API 라우트를 생성·번들.
- **이유:** ① 디자인 미확정이라 **UI 이식은 시기상조**(트랜스파일은 컴파일러급 과잉·취약) ② 킬 메트릭 측정은 **정적 web/ + 실 백엔드면 충분**(픽셀 완성 불요) ③ **계약 3종**(DB 스키마·API shape·`tokens.css`)만 잠그면 UI가 흔들려도 재작업 최소 ④ 리포의 생성기 철학(`gen-screens`·`통합문서`)과 정합.
- **순서:** (1) `web/`에 **데이터 어댑터 심** 추출(`localStorage`·엔진 호출 → `web/js/data.js` 한 곳) (2) 얇은 백엔드 `/api/diagnose`·`/api/feedback` (3) 생성 스크립트로 `app/` 출력 (4) React 정식 이식은 **디자인 잠긴 뒤**(마일스톤1 §6 E). 엔진은 D-11/D-13대로 **TS 단일 정본**.
- **범위:** DB 스키마는 `db/00_mvp.sql`(PG15 실행 검증 완료)·`db/01`(정규화 목표, MVP 아님). MVP DB = diagnosis·feedback·kill_metric + admin(로그인·대시보드 실배선).
- **영향:** [마일스톤1-측정가능MVP.md](마일스톤1-측정가능MVP.md) §5(이관 전략)의 구체화 — "web→app 수기 이식" 대신 **"어댑터 스왑 생성"**. `web/js/**` 어댑터 리팩터, `db/`, `docs/5_정책·데이터.md`(API 계약).

## D-16. 단계 D — 진단 엔진 서버 이식(해자 보호) + 페이크도어 = 수요 수집만 · 2026-07-16

- **배경:** `app/` 배포 시 두 문제. ① `garments.json`(브랜드 실측표=해자, 공개 금지)을 클라 엔진이 진단 계산에 직접 로드 → 배포하면 실측표가 공개 다운로드 가능. ② 스타일리스트찾기·전문가 포털은 **화면만 있는 목업**이라 노출 시 "빈 껍데기" 인상.
- **결정(엔진):** garments(specs) 의존 계산만 **서버(`/api/diagnose`)로 이식**. 클라는 체형 추정(`BodyModel.estimate`)·8유형 분류(`FitBodyType`)·렌더 유지, 추정 cm+착용경험만 서버로 보내고 서버가 **역산(`bodyFromExperiences`)+추천(`recommend`)**을 계산해 `{eb, topRecs, botRecs}` 반환. `garments.json`은 **서버 번들 전용**(`api`가 require, `app/` 미배포). 입력 화면(diag-fit)은 cm-free **`size-catalog.json`**(브랜드·사이즈 라벨만, `scripts/gen-catalog.js` 생성)만 사용. `engine.js`가 이미 node 호환이라 이식 저비용.
- **결정(페이크도어=B):** 스타일리스트찾기는 **수요 수집만**. api 모드에서 견적요청·오픈알림 → `/api/lead`(`lead` 테이블, `db/02`) 저장 + "접수·오픈 시 알림"으로 종료, **목업 downstream(가짜 견적·결제·진행) 미개방**. 전문가 포털(`pro*`)·개발유틸(`_reset`)은 `app/` 화이트리스트에서 제외. admin은 숨기지 않고 **로그인+noindex 뒤 유지**.
- **결정(스타일리스트 = '준비 중 · 알림' 웨이트리스트로 통일):** api에서 스타일리스트찾기는 목업 목록/견적을 안 보이고 **모든 진입(상단 탭·홈 CTA·결과 페이지 `goExpert`→`#shop`)이 "오픈 준비 중 · 오픈 알림 신청" 웨이트리스트**로 감(`showOnly('stylistWaitlist')`). 탭은 **유지**(빼면 매칭 기대가 사라져 리텐션 손해) — 클릭 시 웨이트리스트. 수요 = **알림 신청**(`saveLead` kind=notify). 진단 후 전환은 `lead.session_id`↔`diagnosis.session_id` 조인으로 측정(같은 브라우저 세션). 인증(로그인·회원가입·My)은 측정 대상 아니라 api에서 **표면 전체 숨김** — 진단·수요 모두 로그인 불필요.
- **이유:** ① 저장된 원칙 "실측표 공개 금지" 준수하면서 킬 메트릭(진단) 유지 ② 페이크도어 본질(수요만 측정, 실제 미연결) ③ 목업 노출 제거로 배포 신뢰도.
- **배선:** `scripts/gen-app.js`(web/→app/ 큐레이트: 화이트리스트 복사·`garments.json` 제외·`FITTING_MODE='api'` 주입·pro 링크 중화·유출 가드), `scripts/gen-catalog.js`(cm-free 카탈로그), `vercel.json`(buildCommand=`gen-app`, outputDirectory=`app`). `app/`는 gitignore(빌드 재생성). `db/02_lead.sql`는 Supabase에 1회 적용 필요.
- **남은 것:** ① `db/02_lead.sql` Supabase 적용 ② `vercel dev`로 api 모드 E2E(진단→저장→피드백→lead) 확인 ③ 배포에서 Vercel nft가 `api`의 `../web/js`·`../web/data/garments.json` require를 함수에 번들하는지 검증. React 정식 이식은 여전히 디자인 잠긴 뒤(마일스톤1 §6 E).

## D-17. admin 실배선 = DB 콘솔 + 측정 2축 · MVP 메뉴 스코프 · 2026-07-16

- **배경:** D-16으로 측정 경로가 서버·DB로 옮겨지며, admin이 실데이터를 관리·측정하는 콘솔이 필요해짐. 기존 admin은 목업 뷰어(파일·샘플)였다.
- **결정(데이터 소스 = DB):** 실측표를 **`garment`/`garment_meta` 테이블**로 이관(`db/03`, RLS admin) → admin 5개 페이지가 DB 조회, **`/api/diagnose`도 garment 테이블에서 실측표 로드**(`rev` 버전 캐시로 편집 즉시 반영·`db/05`). admin-garments는 **직접 CRUD**(cm 편집·행 추가/삭제). 브랜드 **노출 순서**를 `brand` 테이블로 관리(`db/04`, 기본=실착 접근성 순), 진단 추천이 "잘 맞는 브랜드 안에서 순서대로" 반영. body 모델은 공개 기준이라 JSON 유지.
- **결정(측정 2축 = admin 뷰):** ① **진단·정확도**(킬메트릭, feedback+diagnosis RLS) + **정확도 추세**(날짜·엔진버전별). ② **전문가 수요**(`lead`, 웨이트리스트 이메일·진단→수요 전환). + **엔진 강화**(진단 로그 painFlags·openNote → 개선 작업목록, docs/6 §1.1 성장구조·자동학습 아님).
- **결정(MVP 메뉴 스코프):** admin 노출 = **사이즈·데이터 / 진단·정확도 / 엔진 강화 / 전문가 수요** 4개. v2(회원·거래·사업현황·B2B API·내부운영)는 `admin-common.js`가 네비 숨김 + `gen-app` 미배포. 인증은 Google OAuth(`is_admin`=admin_user), admin 로그아웃은 Supabase 세션까지 종료.
- **영향:** `db/03·04·05`, `web/js/admin-*`, `api/diagnose.js`(DB 소스+rev), `web/js/admin-auth.js`(garment/brand/lead/diagnoses 헬퍼), `scripts/import-garments.js`(garment·brand 시드). 자동 학습 루프·A축 검열추정은 여전히 "규모 후"(미구현) — 현재는 데이터 수집 + 수동 강화 우선순위.

## D-18. 거래 금액 라벨 = 단계별 정본 1개 · 고객·쇼퍼 동일 표기 · 2026-07-18

- **배경:** 요청~정산 흐름에서 금액 라벨이 화면마다 흔들렸다 — 같은 오픈 예산을 고객은 `예산`·쇼퍼는 `고객 예산`, 같은 지명 예상가를 `예상 가격`·`예상 98,000원`(축약) 혼용, 상담중 안내가는 `안내할 금액`. 사용자가 "금액 라이팅이 다 다르다"고 지적.
- **결정:** 금액 라벨을 **거래 단계마다 정본 하나로 고정**하고 **고객·쇼퍼 화면 동일 표기**. 오픈 요청=**예산**, 지명 요청=**예상 가격**(축약 금지), 수락 후=**확정 금액**, 상담중(쇼퍼→고객)=**결제 예정 금액**, 결제창 내역=**서비스 금액·수수료·결제 금액**, 정산=**결제 금액**. 안전결제(구 에스크로)는 UI 라벨을 **"안전결제"**로 통일(내부 개념명만 에스크로). 정본 표는 [0_용어집.md](0_용어집.md) §5.
- **이유:** ① 같은 개념을 다르게 부르면 신뢰·이해도 저하 ② 단계별로 개념이 실제 다르므로(예산 범위 vs 예상가 vs 확정가) 단계당 1개로 고정하면 충돌 없음 ③ 고객·쇼퍼가 같은 거래를 양쪽에서 보므로 동일 표기라야 거울로 맞음.
- **범위:** 라벨 문구 통일만. 금액 계산·수수료 정책은 미변경.
- **영향:** `web/js/index.js`(고객), `web/js/pro-quote.js`·`web/js/pro.js`(쇼퍼 — 개발자 소유, 알림 필요), [0_용어집.md](0_용어집.md) §5 신설.
