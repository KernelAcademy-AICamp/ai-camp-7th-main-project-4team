# 핏팅 — 의사결정 기록 (Decision Log)

> **이 문서는 무엇?** 방향을 바꾼 **주요 결정과 그 이유**를 시간순으로 남깁니다. 정본 문서(1~8) 본문에는 "지금 확정된 상태"만 두고, "왜 이렇게 바뀌었나"는 여기로 모읍니다. 그래야 정본이 changelog로 지저분해지지 않고, 새로 온 사람이 맥락을 한 번에 잡습니다.
>
> 표기: 각 항목은 **결정 → 전/후 → 이유 → 영향받는 문서**. 날짜가 확실한 것만 적고, 초기 기획 중 정해진 것은 *(초기)* 로 둡니다.

## D-1. 서비스 피벗 — 쇼퍼 연결 → 사이즈 번역 *(초기)*

- **전:** 전문 쇼퍼를 연결해 주는 서비스(구 브랜드명 *핏메이트*).
- **후:** "평소 입던 옷 기준으로 브랜드를 가로질러 내 사이즈를 번역"하는 진단 서비스. 쇼퍼 매칭은 후속(v2)으로 밀림.
- **이유:** 새 옷 구매가 막히는 이유가 스타일 40% / 사이즈 38%로 반반인데, **사이즈는 정답이 있어 도구로 풀 수 있다**(검증 가능). 스타일은 전문가(후속)로.
- **영향:** [1_제품정의.md](1_제품정의.md) 전체, [2_정보구조.md](2_정보구조.md) §0.

## D-2. 체형 분류 6종 캐릭터 → 8체형(사이즈코리아 근거) *(초기)*

- **전:** 단순 성격 캐릭터 6종.
- **후:** 국가 인체치수(사이즈코리아 8차) 근거 8체형.
- **이유:** 피벗으로 "정확도"가 핵심 가치가 되면서, 표현용 캐릭터도 실측 근거 위에 얹어야 신뢰가 선다.
- **영향:** [4_화면정의.md](4_화면정의.md) §6·§7, [6_사이즈엔진.md](6_사이즈엔진.md) §1.6.

## D-3. 성공 기준 = 킬 메트릭 하나(사이즈 정확도 ≥65%) *(초기)*

- **전:** 만족도 등 여러 지표.
- **후:** **추천 사이즈 정확도 동의율 ≥ 65%** 단일 킬 메트릭. 미달 시 엔진 교체/중단.
- **이유:** 판정 기준이 여럿이면 흔들린다. "생사 기준" 하나로 우선순위·튜닝을 통일.
- **영향:** [1_제품정의.md](1_제품정의.md) §9, [6_사이즈엔진.md](6_사이즈엔진.md) §0.

## D-4. 엔진 정책 — 숫자는 규칙, 말은 LLM *(초기)*

- **결정:** 추천 사이즈·여유 같은 **숫자는 결정론적 규칙**으로, "어깨가 넓은 편"같은 **자연어 서술만 LLM**으로.
- **이유:** 추천 숫자를 LLM에 맡기면 맞았는지 검증할 방법이 없다.
- **영향:** [2_정보구조.md](2_정보구조.md) §0-6, [6_사이즈엔진.md](6_사이즈엔진.md) §0.

## D-5. 진단 = 옷 종류(카테고리) 단위, 기반 = 상의+하의 *(초기)*

- **결정:** 상의 경험으로 상체만, 하의로 하체만 역산. 상의+하의 둘 다 끝나야 8유형 확정. 파생(아우터·원피스·치마)은 선호핏만.
- **이유:** 옷 종류마다 알 수 있는 몸 부위가 다르다. 실측 데이터 없는 카테고리는 구조적 예외로 선호핏만.
- **영향:** [4_화면정의.md](4_화면정의.md) §5, [6_사이즈엔진.md](6_사이즈엔진.md) §0.

## D-6. 매칭 = 입찰(역경매) + 지명 병행

- **결정:** 오픈 요청(전문가들이 입찰 → 사용자 낙찰) + 지명 요청(특정 전문가에게 직접) 병행.
- **이유:** 초기 공급이 얇을 때 입찰이 매칭률을 높이고, 지명은 재구매를 받친다.
- **영향:** [2_정보구조.md](2_정보구조.md) §0, [7_전문가매칭.md](7_전문가매칭.md) 전반. (3면 화면에 반영: 사용자 1.3.8 · 전문가 2.1.1 · 운영 3.B.3)

## D-7. 진입 동의 게이트(구 G.5) 폐기 → 진단은 동의 없이, 엔진 개선만 opt-in

- **전:** 진단 진입 전 개인정보 수집·이용 **필수 동의 게이트(G.5)**.
- **후:** 진단은 서비스 제공 근거로 **동의 없이** 시작. **엔진 개선 활용만** 결과화면에서 **opt-in**, 만 14세 확인을 여기에 접목.
- **이유:** 필수 게이트는 진입 이탈을 키우고 표본을 깎는다. 서비스 제공(진단)과 개선 목적(학습)을 분리해, 막지 않되 개선 활용만 동의받는다.
- **영향:** [2_정보구조.md](2_정보구조.md) §0·§G, [3_사용자플로우.md](3_사용자플로우.md) §1, [5_정책·데이터.md](5_정책·데이터.md) §1. **주의:** 구 2_화면과흐름 문서가 이 게이트를 필수로 서술했던 드리프트를 재편 때 정정함.

- **첫 마일스톤:** "측정 가능한 MVP"(1b 본선) — 진단→서버엔진→결과→**피드백 DB 저장**→집계. 계획은 [마일스톤1-측정가능MVP.md](마일스톤1-측정가능MVP.md).

## D-14. "환상의 궁합" 산정 기준 = 정반대 실루엣(상보성) · 2026-07-13

- **배경:** 결과 카드 맨 아래의 `🩷 환상의 궁합`(소희 설계, 카드 구조 고정 요소)은 `web/data/bodytypes.json`의 `gender.*.match` 값으로 하드코딩돼 있었으나 **왜 그 짝인지 산정 근거가 어디에도 없었다.** 바이럴 요소라 근거가 필요.
- **결정:** **환상의 궁합 = 나와 실루엣이 가장 상반된(정반대) 체형.** 8유형을 두 축 ① **상하 균형**(어깨−힙 볼륨) ② **허리 굴곡·중앙 볼륨**에 놓고, 내 좌표에서 **가장 먼 유형**을 궁합으로 본다. 서사는 *"서로에게 없는 라인을 채워주는 정반대 체형 → 정반대라서 끌린다."*
- **현재 값이 이 기준과 일치**(4쌍 모두 대칭 반대):
  - STR(직선·굴곡0) ↔ HRG(잘록한 굴곡) — 허리 굴곡 축 양극단
  - TRI(하체 볼륨) ↔ INV(상체 볼륨) — 상하 균형 축 양극단
  - BAL(표준·슬림) ↔ RND(둥근 볼륨) — 전체 볼륨 축 양극단
  - DIA(중앙 볼륨) ↔ TUB(일자 컬럼) — 중앙 볼륨 축 양극단
- **이유:** ① 근거 없는 매칭은 "왜?"에 답 못 해 바이럴 신뢰도가 떨어짐 ② 상보성(정반대) 프레임은 재미·공유 동기가 크고 실루엣 정의와도 자연 정합 ③ 4쌍이 이미 대칭이라 데이터 변경 없이 기준만 명문화하면 됨.
- **범위:** **기준 명문화까지만.** 데이터(`bodytypes.json`)·카드 마크업은 미변경. 카드에 근거 한 줄 노출(미세카피)은 마크업이 디자이너 소유라 소희와 상의 후 별건으로.
- **영향:** [4_화면정의.md](4_화면정의.md) §7(8유형 데이터), `web/data/bodytypes.json`(`match` 필드 해석 근거), `web/js/card.js`(렌더 위치).
