# 핏팅(Fitting) — 화면 흐름도(User Flow)

> 이 문서의 목적: IA(화면 *목록*)를 실제 **전환·분기·state 흐름**으로 확정한다.
> IA가 "무슨 화면이 있나"였다면, 이 문서는 "어떤 조건에서 어디로 가나"이다.
> 범위: **MVP(★) 진단 플로우를 끝까지**. ○(목업)은 흐름의 *착지점*으로만 표시(내부 전개는 v2).
> 표기: `★`=구현 / `○`=목업 종착 / `[state]`=같은 화면의 상태 / `◇`=분기 / `⟶`=전환

---

## 0. 흐름 설계 3원칙 (IA에서 계승)

1. **Lazy registration** — 가입은 진단 *뒤*로. 비회원이 랜딩~진단~결과~피드백을 전부 통과한다(표본 손실 방지·킬 메트릭 극대화). 가입은 두 전환 트리거에서만 요구: ① 결과 저장, ② 전문가 매칭 실이용.
2. **동의 게이트는 입력 직전** — 민감정보(체형·신체치수)를 받기 바로 전(1.1 진입)에서 G.5 동의를 받는다. 랜딩에서 미리 받지 않는다(이탈 유발).
3. **막다른 길 금지** — 모든 실패·empty state는 되돌아갈 출구를 갖는다(추정실패→옷 추가, 저신뢰→재진단, 공유URL 만료→랜딩).

## 3. 전환 트리거(가입) 상세 — Lazy Registration

흐름 본선은 **비회원 그대로** 끝까지 통과한다. 가입(1.3)은 아래 두 지점에서만 *옆으로* 분기한다.

```
① 결과 저장 트리거 (1.1.1.2 결과카드)
   비회원: 결과를 임시 URL로 보존(만료 있음)
   "결과 저장하기" ⟶ 1.3 회원가입 ⟶ 결과 영구귀속(1.4 마이페이지 ○)
   ※ 저장 안 해도 흐름은 1.1.1.3로 계속 진행 (가입은 선택)

② 매칭 실이용 트리거 (1.2 페이크도어)
   둘러보기·유형선택: 비회원 허용
   "요청/알림신청" 실행 ⟶ 가입 게이트 ⟶ 1.3 회원가입 ⟶ 1.2.1 착지
   ※ MVP는 알림신청까지가 페이크도어. 실매칭(1.2.2~)은 v2 목업
```

원칙: **가치를 경험한 뒤** 가입을 요구한다(전환율↑, 표본 손실 방지).

## 6. MVP 흐름 경계 (어디까지가 ★인가)

- **본선 종료점 = 1.2.1 알림 완료.** 그 뒤 전문가 목록·요청·결제(1.2.2~)는 ○(v2 목업), 흐름도에 미전개.
- **측정 지점 2곳**: 1.1.1.3(킬 메트릭 ≥65%) · 1.2(페이크도어 클릭 ≥25%).
- 가입(1.3)·마이페이지(1.4)·공유(1.1.1.5)는 흐름에 *연결만* 하고 내부는 v2.
