# 핏팅 — 화면 정의서

> **이 문서는 무엇?** 화면마다 **무엇을 보여주고, 상태(state)에 따라 어떻게 달라지는지**를 정의합니다. 화면 목록·URL은 [2_정보구조.md](2_정보구조.md), 화면 사이 이동은 [3_사용자플로우.md](3_사용자플로우.md), 색·말투는 [8_디자인가이드.md](8_디자인가이드.md), 용어는 [0_용어집.md](0_용어집.md).
> 진단 질문 항목의 **정본(기준)은 엔진 쪽**([6_사이즈엔진.md](6_사이즈엔진.md)) — 화면은 그 스키마가 만든 질문을 그리기만 합니다.

**표시 약속:** ★ MVP에서 실제로 만듦 · ◐ 최소만 · ○ 목업·나중(v2)

## 1. 탭1 🏠 홈 ★

> **왜 이렇게?** 처음 온 사람이 3초 안에 "이게 나한테 쓸모 있겠다"고 느끼고 곧장 진단을 시작하게 만드는 게 목표.

| 노드 | 무엇을 보여주나 | 참고 |
|---|---|---|
| 히어로 ★ | 서비스 한 줄 소개 + 큰 버튼 **[내 사이즈 진단 시작]** → 탭2로 | 서비스의 입구 |
| 상황별 쇼퍼 리스트 ★ | 소개팅·면접·결혼식·여행 등 상황별 전문가 목록 | **전문가가 아직 0명이면 "오픈예정/1기 모집"으로 표시** — 손님에겐 "알림 신청", 전문가에겐 "지원하세요" 공고로 동시 작동 |
| 작동 방식 3스텝 ★ | "이렇게 씁니다" 3단계 설명 | 신뢰를 줌 |
| 헤더 | 로그인 · **전문가 지원**(→ 전문가 앱으로) | |

## 2. 탭2 🔍 체형진단 ★ — 서비스의 심장

> **왜 여기에 힘을 쏟나요?** 킬 메트릭이 여기서 결정됩니다. 사용자가 "내 사이즈 맞네!"라고 느끼는 곳이 이 진단 결과입니다.

탭에 들어오면 **진입 동의 게이트 없이 바로** 질문을 차례로 묻는 위저드가 시작됩니다(진단은 서비스 제공에 필요한 최소 수집이라 동의 없이 시작 — 구 G.5 게이트 폐기, 정책은 [5_정책·데이터.md](5_정책·데이터.md)).

> **위저드 순서:** 기본정보(2-1) → 착용경험(2-2) → 로딩(2-3) → **결과카드(2-4)** → 피드백(2-5) → 전문가전환(2-6) → 탭3. 결과카드에서 결과공유(2-7 ○)로도 갈라집니다. (흐름 그림은 [3_사용자플로우.md](3_사용자플로우.md) §2)

| # | 화면 | MVP | 무엇을 하나 | 참고 |
|---|---|:--:|---|---|
| 2-1 | 기본 정보 입력 | ★ | 성별·나이·키·몸무게 | 성별을 고르면 평균값을 미리 채워 줌. 선호핏은 2-2에서 물음 |
| 2-2 | 착용경험 보정 | ★ | ↓ 아래 상세 참고. **질문 항목의 정본은 엔진 쪽**([6_사이즈엔진.md](6_사이즈엔진.md)) | 정확도의 핵심 |
| 2-3 | 진단 로딩 | ★ | 계산이 도는 대기 화면(역산 + 매칭 + AI 설명 문장) | 몇 초 걸릴 수 있음 |
| 2-4 | 결과 카드 | ★ | ① 체형 프로필(8유형 캐릭터) ② 핏 인사이트 ③ 브랜드별 추천 사이즈 · 옷장(진단 현황) | 제품의 얼굴. 링크로 공유 가능 |
| 2-5 | 피드백 수집 | ★ | "실제 내 사이즈와 맞나요?" · "안 사본 브랜드도 사보겠어요?" · 엔진개선 opt-in(만14세 확인) | **여기서 킬 메트릭(정확도 ≥65%) 측정** |
| 2-6 | 전문가 전환 | ★ | "스타일까지 전문가에게 받기" 버튼 → 탭3 | 무료 진단 → 유료 상담으로 넘어가는 다리 |
| 2-7 | 결과 공유 | ◐ | 카드 이미지 저장(🔖 PNG) · **친구 초대 링크 공유**(🔗 navigator.share, OG 미리보기 카드) | 입소문·유입 유도 — 링크 열면 홈에 초대 배너(`index.html?from=CODE`) |

### 2-2 착용경험 — 자세히

> **왜 옷 종류(카테고리)별로 나눠서 묻나요?** 계산 방식 때문입니다. 상의 경험으로는 상체(어깨·가슴)만, 하의로는 하체(허리·엉덩이)만 알 수 있습니다. 그래서 질문·계산·결과를 모두 옷 종류 단위로 나눕니다.

묻는 것:
- **어떤 옷 종류를 진단할지 선택** (상의/하의 등)
- **그 종류의 선호핏** (스키니·슬림·레귤러·루즈·오버 중 택1)
- **같은 종류로 입어본 옷** `[브랜드 + 핏 + 사이즈]` 를 **1~3벌**
- **부위별 느낌** → 큼 / 여유 / 딱맞음 / 끼임
  - 상의 부위: 어깨·가슴·소매·배·총장
  - 하의 부위: 허리·엉덩이·허벅지·밑위·기장

**상태(state)별 처리** — 같은 화면이 상황에 따라 다르게 보이는 경우:

| 상황(state) | 어떻게 처리하나 |
|---|---|
| 입력이 길어짐 | 진행 단계 표시 + 뒤로가서 수정 가능 → 중간 이탈 방지 |
| 입어본 옷이 0벌 | "건너뛰기" 허용(대신 결과 신뢰도는 낮게 표시) |
| 1벌만 기억남 | "1벌로 진단" vs "다른 핏 옷 더 넣기" 중 선택 + 신뢰도 안내 |
| 목록에 그 브랜드가 없음 | "직접 입력" / "해당 없음" 선택 (MVP는 브랜드가 몇 개뿐이라 자주 생김) |

### 2-4 결과 카드 — 상태별 처리

| 상황(state) | 언제 | 어떻게 보이나 |
|---|---|---|
| **신뢰도 3단계** | 입력한 옷 수에 따라 | **0벌 = 낮음**(주의 배너) / **1벌 = 보통**("한 벌 더 넣으면 정확해져요") / **2~3벌 = 정확**(배너 없음) |
| 추정 실패 | 계산이 확신을 못 할 때 | "옷을 더 추가해 주세요" → 2-2로 되돌림 |
| 재진단 | 결과가 마음에 안 들 때 | 입력을 고쳐 다시 → 2-2 |
| 결과 저장 | 저장 버튼 | 비회원이면 임시 링크로 잠깐 보관 / 가입하면 영구 보관 = **가입 유도 지점 ①** |
| 옷장 / 이어서 진단 | 계속 오게 만드는 장치 | 종류별 완료 현황(옷장)을 보여주고, 한 번 누르면 다음 종류 진단으로 이어짐. 기본정보·체형은 재사용하므로 두 번째부터는 옷만 새로 입력 |

## 4. 별도 앱 2개 (지금은 껍데기)

> **왜 별도인가요?** 사용자용 앱과 화면·로그인이 완전히 다르기 때문. 지금은 껍데기(목업)만 두고 실제 개발은 나중. **전문가 앱 전체 화면·플로우는 [7_전문가매칭.md](7_전문가매칭.md), 노드 목록은 [2_정보구조.md](2_정보구조.md) §3·§4.**

- **② 전문가 앱** — 홈 헤더 "전문가 지원"으로 진입. MVP에서 유일하게 만드는 것 = **상품 관리(E8)**(사용자에게 보일 메뉴판이라 가벼움). 핵심 화면 **E5 매칭요청 상세**에서 사이즈 번역 결과를 받는 것이 차별점.
- **③ 운영 어드민** — 4영역: **A 사이즈 엔진/데이터**(엔진의 연료·정확도 모니터링) · **B 회원·거래 운영** · **C 사업 현황** · **D B2B API(v3)**. 전부 v2 이후, 초기엔 스프레드시트로 손수 처리.

## 5. 진단 카테고리: "기반"과 "파생"

> **왜 나누나요?** 옷 종류마다 알 수 있는 정보가 다르기 때문. 상·하의는 몸을 직접 재는 "기반", 아우터·원피스·치마는 취향(선호핏)만 물으면 되는 "파생".

```iatree
★ | | 기반(基盤) | 몸(체형)을 실제로 확정하는 옷
  · 상의 + 하의 둘 다 끝나야 체형 카드 완성 (= 8유형 중 하나로 확정)
  · 상의만 끝나면 상체만 나온 "부분 카드" (8유형은 아직 미확정)
  ★ | | 상의 | 상체 역산 — 어깨·가슴·소매·배·총장
  ★ | | 하의 | 하체 역산 — 허리·엉덩이·허벅지·밑위·기장
○ | | 파생(派生) | 선호핏(취향)만 물음 · 기반이 끝나야 열림
  ○ | | 아우터 | 선호핏만
  ○ | | 원피스 | 선호핏만
  ○ | | 치마 | 선호핏만
```

- **기반 = 상의 + 하의 둘 다.** 둘 다 끝나야 체형 카드가 완성되고 8유형 중 하나로 확정. 상의만 끝나면 상체만 나온 부분 카드(8유형 미확정).
- **파생(아우터·원피스·치마)은 선호핏만** 받고, 기반이 끝나 체형이 확정된 뒤 열립니다.
- 실제 구현은 상의부터 시작(브랜드 종류와 옷 종류를 동시에 벌리지 않기 위한 좁히기).

> **지금 남은 일(엔진 쪽):** 현재 시제품은 상의만 되어 있어 **하의 진단 흐름 추가** + 하의 계산 규칙 **정식화** 필요. 상세 [6_사이즈엔진.md](6_사이즈엔진.md).

## 6. 8유형과 엔진을 잇는 문제 (결과화면 만들기 전에 별도 설계)

> **왜 미리 못박아 두나요?** 결과 카드를 그리는 방식(8유형 그림 1개)과 엔진이 계산하는 방식(상·하체 따로)이 서로 어긋나기 때문. 지금 답을 정하진 않되, "이 연결 작업이 반드시 필요하다"를 명시.

- **8유형(카드 그림)** = 몸 전체 실루엣 1개(STR / TRI / INV / HRG / BAL / DIA / RND / TUB).
- **엔진(계산)** = 상체·하체를 **따로** 계산. 상의 진단이면 상체 라인만, 하의면 하체 라인만.
- **그래서 필요한 것:** `엔진 상체 결과 × 하체 결과 → 8유형 코드` **매핑 표**. 그리고 상의만 끝났을 때 **부분 카드**를 어떻게 보여줄지 규칙.
- 이 작업은 [6_사이즈엔진.md](6_사이즈엔진.md)의 체형 결과 구조와 아래 §7의 8유형 데이터를 잇는 일. **결과화면 착수 전 별도 항목으로 설계.**

### 겹치던 부분을 어떻게 정리했나 (확정)

| 겹치던 지점 | 정리 결과 |
|---|---|
| 결과 화면 | 그림(카드)은 8유형 디자인, 그 안 **숫자·값은 엔진이 공급**. 배치는 좌측 카드 / 우측 브랜드 사이즈표·검증 |
| 진단 질문 항목 | **엔진 스키마를 기준**으로 맞춤([6_사이즈엔진.md](6_사이즈엔진.md)) |
| 전문가 영역 | 사용자 쪽 연결(2-6·3-1)은 진단 쪽에서 / 전문가 마켓·앱은 [7_전문가매칭.md](7_전문가매칭.md) |

## 7. 8유형 데이터는 딱 한 곳에서만 온다

> **왜 한 곳으로?** 8유형의 이름·색·설명이 여러 화면 코드에 흩어지면 하나 바뀔 때 전부 찾아 고쳐야 하고 실수가 납니다.

**`web/data/bodytypes.json`** 이 8유형 정보의 유일 원본(카드 표시용).

- 8유형의 **코드·이름·색·프로필·핏 설명·궁합·실루엣 좌표**는 오직 이 파일에서 읽음. 화면 코드에 다시 적지 않음.
- 8유형 코드(STR/HRG…)와 실루엣 모양 연결은 이 파일의 `silhouette` 값으로만.
- 이 파일은 "보여주기"용. 엔진 계산 결과를 8유형 코드로 바꾸는 매핑 표는 §6이 담당.
- 결과 카드는 예외적으로 어두운 배경 + 유형별 시그니처색(디자인 예외, [8_디자인가이드.md](8_디자인가이드.md) §4).
