# 핏팅 — 쇼퍼 가입(온보딩) 화면정의서 & 와이어프레임

> **이 문서는 무엇?** 메인 화면에서 **"쇼퍼 지원"을 누른 뒤, 쇼퍼(공급자)가 가입하고 이력·포트폴리오를 등록해 쇼퍼 포털로 진입**하기까지의 화면을 정의합니다. 이 흐름이 끝나면 [web/pro.html](../web/pro.html)(쇼퍼 지원 포털)로 들어갑니다.
> 상위 문맥: 화면 전체 목록은 [4_화면정의.md](4_화면정의.md), 화면 사이 이동은 [3_사용자플로우.md](3_사용자플로우.md), 전문가(쇼퍼) 매칭 정책은 [7_전문가매칭.md](7_전문가매칭.md), 색·말투는 [8_디자인가이드.md](8_디자인가이드.md).

**표시 약속:** ★ MVP에서 실제로 만듦 · ◐ 최소만 · ○ 목업·나중(v2)
**작성:** 팀장 · 담당 화면 = 쇼퍼 가입 온보딩 → 포털

---

## 0. 왜 이 화면이 필요한가

지금은 메인 우상단 **"쇼퍼 지원"**을 누르면 아무 설명·가입 절차 없이 [pro.html](../web/pro.html)로 **툭 점프**합니다([index.html:489](../web/index.html#L489)). 하지만 쇼퍼는 고객에게 **프로필이 곧 첫인상**이라, 포털에 들어가기 **전에** 자기 이력과 포트폴리오를 갖추게 해야 합니다.

> **목표:** "쇼퍼 지원" 클릭 → **가입 → 이력 작성 → 포트폴리오 등록 → 완료 → 포털 진입**을 하나의 매끄러운 위저드로 만든다. (체형진단 위저드와 같은 패턴 — [4_화면정의.md](4_화면정의.md) §2)

## 1. 정보구조(IA) — 이 기능의 위치

```
index.html (고객/메인)
   └─ [헤더] "쇼퍼 지원" 클릭
          │
          ▼
   ┌──────────────────────────────────────────┐
   │  쇼퍼 가입 온보딩 (신규 · 이번 작업 범위)   │
   │   SP-0 소개  →  SP-1 가입  →  SP-2 기본정보 │
   │        →  SP-3 이력·전문분야  →  SP-4 포트폴리오 │
   │        →  SP-5 완료                        │
   └──────────────────────────────────────────┘
          │  등록 완료
          ▼
   pro.html (쇼퍼 지원 포털 — 이미 존재)
```

- **화면 파일(안):** `web/pro-signup.html` (온보딩 위저드) + `web/js/pro-signup.js`
  → 완료 시 `pro.html`로 이동.
- **재방문 처리:** 이미 가입한 쇼퍼가 "쇼퍼 지원"을 다시 누르면 **온보딩을 건너뛰고 바로 `pro.html`** (SP-1에서 판단 — §4 상태표 참고).

## 2. 사용자 플로우(스토리보드)

각 단계는 **위저드 한 스텝**이며, 상단에 **진행 표시(1/5 …)**와 **← 뒤로**가 항상 있습니다.

```
 [index.html] 헤더 "쇼퍼 지원"
        │ 클릭
        ▼
 ┌─ SP-0 소개 ───────────────┐   "쇼퍼로 수익을 만들어보세요"
 │  · 무엇을 하는 역할인지 3줄 │   [쇼퍼 시작하기]  /  [← 고객 화면으로]
 └───────────┬───────────────┘
             ▼
 ┌─ SP-1 가입(계정) ──────────┐   이메일 · (연락처) · 약관 동의
 │  이미 가입자면 → pro.html   │   [다음]
 └───────────┬───────────────┘
             ▼
 ┌─ SP-2 기본 정보 ───────────┐   활동명 · 서비스 유형별 가격(온라인/방문)
 └───────────┬───────────────┘   [다음]
             ▼
 ┌─ SP-3 이력·전문분야 ───────┐   한 줄 소개 · 경력 · 전문분야 태그
 └───────────┬───────────────┘   [다음]
             ▼
 ┌─ SP-4 포트폴리오 ──────────┐   대표 작업 사진 여러 장 (+설명)
 │  0장이면 "나중에" 허용(◐)   │   [등록 완료]
 └───────────┬───────────────┘
             ▼
 ┌─ SP-5 완료 ────────────────┐   "쇼퍼 등록 완료! 🎉"
 └───────────┬───────────────┘   [쇼퍼 포털 시작하기] → pro.html
             ▼
        [pro.html] 대시보드
```

**되돌아가기/이탈:** 어느 스텝에서든 **← 뒤로**로 앞 스텝 수정 가능, **✕ 닫기**로 index.html 복귀(입력은 임시저장 — §4).

## 4. 화면정의서 (화면별 상세)

각 화면의 **구성요소 · 사용자 동작 · 상태(예외) 처리**를 정의합니다.

### SP-0 소개 ★
| 요소 | 무엇을 보여주나 | 동작 |
|---|---|---|
| 히어로 문구 | "내 안목으로 수익을…" + 역할 3줄 | — |
| 혜택 3칩 | 요청 받기 / 제안·코디 / 정산 | — |
| [쇼퍼 시작하기] | 주 버튼 | → SP-1 |
| [← 고객 화면으로] | 보조 링크 | → index.html |
| [✕ 닫기] | 우상단 | → index.html |

### SP-1 가입(계정) ★
| 요소 | 입력/표시 | 동작·검증 |
|---|---|---|
| 이메일 | 텍스트(필수) | 형식 검증(`@` 포함). 빈 값이면 [다음] 비활성 |
| 연락처 | 텍스트(선택) | 숫자·하이픈만 |
| 약관 동의(필수) | 체크박스 | **미동의 시 [다음] 비활성** |
| 마케팅 수신(선택) | 체크박스 | 저장만 |
| [다음] | 주 버튼 | 검증 통과 시 → SP-2 |

**상태(state) 처리**
| 상황 | 처리 |
|---|---|
| 이미 가입한 쇼퍼 | 이 화면 **건너뛰고 pro.html로 바로** 이동(SP-1 진입 시 판단) |
| 이메일 형식 오류 | 인풋 아래 빨간 안내, [다음] 비활성 |
| 약관 미동의 | [다음] 비활성 + "약관 동의가 필요해요" |

### SP-2 기본 정보 ★
| 요소 | 입력/표시 | 동작·검증 |
|---|---|---|
| 활동명 | 텍스트(필수) | 고객에게 노출되는 이름. 빈 값 불가 |
| 서비스 유형별 가격 | 유형 토글 + 가격 입력 | **온라인 스타일링 / 방문** 2종. 각 유형을 켜면 가격 입력칸 활성. 예: 온라인 35,000 · 방문 120,000. 1,000원 단위 콤마 표시 |
| [다음] | 주 버튼 | → SP-3 |

**상태(state) 처리**
| 상황 | 처리 |
|---|---|
| 켠 유형이 0개 | [다음] 비활성 + "제공할 서비스 유형을 최소 1개 켜주세요" |
| 켠 유형의 가격이 빈 값/0 | 해당 칸 빨간 안내, [다음] 비활성 |
| 화상 서비스 | **이번 범위에서 보류** — 추후 유형 추가 시 같은 구조로 확장 |

### SP-3 이력·전문분야 ★
| 요소 | 입력/표시 | 동작·검증 |
|---|---|---|
| 한 줄 소개 | 텍스트(필수) | 카드 상단에 노출 |
| 경력 소개 | 여러 줄(선택) | 프로필 "소개" 영역 |
| 전문분야 태그 | 다중 선택 칩 | **최소 1개** 선택 권장. 프리셋 + 직접추가 |
| [다음] | 주 버튼 | → SP-4 |

### SP-4 포트폴리오 ★ / ◐(업로드 최소)
| 요소 | 입력/표시 | 동작·검증 |
|---|---|---|
| 사진 그리드 | 최대 8칸(+ 추가 버튼) | 파일 선택 → 썸네일. MVP는 **로컬 미리보기만**(실업로드는 v2 ○) |
| 대표작 설명 | 사진 클릭 시 입력 | 선택 |
| [등록 완료] | 주 버튼 | 데이터 저장 → SP-5 |
| [나중에 하기] | 보조 링크 | 사진 0장이어도 진행 허용(◐) → SP-5 |

**상태(state) 처리**
| 상황 | 처리 |
|---|---|
| 사진 0장 | "나중에 하기" 허용, 프로필에 "포트폴리오 준비 중" 표시 |
| 파일이 이미지 아님 | 무시 + "이미지만 올릴 수 있어요" 토스트 |

### SP-5 완료 ★
| 요소 | 표시 | 동작 |
|---|---|---|
| 완료 메시지 | "쇼퍼 등록 완료!" + 활동명 | — |
| [쇼퍼 포털 시작하기] | 주 버튼 | → **pro.html** |

## 5. 수집 데이터 → 쇼퍼 포털 프로필 매핑 (데이터 계약)

가입에서 모은 값은 **저장소 키 `fitting.pro.profile`(localStorage)** 에 저장하고, [pro.html](../web/pro.html)/[pro.js](../web/js/pro.js)가 이를 읽어 프로필·헤더에 표시합니다. (지금 `pro.js`는 "소희 쇼퍼"가 하드코딩 — 이 계약대로 **읽어오게 바꾸는 건 다음 구현 작업**.)

| 가입 입력(SP-#) | 저장 필드 | pro 포털에서 쓰이는 곳 |
|---|---|---|
| SP-1 이메일 | `email` | (계정 식별 · 정산 v2) |
| SP-2 활동명 | `name` | 헤더 "○○ 쇼퍼", 프로필 이름 |
| SP-2 서비스 유형별 가격 | `services[]` | 프로필 "서비스·가격" 목록. 예: `[{type:'online', label:'온라인 스타일링', price:35000}, {type:'visit', label:'방문', price:120000}]` |
| (대표 가격) | `services[0].price` | 제안폼 기본값(`MY_PRICE`) = 켠 유형 중 첫 번째(대표) 가격 |
| SP-3 한 줄 소개 | `tagline` | 프로필 상단 소개 |
| SP-3 경력 소개 | `bio` | 프로필 "소개" 문단 |
| SP-3 전문분야 태그 | `specialties[]` | 프로필 "전문 분야" 태그 |
| SP-4 포트폴리오 | `portfolio[]` | 프로필 포트폴리오 갤러리(`#pgal`) |
| (내부) 가입완료 여부 | `registered: true` | SP-1 재방문 시 온보딩 건너뛰기 판단 |

> **데이터 계약이 왜 중요?** 이 표가 있으면, 화면(디자이너)·로직(개발자)·데이터(팀장)가 **같은 필드 이름**으로 각자 작업해도 아귀가 맞습니다. ([협업가이드.md](협업가이드.md)의 "한 파일 = 한 역할" 계약과 같은 취지.)
