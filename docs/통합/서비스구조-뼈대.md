# 서비스 구조 뼈대 (확정) — PRD·IA 정합

> Phase A ①: sohee(03_IA + 구버전 00~02 + 04_IA) ↔ sangmin(product.md · screens.md) 대조.
> 결론: **두 사람의 현 정본은 이미 같은 피벗 이후 문서라 방향이 일치**한다. 갈라진 건 (1) 구버전 잔재, (2) 번호 체계, (3) 진단 입력/결과 스펙의 성숙도. 아래로 뼈대를 확정한다.

---

## 1. 큰 그림 — 이미 한 번 피벗했다

두 사람이 출발한 "하나의 기획 문서"는 **핏메이트(쇼퍼 매칭 중심)**. 이후 **핏팅(사이즈 번역 중심)**으로 피벗했고, **sohee `03_IA`와 sangmin `product.md`는 둘 다 피벗 이후 = 정본**이다. sohee의 `00~02`만 피벗 이전(폐기 대상).

```
[구] 핏메이트 (sohee 00·01·02)              [정본] 핏팅 (sohee 03·04 + sangmin 전체)
 쇼퍼 매칭이 심장                     ──▶    사이즈 번역 엔진이 심장, 쇼퍼는 v2
 6체형 정성 캐릭터                    ──▶    8체형(사이즈코리아) + 착용경험 역산(정량)
 만족률 ≥70% / 매칭완료 건수          ──▶    사이즈 정확도 ≥65% (킬 메트릭)
 김지우 28·결혼식 하객룩              ──▶    김도현 33·안 입어본 브랜드 도전
```

## 2. PRD 변동점 (구 → 정본)

| 축 | 구버전 (핏메이트, sohee 00/01/02) | 정본 (핏팅, sangmin product.md) |
|---|---|---|
| 무게중심 | 쇼퍼 매칭(스타일) | **브랜드 교차 사이즈 번역 엔진** |
| 서비스명 | 핏메이트 | **핏팅** |
| 진단 방식 | 사진·자가문진 정성 판정 | **착용경험 기반 사이즈 역산(카테고리 단위, 규칙+LLM)** |
| 체형 분류 | 6체형 캐릭터 | **8체형(사이즈코리아 근거, 자체 네이밍)** |
| 페르소나 | 김지우/28, 결혼식 하객룩 | 김도현/33, 안 입어본 브랜드 도전 |
| 킬 메트릭 | "잘 입었다" 만족률 ≥70% | **"내 사이즈와 맞다" 정확도 ≥65%** |
| 보조지표 | 결제 전환·재호출 | 브랜드 도전의향 ≥50% · 유료클릭 ≥25% |
| 상황(소개팅·결혼식) | 과금축·매칭 트리거(핵심) | **v3로 이연**(상황별 카테고리·과금) |
| 쇼퍼 매칭 | MVP 핵심 | **페이크도어(전환신호만)**, 실매칭 v2 |
| 최대 리스크 | "따뜻한 리드면 쇼퍼가 들어오나" | "엔진이 실제로 정확한가"(킬 메트릭) |

→ **정본 PRD = sangmin `product.md`.** sohee `00/01/02`는 참고용 히스토리로만 보존, 뼈대에서 폐기.

## 3. IA 정합 — 두 정본은 사실상 동일

sohee `03_IA`와 sangmin `screens.md`는 **같은 4탭·같은 ★/○·같은 킬메트릭·같은 lazy 가입·같은 페이크도어**. 차이는 아래 5개뿐.

| # | 차이 | sohee 03_IA | sangmin screens.md | 정본 결정 |
|---|---|---|---|---|
| A | **번호 체계** | 탭 기반 `2-1~2-7 / 3-1~` | 계층 `1.1.1.x` | **§5 통일 번호 채택**(탭 기반) |
| B | 홈 vs 랜딩 | "🏠 홈 탭" | "1.0 랜딩" | 동일물 — **홈 탭 = 랜딩** |
| C | **선호핏 위치** | 2-1 기본정보 안 | 착용경험(카테고리 스코프)으로 이동 | **sangmin 정본** — 선호핏은 카테고리별(파생은 선호핏만) |
| D | **진단 입력/결과 성숙도** | 부위 느낌 나열 수준 | 카테고리 선택·1~3벌·신뢰도3단계·리텐션 옷장·시점 버저닝 | **sangmin 정본**(엔진 담당) |
| E | **기반 카테고리 범위** | 언급 옅음 | screens.md=상의만 작동 / **root CLAUDE=상의+하의 둘 다 있어야 체형카드 완성** | **최신=상의+하의 기반** (§6·§7 열린질문) |

→ **정본 IA 골격 = sohee 03_IA(4탭·번호), 진단 입력·결과·리텐션·버저닝 스펙 = sangmin screens.md.**

## 4. 전문가(쇼퍼찾기) 도메인 = sohee 04_IA로 확정

MVP는 페이크도어(S1 진입+유형선택 → 알림신청), 실매칭 S2~S8·전문가앱 E0~E11·어드민 O1~O5는 v2. **핵심 다리** = 진단 결과(체형·사이즈)가 매칭 요청에 자동 첨부(유일한 구조적 차별점). 두 문서 완전 일치.

## 6. 최대 통합 seam — 8유형(sohee) ↔ 엔진 upper/lower(sangmin)

두 사람이 겹치는 **결과 화면**의 뿌리 문제. 후속 설계 필요(지금 확정 X, 존재만 못박음).

- **sohee 8유형** = 전신 원형 1개(사각 STR / 삼각 TRI / 역삼각 INV / 모래시계 HRG …). 카드가 이걸 그림.
- **sangmin 엔진** = 상체·하체를 **따로** 역산. 상의 진단→상체 라인만, 하의 진단→하체 라인만. **상의+하의 둘 다 완료해야 8유형 중 하나로 확정**(root CLAUDE "상의 1종만으론 반쪽 체형"과 일치).
- **필요 산출물:** `엔진 upper × lower 조합 → 8유형 코드` **매핑 테이블**. 상의만 완료 시 = 부분 카드(상체만), 8유형 미확정 상태 표현 규칙도 필요.
- → Phase C(결과 화면 통합) 전에 이 매핑을 별도로 설계. `engine/reverse-inference.md`의 체형카드 스키마와 `진단카드_8유형.json`을 잇는 작업.

## 8. 결정 로그 (확정됨)

1. **번호 체계** — §5 **탭 기반(2-1…)으로 통일**. sangmin 프로토타입 파일명 01~10은 유지, 문서 번호만 정렬.
2. **기반 카테고리 범위** — ✅ **상의+하의 둘 다 기반**(root CLAUDE 원칙대로). 체형카드 완성 조건 = 상의+하의 둘 다 완료. 상의만 완료 = 상체 부분 카드(8유형 미확정).
   - ⚠️ **파급:** 현재 프로토타입은 상의만 구현 → **하의 착용경험 플로우 추가** 필요. 엔진도 하의 `EASE_BANDS`가 "미검증 목업용" 상태 → **하의 규칙·데이터 정식화** 필요(하의 부위: 허리·엉덩이·허벅지·밑위·기장). Phase C/D 작업량에 반영.
3. **8유형↔엔진 매핑**(§6) — ✅ **Phase C(결과화면 통합) 전 별도 설계 항목**으로 승인. `upper×lower → 8유형 코드` 매핑 테이블 + 부분(상체만) 카드 표현 규칙.
