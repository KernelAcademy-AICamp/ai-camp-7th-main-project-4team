# 핏팅 — 전문가 매칭 (쇼퍼찾기) 도메인 정의서

> **이 문서는 무엇?** 전문가 매칭(쇼퍼찾기)의 **3면 구조·화면·플로우·데이터**를 담은 도메인 정의서입니다. MVP에서 실제로 만드는 건 **페이크도어(3-1) 하나**뿐이고, 아래 마켓·전문가앱·어드민은 대부분 **v2 확장**입니다.
> 노드 목록은 [2_정보구조.md](2_정보구조.md) §3·§4, 사용자 쪽 흐름은 [3_사용자플로우.md](3_사용자플로우.md), 용어는 [0_용어집.md](0_용어집.md).

## 0. 왜 "3면"으로 나뉘는가

**배경:** 무료 진단은 사용자 한 명만 상대하면 되는 **단순한 도구**다. 하지만 전문가 매칭은 사용자와 전문가를 서로 연결하고, 그 사이에서 문제가 안 생기게 관리까지 해야 한다. 그래서 관련된 사람이 세 종류로 늘어난다(**마켓플레이스**).

**[수요] 사용자 측**(프로필·요청·결제) ──요청──▶ **[엔진/중개]**(매칭·정산·분쟁) ──전달──▶ **[공급] 전문가 측**(수신·작성·정산). 이 중개를 다시 **[운영 어드민]**(승인·중재·정산집행)이 받친다.

| 면 | 누구 | 무엇을 하나 | 어디서 |
| --- | --- | --- | --- |
| 수요(사려는 쪽) | 사용자 | 서비스 고르기·요청·상담·결제·후기 | 사용자 앱 탭3 쇼퍼찾기 |
| 공급(팔려는 쪽) | 전문가 | 요청 받기·수락·상담·결과 작성·정산 받기 | 따로 있는 전문가 앱 |
| 중개(관리하는 쪽) | 운영 어드민 | 전문가 심사·분쟁 해결·돈 정산·수수료 정하기 | 따로 있는 관리자 화면 |

**우리만의 강점(§5):** 무료 진단에서 나온 "이 사람은 이 브랜드에서 몇 사이즈"라는 결과가, 전문가에게 요청이 갈 때 **자동으로 같이 딸려간다.** 그래서 전문가는 사용자를 직접 안 만나도 옷 사이즈를 정확히 고를 수 있다. 다른 스타일링 매칭 서비스에는 없는 무기다.

```iatree
  ○ | A | 사용자 측 (수요) | 탭3 쇼퍼찾기
    ★ | S1 | 매칭 진입·서비스 유형 선택 | 지금 만드는 유일한 사용자 화면

    ○ | E6~E7 | 상담 → 추천/결과 작성 | 유형별

    ○ | O3 | 정산 집행·수수료 정책 |
    ○ | O4 | 리뷰 관리 |
    ○ | O5 | 매칭 성사율·거래액 대시보드 |
```

## 2. 사용자 측 플로우 S1~S8

**배경:** 사용자가 "전문가에게 스타일링 받고 싶다"고 마음먹은 순간부터 후기까지의 순서다. 두 가지 길로 들어온다.

- 🔥 **따뜻한 리드(warm lead)** — 진단 결과카드를 다 보고 "스타일까지 전문가에게" 버튼을 눌러 들어온 사람. 이미 자기 사이즈 정보가 있다.
- ❄️ **차가운 리드(cold lead)** — 진단 없이 홈/탭에서 바로 들어온 사람. 사이즈 정보가 없으니 "진단부터 하세요"로 안내.

**순서:** S1 진입·유형선택 → S2 목록 → S3 프로필 → S4 요청 → S5 상담 → S6 결과 → S7 결제 → S8 후기. **MVP는 S1까지**(알림 신청받고 끝), S2~S8은 v2.

| # | 화면 | 무엇을 하나 | 구분 |
| --- | --- | --- | --- |
| **S1** | 매칭 진입·유형 선택 | 서비스 유형 3종 중 고르기 · 상황(소개팅·면접·결혼식·여행) 고르기 · **(지금은)"열리면 알려주세요" 신청 + 1기 전문가 모집** · 회원가입 요구(가입 게이트②) | ★MVP |
| **S2** | 전문가 목록 | 정렬(잘 맞는 순·평점·후기·가격) · 필터(유형·상황·예산·지역·일정) · 전문가 카드(작업 결과물·전문분야·평점·가격·**매칭도 뱃지**) | v2 |
| **S3** | 전문가 프로필 | **작업 결과물 갤러리** · 후기·평점 · 경력·전문분야·상황 태그 · 제공 서비스/가격/소요시간 · 예약 가능 일정 (얼굴·팔로워 ❌, 결과물·후기 ✅) | v2 |
| **S4** | 매칭 요청 | **공통:** 내 체형·사이즈 정보 자동 첨부(§5) + 상황·날짜 / **유형별로 묻는 항목이 다름**(§6) | v2 |
| **S5** | 상담 / 메시지 | 전문가와 1:1 채팅 · 예산·취향 세부 파악 | v2 |
| **S6** | 추천 / 예약 결과 | 유형별 결과물(§6) — 온라인=코디·상품·구매링크, 오프라인=예약 확정·세션 안내 | v2 |
| **S7** | 결제 / 예약 | 온라인=매칭 수수료·패키지, 오프라인=예약금·세션비. 상황(면접·결혼식 등)이 과금 단위 | v2 |
| **S8** | 완료·후기 | 만족도·후기·평점·"또 부르고 싶다" 의향 → 다음 사용자의 S2 목록 정렬에 반영 | v2 |

## 3. 전문가 측 E0~E11 (전문가 앱)

**배경:** 전문가는 사용자와 다른 별도 앱을 쓴다. 일감을 받고, 일하고, 돈을 받는 흐름. 홈 상단 '전문가 지원'으로 E0 진입. (지금은 전문가가 적어 운영자가 직접 손으로 모집.)

**온보딩:** E0 랜딩 → E1 지원/등록 → E2 심사/승인 → E3 정산계좌·세금. **일감:** E4 대시보드 → **E5 요청상세 ★**(사이즈 결과 받음) → 수락 → E6 상담 → E7 결과작성. **언제든:** E8 상품관리 · E9 프로필/포트폴리오 · E10 리뷰 · E11 수익/정산.

| # | 화면 | 무엇을 하나 |
| --- | --- | --- |
| E0 | 전문가 랜딩 | 수수료·정산·기대 수익 안내 · '1기 지원' 버튼 |
| E1 | 지원/등록 | 경력·전문분야·잘하는 상황·포트폴리오 제출 |
| E2 | 심사/승인 대기 | 플랫폼이 품질을 거른다(운영자 O1과 짝). 승인 전엔 사용자에게 안 보임 |
| E3 | 정산계좌·세금 | 돈 받을 계좌·사업자/원천징수 정보 등록 |
| E4 | 대시보드 | 들어온 요청·진행 중인 일 · **"일 받을 수 있음/마감" 스위치** |
| **E5** | **매칭 요청 상세 ★강점** | 사용자 체형 정보 + **★사이즈 번역 결과를 받는 화면** · 수락/거절. 숨고·크몽 같은 단순 매칭과 갈리는 지점 |
| E6 | 상담/메시지 | 추천 전에 사용자와 대화로 취향 파악 |
| E7 | 추천/결과 작성 | 유형별(§6) — 온라인: 상품·코디·링크 / 오프라인: 예약·세션 노트 |
| **E8** | **서비스 상품 관리** | 내가 파는 서비스를 진열(메뉴판) · 유형별 범위·소요시간·가격. 사용자 S3에 이대로 보임 |
| E9 | 프로필/포트폴리오 | 사용자에게 보일 내 소개 편집(= S3의 원본) |
| E10 | 리뷰/평점 | 받은 리뷰 보기·답하기 (평판이 곧 다음 일감) |
| E11 | 수익/정산 | 정산 내역 · 어떤 유형·상황이 돈이 되는지 통계 |

**전문가 앱 중 지금(MVP) 만드는 것은 E8(상품관리)뿐이다.** 사용자에게 보일 메뉴판이라 가벼워서다. 나머지는 목업.

## 4. 운영 어드민 O1~O5 (중개)

**배경:** 사용자와 전문가만 있으면 사고가 난다(자격 미달, 노쇼, 환불 분쟁). 운영자가 중간에서 관리한다. 지금은 다 v2, 초기엔 스프레드시트로 손수(§8).

| # | 화면 | 무엇을 하나 |
| --- | --- | --- |
| O1 | 전문가 승인/심사 대기줄 | E2의 운영자 쪽 · 승인·반려로 품질을 거른다 |
| O2 | 분쟁·환불·신고 처리 | 노쇼·불만족 환불 · 신고 대응 → **믿고 쓸 수 있는 장터의 핵심** |
| O3 | 정산 집행·수수료 정책 | 전문가별 지급 · 수수료율·유형별 요금 결정 = **돈 버는 조절 손잡이** |
| O4 | 리뷰 관리 | 가짜·악성 리뷰 정리 |
| O5 | 매칭 성사율·거래액 대시보드 | 거래액·성사율·활동 중인 전문가 수 |

## 5. 핵심 다리 — 진단 결과가 전문가에게 자동으로 넘어간다 ★

**배경:** 이게 우리 서비스가 다른 스타일링 매칭과 다른 **딱 하나의 진짜 차별점**이다. 사람만 연결하는 게 아니라 "이 사람 사이즈 데이터"를 함께 넘긴다(**핸드오프**). 진단에서 만든 체형카드가 전문가 화면(E5)으로 그대로 넘어간다.

| 진단이 넘겨주는 것 (결과카드) | ──넘겨줌──▶ 전문가가 받는 것 (E5 요청상세) |
|---|---|
| 체형 프로필(캐릭터) · 핏 인사이트(주의할 부위) · 브랜드별 추천 사이즈 | 고객 체형·핏 인사이트 · **★ 브랜드별 추천 사이즈** → "안 만나도 정확히 고를 수 있다" |

- **사용자에게:** 요청할 때 키·몸무게를 다시 안 적어도 됨(귀찮음↓) → 전환↑.
- **전문가에게:** 줄자 없이 사이즈 확신 → 비대면(온라인)인데도 반품↓. 도구가 실력을 올려줌.
- **플랫폼에게:** 데이터가 쌓일수록 매칭이 정확해지고, 나중엔 B2B 사이즈 API(v3) 자산.

기술적으로 요청 데이터에 `body_profile_ref`(진단 결과 참조)가 붙는다. 진단은 **불변 스냅샷**으로 저장되므로, 요청한 시점의 체형이 그대로 고정돼 넘어간다.

## 6. 서비스 유형 3종과 유형별 분기

**배경:** 전문가가 파는 서비스는 3종류인데, 종류마다 **요청 항목(S4)·결과물(S6)·과금(S7)이 다르다.** ("퍼스널 쇼퍼"는 서비스 이름이 아니라 전문가 직함.)

| 항목 | 💻 온라인 스타일링 | 🛍️ 동행 쇼핑 | ✨ 이미지 컨설팅 |
| --- | --- | --- | --- |
| **요청 폼(S4)** | 취향·예산·좋아하는 브랜드 | 일정·지역·만날 장소 | 목적(면접 등)·일정·지역 |
| **진행 방식(S5·S6)** | 안 만나고 채팅으로 큐레이션 | 오프라인에서 같이 쇼핑 | 오프라인 세션 |
| **결과물(S6)** | 코디·상품·**구매링크** | 현장 구매·세션 노트 | 이미지 가이드·세션 노트 |
| **과금(S7)** | 매칭 수수료/패키지 | 예약금 + 세션비 | 세션비 |
| **사이즈 결과 중요도** | ★가장 중요(원격 구매) | 보조(현장 시착) | 보조(체형 이해용) |

흐름: 전문가가 E8에서 유형별로 진열·가격 → 사용자가 S1에서 관심 유형 선택(인기 유형 측정) → S2 유형 탭 탐색 → S4에서 유형에 맞는 폼.

## 7. 상태(state)와 예외 상황

**배경:** 행복한 경우만 설계하면 안 된다. 무응답·후보 0명·노쇼도 미리 정해 사용자가 막다른 길에 안 갇히게.

- **요청 상태 흐름:** 대기 → `수락` → 진행 → 완료 / 중간 `거절`(→ 다른 전문가 자동 추천) / `만료`(무응답)
- **매칭 실패:** 조건 맞는 전문가 0명 → "열리면 알림" 등록 · "예산·지역 넓혀 보세요"
- **취소/노쇼:** 취소 시점에 따라 환불 규칙 → O2에서 운영자 조정
- **분쟁:** 신고 → 운영자 중재 → 환불/제재
- **품질 게이트:** 심사(E2) 통과 전 전문가는 사용자에게 안 보임
- **콜드스타트(전문가 0명):** S1/홈에 "오픈 예정·1기 모집". 같은 화면이 사용자에겐 "수요 측정", 전문가에겐 "모집 공고"로 동시 작동 — 닭·달걀 문제를 한 화면으로 푼다.

## 8. 지금(MVP) vs 나중(v2)

**배경:** 처음부터 매칭을 다 만들면 돈·시간이 크게 든다. 먼저 "수요가 있는지"만 싸게 확인한다(**페이크도어** + **위저드 오브 오즈**).

| | 지금 (MVP) | 나중 (v2, 진짜 작동) |
| --- | --- | --- |
| 사용자 측 | **S1 진입+유형선택 → 페이크도어 알림신청** | S2~S8 전체 흐름 |
| 전문가 측 | 없음(운영자 손으로 1기 모집), E8만 진열용 | 전문가 앱 E0~E11 |
| 운영 | 스프레드시트로 손수 | 어드민 O1~O5 |
| 매칭 | **위저드 오브 오즈:** 알림 신청자 ↔ 1기 전문가를 사람이 카톡으로 연결 | 자동 매칭·결제·정산 |

**MVP에서 확인하려는 것:**
1. **유형별 전환신호 ≥25%** — 어떤 유형에 진짜 수요가 있는지.
2. **전문가 수락률** — 손으로 연결해 보며, 핵심 가정 *"따뜻한 리드(사이즈 정보까지 딸려오는 좋은 고객)라면 수수료가 낮아도 전문가가 들어올까?"* 를 코드 없이 확인.

## 9. 데이터 모델 (매칭 부분)

```
expert            id · 전문분야 · 상황태그[] · 평점 · 가능상태 · 심사상태
service_offering  expert_id · type(online|shopping|image) · 범위 · 소요 · price
match_request     user_id · expert_id · service_type · occasion · date
                  · body_profile_ref(진단 결과 참조 = 핸드오프) · form{유형별} · status
consultation      request_id · messages[]
result            request_id · type · items[](구매링크) | booking(예약·세션노트)
review            request_id · rating · text · 재호출의향
settlement        expert_id · request_id · amount · fee · status
```

- `match_request.body_profile_ref` = §5 핸드오프가 실제로 구현되는 지점(진단 결과를 가리켜 요청 시점에 고정). `form`은 §6 유형별 입력.
- 진단 코어 데이터 모델은 [5_정책·데이터.md](5_정책·데이터.md) §4.
