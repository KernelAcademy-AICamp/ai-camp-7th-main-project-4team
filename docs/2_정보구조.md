# 핏팅 — 정보구조 (IA) · 정본

> **이 문서는 무엇?** 화면·URL 구조·역할·범위·구현상태의 **단일 기준(SSOT)** 입니다. "어떤 화면이 있고, 어떤 URL로, 누가 쓰고, 지금 어디까지 만들었나"는 전부 여기서 정합니다.
> 화면 안에서 사용자가 **어떻게 움직이는지**(플로우)는 [3_사용자플로우.md](3_사용자플로우.md), 화면별 상세 요소는 [4_화면정의.md](4_화면정의.md). 용어는 [0_용어집.md](0_용어집.md).

**범례(우선순위)** ● MVP · ◐ 일부 MVP · ▷ v2 · ○ 선택/후순위 · ▢ 목업(화면만)
**SEO 색인** 🔍 색인(공개 콘텐츠) · 🔒 noindex(개인·앱 상태)

> 이 문서는 **설계(target)** 만 담습니다. "지금 어디까지 만들었나"(구현·배포/목업/미착수)는 여기 표시하지 않고 **[README.md](README.md) "지금 상태" 한 곳**에서만 관리합니다 — 코드와 어긋나는 상태 표식을 문서 곳곳에 두지 않기 위함(우선순위 ●◐▷○는 설계 결정이라 유지).

## 0. 이 IA를 규정하는 핵심 결정

1. **3-사이드 마켓플레이스** — 사용자 · 전문가 · 운영자 + 사이즈 진단 엔진.
2. **매칭 = 입찰(역경매) + 지명 병행**
 - 오픈 요청: 사용자 요청 → 전문가들이 **입찰** → 사용자 **낙찰**
 - 지명 요청: 사용자가 특정 전문가에게 **직접** 요청 → 수락/거절
3. **진단은 동의 없이 이용**(서비스 제공 근거) · **엔진 개선만 opt-in**(결과화면). 진입 동의 게이트 없음 — *구 IA의 G.5 필수 게이트는 폐기.* 만 14세 확인은 opt-in에 접목. (근거·정책 상세는 [5_정책·데이터.md](5_정책·데이터.md))
4. **비회원도 수집**(진단 제공에 필요한 최소) · **lazy registration**(저장·매칭 시점에 가입).
5. **SEO**: 공개 콘텐츠(랜딩·소개·전문가·가이드)만 색인, 진단·결과·매칭·마이·어드민은 noindex.
6. **엔진 정책**: 사이즈 수치는 규칙 기반(검증 가능), 캐릭터·핏 인사이트 서술만 LLM. (상세 [6_사이즈엔진.md](6_사이즈엔진.md))

## 1. 내비게이션 구조 (사용자 앱)

사용자 앱은 화면 **맨 아래에 항상 떠 있는 탭 4개**로 이동합니다. 그 위에 상황에 따라 잠깐 뜨는 요소(로그인창·약관)와, 사용자 앱과 완전히 분리된 **별도 앱 2개**(전문가 앱·운영 어드민)가 있습니다.

| 탭 | 이름 | 한 줄 설명 | MVP |
|---|---|---|---|
| 탭1 | 🏠 홈 | 첫 화면(=랜딩). "이게 뭐 하는 서비스"를 3초에 보여주고 진단을 시작시킴 | ● |
| 탭2 | 🔍 체형진단 | **핏팅의 심장.** 무료로 사이즈를 진단해 주는 핵심 화면들 | ● |
| 탭3 | 👔 쇼퍼찾기 | 전문가(쇼퍼) 연결. MVP(api)에서는 **'준비 중·오픈 알림(이메일)' 웨이트리스트** — 수요만 수집(목업 목록/견적 미노출) | ●(페이크도어) |
| 탭4 | 👤 마이 | 내 진단 기록·저장·로그인 | ◐ |

> 탭에 보이는 이름은 "쇼퍼찾기", 내부 용어·코드에서는 "전문가(expert)"로 통일합니다.
>
> **MVP(api) 인증 정책:** 인증은 측정 대상이 아니고 소비자 로그인도 목업이라, **api 모드에선 인증 표면 전체를 숨긴다**(로그인·회원가입·My·알림·프로필). 상단=홈+쇼퍼찾기만, 진단·수요 수집 모두 로그인 불필요. proto(데모)는 로그인·마이·거래 목업 전부 유지. 전문가 앱(pro*)은 MVP 미배포 — 수요는 소비자 웨이트리스트로만 측정.

## 4. 운영 어드민 (Admin) — MVP 실배선 · `/admin/*` 🔒 (Supabase Auth Google + RLS)

> **MVP admin 메뉴 = 4개**: `3.A 사이즈·데이터` · `3.C.2 진단·정확도` · `3.C.4 엔진 강화` · `3.C.3 전문가 수요`. 측정 2축(진단 정확도=킬메트릭 / 매칭 수요)을 보고, 엔진 데이터를 직접 관리한다. **v2 섹션(3.B 회원·거래 / 3.C.1 사업현황 / 3.D B2B API / 3.E 내부운영)은 admin-common.js가 네비 숨김 + gen-app 미배포** — 화면은 구현돼 있으나 MVP 밖. 인증은 Google OAuth 실연결(`is_admin` = admin_user 멤버십), 데이터는 RLS(admin 전용).

- **■ 3.A 사이즈·데이터** — *해자의 실행 도구* · **DB 실배선**
 - ■ 3.A.1 브랜드·상품 사이즈 = **`garment` 테이블(DB·RLS)** — 실측표 뷰어 + **직접 CRUD**(cm 편집·행 추가/삭제, 변경 시 `rev`↑ → 진단 즉시 반영). 대량 수집은 CSV→build-sizespec.py→import. 브랜드 관리에 **진단 노출 순서 편집**(brand 테이블·실착 접근성 기본순).
 - ■ 3.A.2 수집/커버리지 모니터링 — 브랜드×카테고리 커버리지·갭. (정확도 모니터링은 3.C.2로, 미표기 부위는 3.C.4로 이관)
 - ■ 3.A.3 번역 규칙/파라미터(EASE_BANDS·역산 튜닝) — 엔진 시뮬레이터. 신체 사이즈(body-*)는 공개 기준이라 JSON 유지.
- **■ 3.C.2 진단·정확도(킬메트릭)** — 실 DB(feedback+diagnosis·RLS). 정확도×신뢰도 캘리브레이션·8유형/앵커/페인 분해 + **정확도 추세**(날짜별·엔진버전별 맞음율 → 강화 확인).
- **■ 3.C.3 전문가 수요(페이크도어 측정축)** — `lead` 테이블(RLS). 웨이트리스트 오픈알림(이메일)·견적요청 수요 + **진단→수요 전환**(session 조인) + 서비스·상황별.
- **■ 3.C.4 엔진 강화(성장구조)** — 진단 로그(painFlags·openNote+feedback) → 개선 작업목록: ① 미표기∩페인→A축 역산 실측보정 ② 새 페인→PAIN_FLAGS 승격 ③ 확신 틀림→규칙 튜닝 ④ 수요 대비 데이터 부족→수집. (docs/6 §1.1·자동학습 아님, 사람 액션 우선순위)
- *(v2 · MVP 미노출)* **▢ 3.B 회원·거래** — 심사·회원·매칭 모니터링·분쟁/환불·정산·리뷰·CS. **▢ 3.C.1 사업현황**(GMV·정산). **▢ 3.D B2B API**(v3·미표기 검열추정 해자). **○ 3.E 내부운영**(RBAC·감사로그·쿠폰).
