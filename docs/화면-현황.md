## 비고
- **오류·빈 상태(G.7)** — 전역 오류 페이지. ?e=notfound/expired/network 유형별 카피 + 홈·진단·1:1문의 진입. 오류 컨텍스트를 고객센터 문의에 프리필. Vercel 404.html 컨벤션 + serve.js 라우팅(noindex). 서버 라우팅으로 도달 — incoming 링크 없음이 정상(util=플래그 제외).
- **데모 초기화** — sessionStorage 리셋 유틸.
- **서비스 소개·작동원리** — 소개→diag-basic 전환 CTA 있음.
- **관리자 · B2B API(3.D)** — 화면 구현(v3) — API 고객·플랜·키·호출/한도·MRR + 엔드포인트 사용량(FitEngine 메서드 기준). 샘플 콘솔, 실 게이트웨이·과금은 v3.
- **관리자 · 브랜드 관리(3.A.1)** — 브랜드별 커버리지·앵커·다양성·갭(앵커인데 빈 카테고리×성별). 실측표/수집 진입점. 앵커·실측은 생성물(뷰).
- **관리자 · 사업현황(3.C)** — 3.C.1 운영 대시보드(진단→수요→매칭→전문가 퍼널 개요·KPI, 샘플). 3.C.2 진단 응답·정확도는 admin-diagnostics. 금전(GMV·정산)은 v2.
- **관리자 · 이미지 붙여넣기 수집(3.A.1)** — 사이즈표 이미지→CSV 3방법: 수동전사·OCR(베타)·AI프롬프트 브릿지(Claude/ChatGPT). CSV 가져오기·브랜드 datalist. CSV→build-sizespec.py→garments.json.
- **관리자 · 진단 응답·정확도(3.C.2)** — 진단 응답+정확도 검증(킬 메트릭) 집계 뷰어. 정확도×신뢰도 캘리브레이션·8유형/앵커브랜드/페인 분해·응답로그. 화면 우선 — 소스는 로컬 fitting.feedback+샘플, 실수집 배선은 마일스톤1.
- **관리자 · 엔진 시뮬레이터·튜닝(3.A.3)** — 몸→사이즈 파이프라인을 실엔진(BodyModel·FitEngine)으로 재현. 신체추정→추천→핏스코어 분해→사이즈 래더+목표여유 what-if→파라미터(여유밴드·목표·가중치). 튜닝 인사이트.
- **관리자 · 브랜드·제품 실측표(3.A.1)** — garments.json을 사이즈표(사이즈×부위 cm) 형태로 제품별 드릴다운. 생성물 뷰어(수정은 CSV+빌더).
- **관리자 로그인(구글+이메일2FA)** — 프로토타입 — 인증은 목업(실제 아님). 승격 시 Supabase Auth(구글)+이메일 OTP.
- **관리자 · 회원·거래(3.B)** — 회원(3.B.2)·전문가 심사(3.B.1)·매칭 수요 신호(3.B.3)·분쟁·환불·신고(3.B.4) 4탭. 분쟁 큐=소비자(fitting.reqs)·전문가(fitting.pro.reqs) 분쟁 집계+샘플 → [중재] 상세 모달(신고 사유·상세·증빙·쇼퍼 소명 확인)에서 전액/부분 환불·기각·소명 요청 판정 → 거래 상태 되씀. 매칭·정산·리뷰 모니터링(3.B.5·6, 5번째 탭): 매칭 개입(리마인드·유찰 구제)·완료 거래 정산 집행·신고 후기 모더레이션 — fitting.reqs 실데이터+샘플, 조치는 데모. 고객 문의 CS(3.B.7, 6번째 탭): 고객센터(fitting.support) 티켓 큐+샘플 → 답변 모달(답변 전송→답변완료)·거래문의 분쟁 전환·FAQ 승격. 3.B 전탭 구현.
