# 핏팅(Fitting) — Claude Code Kickoff

> 이 문서의 목적: 검증 완료된 가설을 코드 작업으로 옮기기 위한 seed.
> "무엇을 만들지"보다 "왜 이렇게 만들어야 하는지"의 맥락을 함께 전달해, 구현 중 판단이 흔들리지 않게 한다.

## 0. 한 줄 정의

새 브랜드 옷을 사고 싶어도 사이즈 확신이 없어 주저앉는 사람에게, **평소 입던 옷의 착용 경험을 기준으로 브랜드를 가로질러 내 사이즈를 번역**해주는 AI 진단 서비스.
(사이즈 번역이 심장, 스타일리스트 매칭은 후속 가설)

## 1. 검증된 것 / 검증 안 된 것 (구현 전 반드시 인지)

**검증된 가정 (이 위에서 구현)**
- 사람마다 "맞는 브랜드"가 따로 있고, 사용자들도 이를 경험적으로 안다.
- 막힘의 이유는 스타일 40% / 사이즈 38% (설문). 사이즈는 정답이 있어 도구화 가능.
- 기존 몰의 사이즈 추천은 *그 몰 안 + 구매 이력 있을 때만* 작동 → 몰 교차 구간이 빈자리.

**아직 검증 안 된 핵심 가정 (= 이 MVP가 답하려는 것)**
- ⚠️ **착용 경험으로 체형을 역산해 브랜드별 사이즈를 맞힐 수 있는가.** ← 서비스의 심장. 이게 안 되면 전부 무너짐.
- 사람들이 "안 사본 브랜드도 이 추천이면 사겠다"고 실제로 움직이는가.

> Claude Code 유의: 이 서비스의 가치는 **번역 정확도 하나**에 걸려 있다.
> 화려한 UI보다 ③ 브랜드별 사이즈 출력의 정확도 검증 루프를 우선한다.

## 2. MVP 범위

**구현 (2개)**

1. **AI 체형·사이즈 진단**
   - 입력: 성별 · 키 · 몸무게 · 나이 · 선호핏(스키니/슬림/레귤러/루즈/오버) · **착용 경험 보정**
   - 착용 경험 보정: 사용자가 입어본 옷을 [브랜드 + 핏 + 사이즈]로 고르고,
     주요 부위(목·어깨·가슴·소매길이·배·총장)를 [큼/여유/딱맞음/불편]으로 평가.
     **핏이 다른 2~3벌**을 받아 역산 자유도 확보.
   - 출력 카드: ① 체형 프로필(캐릭터) ② 핏 인사이트 ③ **주요 브랜드별 추천 사이즈**

2. **전문가 추천 받기 (페이크도어)**: 버튼 클릭만 로깅. 실매칭 없음.

**Out of Scope**
- 브랜드는 초기 N개만 (전 브랜드 X). 로그인/계정/결제/어드민 없음. 퍼스널컬러·카테고리 확장 없음.

## 3. 데이터 구조 (seed 작업의 핵심)

이 서비스는 두 데이터의 매칭이다. Claude Code가 가장 먼저 골격을 잡아야 할 부분.

- **A. 브랜드 실측 데이터 (시드로 준비)**: `브랜드 × 핏라인 × 사이즈 × 부위별 실측치(cm)`.
  초기 N개 브랜드의 공개 사이즈표를 **통일 스키마로 정규화**. 이게 매칭의 한 축.
- **B. 사용자 체형 프로필 (역산 결과)**: 착용 경험 입력 → A를 참조해 사용자 부위별 추정 치수·여유 선호를 역산.
- **매칭**: B를 A의 다른 브랜드에 투영 → "이 브랜드에선 당신 사이즈는 ○○".

> 양방향 학습 메모: 사용자 착용 경험은 B(체형)뿐 아니라 A(브랜드가 실제로 어떻게 떨어지는가)도 교정한다.
> 데이터 구조를 단방향으로 잠그지 말 것.

## 4. 진단 로직 — 결정 대기 (구현 시 먼저 정할 것)

- 체형·사이즈 추정 방식: **규칙 기반 / LLM 단독 / 혼합** 중 택.
  → 권장 출발점은 **혼합**: 실측 매칭·여유 계산은 규칙(결정론·검증가능),
    자연어 인사이트·캐릭터 서술은 LLM.
- 초기엔 사이즈코리아 평균을 디폴트로, 착용 경험이 들어오면 그 값으로 교정(개인 실측 미입력 보완).

## 5. 검증 루프 (이게 곧 성공 기준)

★ **킬 메트릭**: 사용자가 **이미 아는 브랜드**로 추천을 돌렸을 때
"내가 실제 입는 사이즈와 맞다" 동의율 **≥ 65%**. (실구매 없이 번역 엔진 정확도를 직접 측정)

보조: 진단 완료율 ≥ 60% / "안 사본 브랜드도 사보겠다" 의향 ≥ 50% / 전문가 클릭률 ≥ 25%.

> 구현에 반드시 포함: 추천 결과에 대해 사용자가 "맞다/안 맞다"를 응답할 수 있는 피드백 수집 지점.
> 이게 없으면 킬 메트릭을 못 잰다.

## 6. 첫 작업 제안 (Claude Code 진입점)

1. 데이터 스키마 확정 — A(브랜드 실측)·B(사용자 프로필)·피드백 로그.
2. 브랜드 N개(예: 유니클로·무신사스탠다드·스파오) 공개 사이즈표 정규화 → 시드 데이터.
3. 역산 + 매칭 규칙 엔진(부위별 실측 ↔ 여유 선호) 프로토타입.
4. 입력 폼(착용 경험 2~3벌) → 결과 카드(①②③) → 피드백 수집까지 한 흐름.
5. 전문가 버튼 페이크도어 + 클릭 로깅.

## 7. 데이터 수집·정규화 파이프라인 (엔진의 A축 = 브랜드 실측 채우기)

> 목적: 사이즈 번역 엔진의 학습 데이터(섹션 3의 A. 브랜드 실측) 구축.
> 핵심 인식: **진짜 과제는 "수집"이 아니라 "제각각인 소스를 한 스키마로 정규화"하는 것.** 작업의 80%가 추출·정규화에 있다.

### 7-0. 선행 규칙 정의 (수집보다 먼저)

엔진의 핵심 로직. 이게 없으면 수집해도 비교가 안 된다.
- **인체 치수 ↔ 의류 치수 변환**: 사이즈코리아=인체 치수(둘레/길이), 브랜드 실측=의류 치수(단면/평면). 가슴 등은 `의류 단면 ×2`로 둘레 환산 필요.
- **여유분(ease) 계산**: `의류 둘레환산 − 인체 둘레 = 그 옷이 의도한 여유`. 핏라인(슬림/레귤러/루즈)의 정체가 곧 이 여유값.
- **주관 평가 ↔ cm 매핑**: 사용자 입력("딱맞음/여유/큼")을 cm 여유 구간으로 변환하는 테이블(초기엔 가설, 피드백으로 보정). 사용자 입력을 받으려면 필수.

### 7-1. 수집 대상 (5단계)

1. 사이즈코리아 표준체형 데이터.
2. 시장 점유율 높은 브랜드 리스트 (브랜드명·컨셉·타겟 등 추천용 메타 포함).
3. 브랜드별 핏라인 리스트 (시즌·옷유형 구분).
4. [브랜드+핏라인+시즌+옷유형]별 대표상품 5개의 사이즈별 실측치.
   - ⚠️ **판매량은 비공개** → 대리지표(리뷰 수·별점·"인기순" 상위·베스트 노출)로 프록시. 스키마에 "프록시"임을 명시.
5. 표준체형 ↔ 브랜드 실측 비교 → 경향성 분석("이 핏은 이 체형에 맞음").

### 7-2. 파이프라인 구조

```
collect/    브랜드별 어댑터. robots.txt 확인 → HTML·이미지 수집 → 원본 스냅샷 저장
extract/    텍스트 파싱 + 이미지 OCR → raw 추출치 (출처 URL·시점 메타 포함)
normalize/  브랜드별 명칭·단위·측정기준 매핑 → 통일 스키마 + 7-0 변환·여유 계산
validate/   범위 체크·이상치 탐지·사람 검수용 리포트 (특히 OCR 결과는 육안 검수)
store/      검증 통과분만 적재 (신뢰도·시점 메타 동반)
analyze/    5번 경향성 분석 (표준체형 ↔ 브랜드 실측)
```

> "완전 자동 크롤러"는 환상. 실제론 **브랜드별 어댑터 + 공통 정규화** 구조. 정규화 매핑은 브랜드마다 1회 수동 규칙 작성 필요.

### 7-3. 크롤링 주의 (시작 전 확인)

- **표적은 플랫폼이 아니라 브랜드 공식몰.** 무신사·지그재그 등은 약관/robots에서 크롤링 금지 + 봇 차단 강함. 자사몰(유니클로·스파오 등)이 안전하고 1차 출처라 정확.
- 각 사이트 robots.txt 확인 → 공식 API 우선 확인 → 요청 간격(rate limit) 준수.
- **사이즈표가 이미지(JPG)인 경우 다수** → OCR 필요. 텍스트 크롤링만으론 안 긁힘.
- 모든 실측치에 **출처·수집시점·신뢰도** 메타 저장. 브랜드는 시즌마다 실측을 바꾸므로 시점 메타 없으면 낡은 데이터로 오추천.

### 7-4. 진행 순서 — 1개 브랜드 수직 관통 먼저

5개 브랜드 수집기를 동시에 짜면 정규화 지옥에 빠짐. **유니클로 하나만** collect→extract→normalize→analyze 끝까지 관통 → 진짜 문제(이미지 사이즈표, 여유 계산 오차 등) 발견 → 그 뒤 2번째 브랜드로 어댑터 구조 일반화.

### 7-5. Out of Scope (인지만, 지금은 미룸)

- 원단 신축성/소재 (정확도 향상되나 수집비용 큼) → v2.
- 브랜드 간 사이즈 명칭 교차 번역표(A의 M = B의 L) → 따로 수집 아님. 5번 경향성이 쌓이면 **분석 산출물로 자동 도출**.

## 8. 미해결 질문 (구현하며 답할 것)

- 착용 경험을 몇 벌·어떤 부위까지 받아야 역산 오차가 실용 수준(±2cm 목표)에 드는가.
- 브랜드 실측을 공개 사이즈표만으로 시작할지, 실착 보정을 언제부터 섞을지.
- 주관 평가 ↔ cm 매핑 테이블의 초기 가설값을 어떻게 잡을지.
- 페이크도어 클릭의 허수 보정 방법.
