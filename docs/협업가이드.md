# 협업 가이드 — 3인 분담 & 충돌 최소화

> **이 문서는 무엇?** 팀장·디자이너·개발자 3인이 **각자 작업해도 git 충돌이 안 나도록** 파일 소유·경계·작업 관례를 정한 문서입니다. 소유 강제는 [`.github/CODEOWNERS`](../.github/CODEOWNERS)가, 규칙 설명은 이 문서가 합니다.

## 1. 역할 · 소유 경로

| 역할 | 무엇 | 소유 경로 |
|---|---|---|
| **팀장** | 시드 데이터 수집·정규화 | `data/**` · `web/data/{garments,body-base-model,body-distribution,archetypes}.json`(생성물) |
| **디자이너** | 화면 레이아웃·디자인 컨셉 | `web/*.html`(마크업) · `web/css/**`(`tokens.css` 포함) · `web/photos/` · `web/data/bodytypes.json` |
| **개발자** | 엔진·구현 로직 | `engine/**` · `web/js/**` |
| 공동 | 기획 | `docs/**` |

## 2. 충돌이 안 나는 이유 — "한 파일 = 한 역할"

가장 잘 부딪히던 곳은 **HTML 안에 레이아웃(디자이너)과 로직(개발자)이 섞여 있던 것**이었습니다. 그래서 **각 화면의 로직을 `web/js/<화면>.js`로 분리**했습니다.

```
web/
  diag-fit.html      ← 디자이너: 마크업·클래스·구조만
  js/diag-fit.js     ← 개발자: 그 화면의 로직만
```

| 화면 | 마크업(디자이너) | 로직(개발자) |
|---|---|---|
| 홈 | `index.html` | `js/index.js` |
| 결과 | `result.html` | `js/result.js` |
| 결과카드 | `card.html` | `js/card.js` |
| 진단① 기본정보 | `diag-basic.html` | `js/diag-basic.js` |
| 진단② 착용경험 | `diag-fit.html` | `js/diag-fit.js` |
| 진단③ 로딩 | `diag-loading.html` | `js/diag-loading.js` |

> `about/privacy/terms.html`은 로직 없는 순수 마크업(디자이너 단독). 엔진 코어는 `js/engine.js`·`body-model.js`·`engine-mock.js`(개발자).

## 3. 경계(인터페이스) — 서로의 계약

두 역할이 만나는 지점은 **파일이 아니라 계약**으로 잇습니다. 계약만 지키면 서로의 파일을 안 건드립니다.

| 계약 | 정의하는 쪽 | 참조하는 쪽 |
|---|---|---|
| **DOM `id`/`class`** | 디자이너(HTML) | 개발자(JS가 `getElementById`/쿼리로 참조) |
| **디자인 토큰** `web/css/tokens.css` 변수 | 디자이너 | 모두(색·간격은 변수만 참조, 하드코딩 금지) |
| **8유형 데이터** `web/data/bodytypes.json` | 디자이너 | 개발자(카드 렌더가 읽음) |
| **엔진 계약** `FitEngine`/`diagnose()` (docs/6 §5) | 개발자 | 화면 JS |
| **시드 스키마** `garmentCm` 등 | 팀장(생성기) | 개발자(엔진이 읽음) |

- **id/class를 바꾸려면** 디자이너 → 개발자에 알림(JS 셀렉터가 깨짐). 반대로 개발자가 새 id가 필요하면 디자이너에 요청.
- 색·폰트·간격은 **`tokens.css` 변수만** 참조 — 화면 JS·HTML에 색값 하드코딩 금지.

## 4. 생성물은 손으로 고치지 않는다

`web/data/`의 시드 JSON은 **생성기 산출물**입니다. 직접 편집하지 말고 소스+생성기로 재생성하세요(안 그러면 다음 재생성 때 덮어써짐).

| 파일 | 소스 | 재생성 |
|---|---|---|
| `garments.json` | `data/brand/raw/*.csv` | `python3 data/brand/build-sizespec.py` |
| `body-*.json` · `archetypes.json` | `data/body/clean/*.csv` | `python3 data/body/build-bodyseed.py` |

> 예외: `bodytypes.json`은 생성물이 아니라 디자이너가 직접 관리하는 표현 데이터(단일 출처).

### 4.1 사이즈표 캡처 (`data/brand/raw/size-tables/`)

수집·인식 테스트용 사이즈표 스크린샷을 두는 곳입니다. **폴더 자체는 유지(`.gitkeep`)하되 안의 파일은 전부 `.gitignore`로 제외**합니다 — 쇼핑몰 스크린샷은 브랜드 저작물이고, 실측표 노출은 해자 유출입니다. 캡처는 각자 로컬에만 둡니다.

파일명은 `<브랜드>_<상품>_<종류>.png`. `<종류>`가 인식 파이프라인의 첫 분기라 정답 대조에 그대로 씁니다.

| 종류 | 뜻 | 엔진에서의 취급 |
|---|---|---|
| `garment` | 옷 실측표(단면·길이) | A축 — 여유 계산에 바로 사용 |
| `body` | 신체 권장범위표(범위값) | B축 prior — **여유 계산 불가**, 중간값만 사용 |
| `mixed` | 둘이 섞인 표 | 분리 후 각각 처리 |

- **원본 URL과 캡처 날짜를 함께 기록**합니다. 브랜드는 시즌마다 실측을 바꾸므로 날짜 없는 값은 못 씁니다([6_사이즈엔진.md](6_사이즈엔진.md) §1.2 출처메타).
- 인식 결과를 `web/data/garments.json`에 직접 넣지 않습니다 — 위 §4대로 소스 CSV에 반영하고 생성기를 돌립니다.
- 같은 `가슴둘레` 라벨이 브랜드마다 단면일 수도 둘레일 수도 있습니다. **라벨이 아니라 값의 분포로 판정**합니다(기존 TOP `chest`는 34~83cm 범위).

## 5. 브랜치 · PR 관례

> 팀원 전원이 Claude Code로 개발합니다. 이 관례는 루트 [`CLAUDE.md`](../CLAUDE.md)에도 요약돼 모든 세션에서 자동 적용됩니다.

**역할별 개인 브랜치 하나를 계속 재사용합니다 — 작업마다 새 브랜치를 파지 않습니다.**

| 역할 | 개인 브랜치 |
|---|---|
| 개발자 | `dev-working` |
| 디자이너 | `design-working` |
| 팀장 | `data-working` |

- 자기 `-working` 브랜치에서 **이어 작업**하고, 내용은 **커밋 메시지로 구분**합니다. (초기 단계라 기능마다 브랜치·PR을 남발하지 않음)
- **`main` 직접 커밋 금지.** main 통합은 PR로.
- **push·PR은 자동으로 하지 않습니다** — 커밋까지 하고, **올릴지는 필요할 때 판단해 진행**(Claude Code는 사용자에게 물어본 뒤).
- **머지 후 재동기화**: PR이 main에 머지되면 개인 브랜치를 최신 main에 맞춘 뒤 **같은 브랜치로** 다음 작업을 이어갑니다.
  - squash 머지 → `git fetch origin && git reset --hard origin/main && git push --force-with-lease`
  - merge/rebase 머지 → `git fetch origin && git merge origin/main`
- PR을 열면 `CODEOWNERS`가 해당 경로 소유자에게 **리뷰를 자동 요청**합니다. **남의 소유 경로를 바꿔야 하면** 그 소유자를 리뷰어로 넣고 PR 설명에 이유를 씁니다.
- 머지 전 확인: 엔진 변경 시 `npm test`, 화면 변경 시 로컬 서버(`npm run serve`)로 해당 화면 로드.
- 커밋 전 문서(`docs/*.md`)를 고쳤으면 뷰어 재생성(git 훅이 자동 강제 — [문서작성규칙.md](문서작성규칙.md)).

## 6. 충돌이 나면

- 대부분 위 경계를 지키면 **서로 다른 파일**을 고치므로 충돌이 없습니다.
- 같은 파일(주로 `*.html`)을 불가피하게 함께 고쳐야 하면: 작은 PR로 자주 머지 + 먼저 알림. 한 화면을 크게 뜯을 땐 그 화면을 한 사람이 잡고 나머지는 대기.
