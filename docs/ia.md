# 핏팅(Fitting) — 정보구조 · 화면 · 흐름

> **이 문서는 무엇인가요?**
> 핏팅 앱에 **어떤 화면이 있고, 사용자가 어떤 순서로 이동하는지**를 한곳에 모은 설계 지도입니다. 이런 문서를 흔히 **IA(Information Architecture, 정보구조)** 라고 부릅니다 — "정보와 화면을 어떻게 배치하고 연결하는가"의 설계도라는 뜻입니다.
> 처음 읽는 사람도 이 문서만으로 앱 전체 그림을 그릴 수 있도록 썼습니다.
>
> **함께 보는 문서(여기서 중복하지 않음):**
> - 색·글꼴·버튼 같은 디자인 규칙 → `design.md`
> - 사이즈를 계산하는 엔진과 데이터 구조 → `engine.md`
> - 쇼퍼찾기(전문가 매칭)의 자세한 내용 → §13(쇼퍼찾기 상세)

---

## 0. 핏팅은 어떤 서비스이고, 무엇으로 성공을 판단하나요?

**핏팅은 "브랜드마다 다른 옷 사이즈를 번역해 주는" 서비스입니다.**
예를 들어 A브랜드에서 M을 입는 사람이 처음 사는 B브랜드에서는 몇 사이즈를 골라야 할지 알려줍니다. 사용자가 **입어본 옷의 경험**(브랜드 + 핏 + 사이즈 + "어깨가 꼈다/배가 남았다" 같은 느낌)을 입력하면, 그걸 거꾸로 계산해(=**역산**, 결과에서 원인을 되짚어 몸의 치수·체형을 추정하는 것) 체형을 파악하고, 안 입어본 브랜드의 내 사이즈를 알려 줍니다.

**성공을 재는 핵심 숫자(킬 메트릭)**
- **킬 메트릭**(kill metric) = "이 하나가 안 되면 서비스가 죽는다"고 볼 만큼 가장 중요한 지표를 뜻합니다.
- 핏팅의 킬 메트릭 = **사이즈 정확도 ≥ 65%**. 즉, 진단 뒤 "실제 내 사이즈와 맞다"고 동의한 사람이 65% 이상이어야 합니다. (2-5 피드백 화면에서 측정)
- 보조 지표 = **전환신호 ≥ 25%** (쇼퍼찾기 진입을 눌러 본 사람 비율. 3-1 화면에서 측정)

> **왜 쇼퍼 매칭이 아니라 사이즈 번역인가요? (피벗 이야기)**
> 핏팅은 원래 "전문 쇼퍼를 연결해 주는" 서비스(옛 이름 핏메이트)로 시작했다가, **"내 사이즈를 정확히 번역해 주는" 서비스로 방향을 크게 틀었습니다**(이렇게 사업 방향을 크게 바꾸는 것을 **피벗**pivot이라 합니다). 그래서 체형 분류도 단순 성격 캐릭터 6종에서 **8체형(국가 인체 치수 자료인 '사이즈코리아' 근거)** 으로, 성공 기준도 만족도에서 **사이즈 정확도 ≥65%** 로 바뀌었습니다. 이 문서는 피벗 이후의 확정 상태만 담습니다.

**이 문서에서 쓰는 표시 약속**
- 우선순위: ★ MVP에서 실제로 만드는 것 · ◐ MVP에서 최소만 · ○ 목업(껍데기만)·나중(v2 이후)
  - **MVP**(Minimum Viable Product) = "최소한으로 만들어 가설을 검증하는 첫 버전"을 뜻합니다.
- 화면 번호: **탭 기준**으로 붙입니다. 홈=탭1, 체형진단=탭2(2-1, 2-2…), 쇼퍼찾기=탭3(3-1…), 마이=탭4. 공통 페이지는 G.*, 별도 앱은 전문가=E.* / 어드민=A~D.

## 1. 화면은 크게 어떻게 나뉘나요? (내비게이션 구조)

> **왜 4탭인가요?** 사용자가 앱 어디에 있든 손가락으로 바로 옮겨 다닐 수 있는 큰 방 4개를 화면 아래에 고정해 둔 것입니다. "지금 내가 어디 있는지" 헷갈리지 않게 하는 가장 단순한 구조입니다.

사용자 앱은 화면 **맨 아래에 항상 떠 있는 탭 4개**로 이동합니다. 그 위에 상황에 따라 잠깐 뜨는 요소(동의창·로그인창·약관)와, 사용자 앱과는 완전히 분리된 **별도 앱 2개**(전문가용·운영자용)가 있습니다.

```
┌──────────────── 사용자 앱 ────────────────┐
│  🏠 홈    🔍 체형진단   👔 쇼퍼찾기   👤 마이 │  ← 화면 아래 4탭(항상 보임)
└────────────────────────────────────────────┘
   위에 잠깐 뜸: 개인정보 동의창 · 로그인창 · 약관

[별도 앱]  ② 전문가 앱(껍데기)   ③ 운영 어드민(껍데기)
```

| 탭 | 이름 | 한 줄 설명 | MVP |
|---|---|---|---|
| 탭1 | 🏠 홈 | 첫 화면. "이게 뭐 하는 서비스"를 3초에 보여주고 진단을 시작시킴 | ★ |
| 탭2 | 🔍 체형진단 | **핏팅의 심장.** 무료로 사이즈를 진단해 주는 핵심 화면들 | ★ |
| 탭3 | 👔 쇼퍼찾기 | 전문가(쇼퍼) 연결. MVP에서는 "수요만 떠보는" 껍데기 | ★(껍데기만) |
| 탭4 | 👤 마이 | 내 진단 기록·저장·로그인 | ◐ |

> **홈 탭이 곧 첫 화면(랜딩)입니다.** 별도의 소개 페이지를 따로 두지 않고, 탭1 자체가 서비스를 소개하는 첫 화면 역할을 합니다.

## 2. 탭마다 무슨 화면이 있나요?

### 탭1 🏠 홈 ★

> **왜 이렇게?** 처음 온 사람이 3초 안에 "이게 나한테 쓸모 있겠다"고 느끼고 곧장 진단을 시작하게 만드는 게 목표입니다.

| 노드 | 무엇을 보여주나 | 참고 |
|---|---|---|
| 히어로 ★ | 서비스 한 줄 소개 + 큰 버튼 **[내 사이즈 진단 시작]** → 탭2로 | 서비스의 입구 |
| 상황별 쇼퍼 리스트 ★ | 소개팅·면접·결혼식·여행 등 상황별 전문가 목록 | **전문가가 아직 0명이면 "오픈예정/1기 모집"으로 표시** — 손님에겐 "관심 있으면 알림 신청", 전문가에겐 "지원하세요" 공고로 동시에 작동(양쪽을 동시에 모으는 방법) |
| 작동 방식 3스텝 ★ | "이렇게 씁니다" 3단계 설명 | 신뢰를 줌 |
| 헤더 | 로그인 · **전문가 지원**(→ 전문가 앱으로) | |

### 탭2 🔍 체형진단 ★ — 서비스의 심장

> **왜 여기에 힘을 쏟나요?** 핏팅의 성공 여부(킬 메트릭)가 여기서 결정되기 때문입니다. 사용자가 실제로 "내 사이즈 맞네!"라고 느끼는 곳이 바로 이 진단 결과입니다.

탭에 들어오면 먼저 **개인정보 수집·이용 동의(G.5)** 를 받고, 그다음 질문을 차례로 묻는 **위저드**(wizard, 여러 단계를 한 걸음씩 안내하는 입력 방식)가 시작됩니다.

```
[동의] → 2-1 기본정보 → 2-2 착용경험 → 2-3 로딩 → 2-4 결과카드 ─┬─ 2-5 피드백 → 2-6 전문가전환 → (탭3)
                                                                  └─ 2-7 결과공유 ○
```

| # | 화면 | MVP | 무엇을 하나 | 참고 |
|---|---|:--:|---|---|
| 2-1 | 기본 정보 입력 | ★ | 성별·나이·키·몸무게 | 성별을 고르면 평균값을 미리 채워 줌. 선호핏은 2-2에서 물음 |
| 2-2 | 착용경험 보정 | ★ | ↓ 아래 상세 참고. **질문 항목의 정본(기준)은 엔진 쪽**(`engine.md`) | 정확도의 핵심 |
| 2-3 | 진단 로딩 | ★ | 계산이 도는 대기 화면(역산 + 매칭 + AI 설명 문장) | 몇 초 걸릴 수 있음 |
| 2-4 | 결과 카드 | ★ | ① 체형 프로필(8유형 캐릭터) ② 핏 인사이트 ③ 브랜드별 추천 사이즈 · 옷장(진단 현황) | 제품의 얼굴. 링크로 공유 가능 |
| 2-5 | 피드백 수집 | ★ | "실제 내 사이즈와 맞나요?" · "안 사본 브랜드도 사보겠어요?" | **여기서 킬 메트릭(정확도 ≥65%) 측정** |
| 2-6 | 전문가 전환 | ★ | "스타일까지 전문가에게 받기" 버튼 → 탭3 | 무료 진단 → 유료 상담으로 넘어가는 다리 |
| 2-7 | 결과 공유 | ○ | 카드 이미지/링크 공유 | 입소문 유도, 나중에 |

#### 2-2 착용경험 — 자세히

> **왜 옷 종류(카테고리)별로 나눠서 묻나요?** 화면 편의가 아니라 계산 방식 때문입니다. 상의를 입어본 경험으로는 상체(어깨·가슴)만 알 수 있고, 하체(허리·엉덩이)는 알 수 없습니다. 그래서 질문·계산·결과를 모두 옷 종류 단위로 나눕니다.

묻는 것:
- **어떤 옷 종류를 진단할지 선택** (상의/하의 등)
- **그 종류의 선호핏** (스키니·슬림·레귤러·루즈·오버 중 택1)
- **같은 종류로 입어본 옷** `[브랜드 + 핏 + 사이즈]` 을 **1~3벌**(원하는 만큼)
- **부위별 느낌** → 큼 / 여유 / 딱맞음 / 끼임
  - 상의 부위: 어깨·가슴·소매·배·총장
  - 하의 부위: 허리·엉덩이·허벅지·밑위·기장

이 화면의 상태(state)별 처리 — **state**란 "같은 화면이 상황에 따라 다르게 보이는 경우"를 말합니다:

| 상황(state) | 어떻게 처리하나 |
|---|---|
| 입력이 길어짐 | 진행 단계 표시(몇 번째인지) + 뒤로가서 수정 가능 → 중간 이탈 방지 |
| 입어본 옷이 0벌 | "건너뛰기" 허용(대신 결과 신뢰도는 낮게 표시) |
| 1벌만 기억남 | "1벌로 진단" vs "다른 핏 옷 더 넣기" 중 선택 + 신뢰도 안내 |
| 목록에 그 브랜드가 없음 | "직접 입력" / "해당 없음" 선택 (MVP는 브랜드가 몇 개뿐이라 자주 생김) |

#### 2-4 결과 카드 — 상태별 처리

| 상황(state) | 언제 | 어떻게 보이나 |
|---|---|---|
| **신뢰도 3단계** | 입력한 옷 수에 따라 | **0벌 = 낮음**(주의 배너) / **1벌 = 보통**("한 벌 더 넣으면 정확해져요") / **2~3벌 = 정확**(배너 없음) |
| 추정 실패 | 계산이 확신을 못 할 때 | "옷을 더 추가해 주세요" → 2-2로 되돌림 |
| 재진단 | 결과가 마음에 안 들 때 | 입력을 고쳐 다시 → 2-2 |
| 결과 저장 | 저장 버튼 | 비회원이면 임시 링크로 잠깐 보관 / 가입하면 영구 보관 = **가입 유도 지점 ①**(아래 §8 참고) |
| 옷장 / 이어서 진단 | 계속 오게 만드는 장치 | 종류별 완료 현황(옷장)을 보여주고, 한 번 누르면 다음 종류 진단으로 이어짐. 기본정보·체형은 재사용하므로 두 번째부터는 옷만 새로 입력 |

### 탭3 👔 쇼퍼찾기 — 전문가 매칭

> **왜 껍데기(페이크도어)만 만드나요?** 실제 전문가 매칭 시스템은 크고 비쌉니다. 그래서 먼저 **버튼만 진짜처럼 만들어 두고 "얼마나 많은 사람이 누르는지"만 측정**합니다. 이렇게 "수요를 먼저 떠보는 가짜 문"을 **페이크도어**(fake door)라고 합니다. 자세한 내용은 §13(쇼퍼찾기 상세).

> 탭에 보이는 이름은 "쇼퍼찾기", 내부 용어·코드에서는 "전문가(expert)"로 통일합니다.

| # | 화면 | MVP | 요약 |
|---|---|:--:|---|
| 3-1 | 매칭 진입(페이크도어) | ★ | 관심 서비스 선택(온라인 스타일링 / 동행 쇼핑 / 이미지 컨설팅) → 누른 것을 기록. **전환신호 ≥25% 측정** · **가입 유도 지점 ②** |
| 3-2 | 알림 완료 / 준비중 | ★ | "오픈하면 알려드릴게요" 확인 = **여기까지가 MVP 흐름의 끝** |
| 3-3~3-7 | 목록·프로필·요청·추천결과·결제 | ○ | 실제 매칭. 나중(v2). 자세히는 §13(쇼퍼찾기 상세) |

**핵심 차별점:** 진단 결과(체형·사이즈)가 전문가 요청에 **자동으로 첨부**되어, 전문가가 더 정확히 추천할 수 있습니다. 다른 매칭 서비스와 갈리는 지점입니다(상세: §13(쇼퍼찾기 상세)).

### 탭4 👤 마이 ◐

> **왜 최소만 만드나요?** MVP 단계에서는 "결과를 저장하려는 사람"만 붙잡으면 충분합니다. 전체 마이페이지는 나중에.

| 노드 | MVP | 내용 |
|---|:--:|---|
| 로그인 / 회원가입 | ◐ | **가입은 꼭 필요한 순간에만** 요구(아래 §8). 저장·매칭 시 창으로 뜸 |
| 진단 이력 | ○ | 시점별로 지우지 않고 쌓아 둔 기록 |
| 체형 변화 추이 | ○ | 체중·버전에 따른 변화 |
| 카테고리별 최신 결과 | ○ | "옷장"이 영구히 사는 곳 |
| 받은 추천 · 알림 | ○ | |

## 3. 공통 페이지 (탭 밖 어디서나)

> **왜 따로 두나요?** 약관·개인정보 같은 페이지는 특정 탭에 속하지 않고 어디서든 열 수 있어야 하기 때문입니다. 한국에서 서비스하려면 법적으로 꼭 있어야 하는 것도 있습니다.

| ID | 페이지 | MVP | 참고 |
|---|---|:--:|---|
| G.1 | 서비스 소개 / 작동 원리 | ★ | 홈 안에도 넣고 독립 페이지로도. 정확도 근거 제시 |
| G.3 | 이용약관 | ★ | 한국 서비스 법적 필수 |
| G.4 | 개인정보 처리방침 | ★ | **체형·신체치수는 민감정보**임을 명시(법적 필수) |
| G.5 | 개인정보 수집·이용 동의 | ★ | **진단 시작(2-1) 직전에 받는 동의창** |
| G.2 | FAQ | ○ | 자주 묻는 질문 |
| G.6 | 알림 | ○ | 사용자·전문가 공용 |
| G.7 | 404 / 에러 | ○ | 잘못된 링크·네트워크 오류 |

## 4. 별도 앱 2개 (지금은 껍데기)

## 5. 진단 카테고리: "기반"과 "파생"

> **왜 나누나요?** 옷 종류마다 알 수 있는 정보가 다르기 때문입니다. 상·하의는 몸을 직접 재는 "기반"이 되고, 아우터·원피스·치마는 취향(선호핏)만 물으면 되는 "파생"입니다.

```
기반(基盤) — 몸(체형)을 실제로 확정하는 옷
├─ 상의 : 상체(어깨·가슴·소매·배·총장)를 역산
└─ 하의 : 하체(허리·엉덩이·허벅지·밑위·기장)를 역산
      └▶ 상의 + 하의 둘 다 끝나야 체형 카드가 완성됨 (= 8유형 중 하나로 확정)
         상의만 끝나면 = 상체만 나온 "부분 카드" (8유형은 아직 미확정)

파생(派生) — 선호핏(취향)만 물음 · 기반이 끝나야 열림
├─ 아우터   ○
├─ 원피스   ○
└─ 치마     ○
```

- **기반 = 상의 + 하의 둘 다.** 둘 다 끝나야 체형 카드가 완성되고 8유형 중 하나로 확정됩니다. 상의만 끝나면 상체만 나온 부분 카드이고 8유형은 미확정입니다.
- **파생(아우터·원피스·치마)은 선호핏만** 받고, 기반이 끝나 체형이 확정된 뒤 열립니다.
- 실제 구현은 상의부터 시작합니다(브랜드 종류와 옷 종류를 동시에 벌리지 않기 위한 좁히기).

> **지금 남은 일(엔진 쪽):** 현재 시제품은 상의만 되어 있어 **하의 진단 흐름을 추가**해야 하고, 하의 계산 규칙도 아직 임시라 **정식화**가 필요합니다. 상세는 `engine.md`.

## 6. 8유형과 엔진을 잇는 문제 (결과화면 만들기 전에 따로 설계)

> **왜 미리 못박아 두나요?** 결과 카드를 그리는 방식(8유형 그림 1개)과 엔진이 계산하는 방식(상·하체 따로)이 서로 어긋나기 때문입니다. 지금 답을 정하진 않되, "이 연결 작업이 반드시 필요하다"는 것을 명시해 둡니다.

- **8유형(카드에 그리는 그림)** = 몸 전체 실루엣 1개(사각 STR / 삼각 TRI / 역삼각 INV / 모래시계 HRG / BAL / DIA / RND / TUB).
- **엔진(계산)** = 상체·하체를 **따로** 계산합니다. 상의 진단이면 상체 라인만, 하의 진단이면 하체 라인만 나옵니다.
- **그래서 필요한 것:** `엔진의 상체 결과 × 하체 결과 → 8유형 코드` 로 바꾸는 **매핑 표**. 그리고 상의만 끝났을 때는 8유형을 아직 못 정하므로 **"부분 카드"를 어떻게 보여줄지** 규칙도 필요합니다.
- 이 작업은 `engine.md`의 체형 결과 구조와 아래 §7의 8유형 데이터를 잇는 일입니다. **결과화면 구현에 착수하기 전 별도 항목으로 설계합니다.**

### 겹치던 부분을 어떻게 정리했나 (확정)

| 겹치던 지점 | 정리 결과 |
|---|---|
| 결과 화면 | 그림(카드)은 8유형 디자인, 그 안에 들어가는 **숫자·값은 엔진이 공급**. 배치는 좌측 카드 / 우측 브랜드 사이즈표·검증 |
| 진단 질문 항목 | **엔진 스키마(데이터 구조)를 기준**으로 맞춤. `engine.md` |
| 전문가 영역 | 사용자 쪽 연결(2-6 전환·3-1 페이크도어)은 진단 쪽에서 / 전문가 마켓·전문가 앱은 §13(쇼퍼찾기 상세)에서 |

## 7. 8유형 데이터는 딱 한 곳에서만 온다

> **왜 한 곳으로 모으나요?** 8유형의 이름·색·설명이 여러 화면 코드에 각각 적혀 있으면, 하나 바뀔 때 전부 찾아 고쳐야 하고 실수가 납니다. 그래서 원본을 파일 하나로 정합니다.

**`web/data/bodytypes.json`** 이 8유형 정보의 유일한 원본입니다(카드 표시용).

- 8유형의 **코드·이름·색·프로필·핏 설명·궁합·실루엣 좌표**는 오직 이 파일에서 읽습니다. 화면 코드에 다시 적지 않습니다.
- 8유형 코드(STR/HRG…)와 실루엣 모양의 연결은 이 파일의 `silhouette` 값으로만 합니다.
- 이 파일은 "보여주기"용입니다. 엔진 계산 결과를 8유형 코드로 바꾸는 매핑 표는 §6이 담당합니다.
- 결과 카드는 예외적으로 어두운 배경 + 유형별 시그니처색을 씁니다(디자인 예외). 상세는 `design.md`.

## 8. 흐름을 짤 때 지키는 3가지 원칙

> 세 원칙 모두 "사용자를 중간에 놓치지 않으려는" 목적에서 나옵니다.

1. **가입은 나중에 (lazy 가입)** — **lazy 가입**이란 "회원가입을 서비스 초반에 강요하지 않고, 사용자가 가치를 느낀 뒤로 미루는 것"입니다. 비회원 그대로 진단~결과~피드백까지 다 하게 두면 이탈이 줄고 측정 표본도 커집니다. 가입은 딱 **두 순간**에만 요청합니다: ① 결과를 저장할 때(2-4), ② 전문가 매칭을 실제로 쓸 때(3-1).
2. **동의는 입력 바로 직전에** — 체형·치수 같은 민감정보를 받기 직전(2-1 들어갈 때)에 동의(G.5)를 받습니다. 첫 화면에서 미리 받으면 부담스러워 나가 버립니다.
3. **막다른 길 금지** — 실패하거나 비어 있는 화면에도 항상 되돌아갈 길을 둡니다(추정 실패 → 옷 추가, 신뢰도 낮음 → 재진단, 링크 만료 → 홈).

## 10. MVP에서 정확히 무엇을 만들고, 무엇을 재나요?

> **왜 이만큼만?** 가장 검증이 급한 것(사이즈 정확도)부터 확인하기 위해서입니다. 전체를 설계하되, 검증에 꼭 필요한 곳만 먼저 만듭니다.

**MVP에서 실제로 만드는 것(★)**
- 탭1 홈
- 탭2 진단 전체(2-1 ~ 2-6)
- 탭3 페이크도어(3-1 · 3-2)
- 공통 G.1 / G.3 / G.4 / G.5

**측정 지점 2곳**
- **2-5 피드백 → 킬 메트릭: 사이즈 정확도 ≥ 65%** (안 되면 서비스 자체를 재검토)
- **3-1 페이크도어 → 전환신호: 클릭/알림신청 ≥ 25%**

**안 만드는 것(껍데기/나중)**: 3-3~3-7 실제 매칭 · 마이페이지 전체 · 전문가 앱 · 어드민 · 결과 공유(2-7, 여유 되면).

> **꿀팁 — 처음엔 코드 없이도 시작 가능.** 2-4의 추천 사이즈는 초기엔 사람이 직접 브랜드 사이즈표를 보고 손으로 번역해도 됩니다. 정확도(킬 메트릭)부터 확인한 뒤 엔진을 자동화합니다. 상세는 `engine.md`.

---

## 11. 나중에 다시 와도, 몸이 변해도 (시점 버저닝)

> **왜 필요한가요?** 사람 몸은 변합니다(체중 등). 재진단할 때 과거 기록을 덮어써 버리면, "예전 결과"와 "정확도 학습용 데이터"가 사라집니다. 그래서 기록을 지우지 않고 쌓습니다. 이렇게 시점별로 판을 나눠 관리하는 것을 **버저닝**(versioning)이라 합니다.

- 진단은 **덮어쓰지 않고 계속 쌓습니다**(append-only, 추가만 함). 체형은 "언제 기준"인지 날짜를 붙여 버전으로 둡니다.
- 그래서 ① 과거 결과 보존 ② 정확도 학습 데이터가 오염되지 않음 ③ 체형 변화 추이를 자산으로 쌓음 — 세 가지를 동시에 얻습니다.
- 결과 화면에는 "어느 시점(몸무게·날짜) 기준"인지 표시하고, 체중이 바뀌면 "새로 진단(과거는 보존)" 경로를 엽니다. 옷장(카테고리) 재사용은 체형이 같을 때만 유효합니다.

## 12. 데이터는 어떻게 저장하나요? (데이터 모델)

> **왜 처음부터 깔끔히?** 여기 쌓이는 데이터가 곧 "브랜드 교차 사이즈 데이터셋"이고, 이것이 나중에 회사의 가장 큰 자산(B2B API 사업, 어드민 D)이 되기 때문입니다.

아래는 저장하는 데이터 묶음(표/테이블)과 그 안의 항목입니다.

```
user                누구 · 성별 · 나이 · 키 · 몸무게 · 선호핏(옷 종류별)

garment_experience  누구 · 브랜드 · 옷종류 · 입은사이즈 ·
                    느낌{ 부위(어깨·가슴·소매·배·총장 / 허리·엉덩이·허벅지·밑위·기장)
                          : 큼|여유|딱맞음|끼임 }           ← 핏팅만의 핵심 데이터

result_card         누구 · 체형(8유형 코드, 아직 미확정일 수 있음) ·
                    추천[{ 브랜드 · 옷종류 · 추천사이즈 · 신뢰도 · 근거 }]

validation          누구 · 브랜드 · 판정(맞음|애매|틀림) ·
                    실제사이즈 · 다른브랜드도전의향(예/아니오)   ← 킬 메트릭의 원천

fakedoor_event      누구 · 관심서비스 · 누른시각 ·
                    알림신청여부 · 연락처                       ← 전환신호의 원천
```

- `result_card`의 체형은 상의+하의를 다 하면 8유형으로 확정, 상의만 하면 미확정(상체 부분)입니다.
- `garment_experience`는 §11대로 덮어쓰지 않고 시점별로 쌓입니다.
- 데이터 구조의 1순위 축은 "옷 종류(category)"입니다. 자세한 스키마는 `engine.md`.

## 13. 쇼퍼찾기(전문가 매칭) 도메인 상세

> MVP에서 실제로 만드는 건 **페이크도어(3-1) 하나**뿐이고, 아래 마켓·전문가앱·어드민은 대부분 **v2 확장**입니다. 사용자앱과 한 서비스로 묶여 있어 이 문서에 함께 둡니다.

### 0. 왜 "3면"으로 나뉘는가

**배경:** 무료 진단은 사용자 한 명만 상대하면 되는 **단순한 도구**다. 하지만 전문가 매칭은 사용자와 전문가를 서로 연결하고, 그 사이에서 문제가 안 생기게 관리까지 해야 한다. 그래서 관련된 사람이 세 종류로 늘어난다.

이런 서비스를 **마켓플레이스(marketplace, 파는 사람과 사는 사람을 연결해 주는 장터. 예: 배달앱은 손님·가게·배달앱 운영을 잇는다)** 라고 부른다. 우리 쇼퍼찾기도 똑같이 세 무리가 얽힌다.

```
        ┌──────────────── 쇼퍼찾기(매칭) ────────────────┐
[수요] 사용자 측  ──요청──▶   [엔진/중개]   ──전달──▶  [공급] 전문가 측
  프로필·요청·결제          매칭·정산·분쟁            수신·작성·정산
                                 ▲
                    [운영 어드민] 승인·중재·정산집행
```

세 무리를 정리하면:

| 면 | 누구 | 무엇을 하나 | 어디서 |
| --- | --- | --- | --- |
| 수요(사려는 쪽) | 사용자 | 서비스 고르기·요청·상담·결제·후기 | 사용자 앱 탭3 쇼퍼찾기 |
| 공급(팔려는 쪽) | 전문가 | 요청 받기·수락·상담·결과 작성·정산 받기 | 따로 있는 전문가 앱 |
| 중개(관리하는 쪽) | 운영 어드민 | 전문가 심사·분쟁 해결·돈 정산·수수료 정하기 | 따로 있는 관리자 화면 |

**우리만의 강점(§5에서 자세히):** 무료 진단에서 나온 "이 사람은 이 브랜드에서 몇 사이즈"라는 결과가, 전문가에게 요청이 갈 때 **자동으로 같이 딸려간다.** 그래서 전문가는 사용자를 직접 안 만나도 옷 사이즈를 정확히 고를 수 있다. 이게 다른 스타일링 매칭 서비스에는 없는 우리만의 무기다.

---

### 1. 도메인 사이트맵 (전체 화면 지도)

**배경:** 위 3면이 각각 어떤 화면들로 이뤄지는지 한눈에 보는 지도다. `★MVP`는 지금 실제로 만드는 것, `v2`는 나중에 만들 것.

```
쇼퍼찾기 (매칭)
│
├─ A. 사용자 측 (수요) ── 탭3 쇼퍼찾기
│   ├─ S1 매칭 진입·서비스 유형 선택 ★MVP
│   ├─ S2 전문가 목록 (정렬·필터)            v2
│   ├─ S3 전문가 프로필 (결과물·후기 확인)    v2
│   ├─ S4 매칭 요청 (유형별 폼 분기)          v2
│   ├─ S5 상담 / 메시지                       v2
│   ├─ S6 추천 / 예약 결과 (유형별)           v2
│   ├─ S7 결제 / 예약                         v2
│   └─ S8 완료 · 후기 · 평점                  v2
│
├─ E. 전문가 측 (공급) ── 전문가 앱
│   ├─ E0 랜딩 → E1 지원/등록 → E2 심사 → E3 정산계좌·세금
│   ├─ E4 대시보드(가능/마감 스위치)
│   ├─ E5 매칭 요청 상세 ★강점(사이즈 결과 받는 화면)·수락/거절
│   ├─ E6 상담 → E7 추천/결과 작성(유형별)
│   └─ E8 상품관리 · E9 프로필/포트폴리오 · E10 리뷰 · E11 수익/정산
│
└─ O. 운영 어드민 (중개)
    ├─ O1 전문가 승인/심사 대기줄
    ├─ O2 분쟁·환불·신고 처리
    ├─ O3 정산 집행·수수료 정책
    ├─ O4 리뷰 관리
    └─ O5 매칭 성사율·거래액 대시보드
```

---

### 2. 사용자 측 플로우 S1~S8

**배경:** 사용자가 "전문가에게 스타일링 받고 싶다"고 마음먹은 순간부터 후기를 남기기까지의 순서다. 사용자는 두 가지 길로 들어온다.

- 🔥 **따뜻한 리드(warm lead — 이미 우리에게 관심을 보인, 전환 가능성 높은 사용자)** — 진단 결과카드를 다 보고 "스타일까지 전문가에게" 버튼을 눌러 들어온 사람. 이미 자기 사이즈 정보가 있다.
- ❄️ **차가운 리드(cold lead — 아직 우리를 잘 모르는 사용자)** — 진단 없이 홈/탭에서 바로 들어온 사람. 사이즈 정보가 없으니 "진단부터 하세요"로 안내한다.

```
S1 진입·유형선택 → S2 목록 → S3 프로필 → S4 요청 → S5 상담 → S6 결과 → S7 결제 → S8 후기
  (MVP=여기까지만, 나머지는 흉내만)    └──────────────── v2에서 진짜 작동 ────────────────┘
```

| # | 화면 | 무엇을 하나 | 구분 |
| --- | --- | --- | --- |
| **S1** | 매칭 진입·유형 선택 | 서비스 유형 3종 중 고르기 · 상황(소개팅·면접·결혼식·여행) 고르기 · **(지금은)"열리면 알려주세요" 신청 + 1기 전문가 모집** · 여기서 회원가입 요구(가입 게이트②) | ★MVP |
| **S2** | 전문가 목록 | 정렬(잘 맞는 순·평점·후기·가격) · 필터(유형·상황·예산·지역·일정) · 전문가 카드(작업 결과물·전문분야·평점·가격·**매칭도 뱃지**=체형·상황이 얼마나 맞는지) | v2 |
| **S3** | 전문가 프로필 | **작업 결과물 갤러리** · 후기·평점 · 경력·전문분야·상황 태그 · 제공 서비스/가격/소요시간 · 예약 가능 일정. (얼굴·팔로워 수 ❌, 실제 결과물·후기 ✅ 로 신뢰를 준다) | v2 |
| **S4** | 매칭 요청 | **공통:** 내 체형·사이즈 정보가 자동으로 붙음(§5) + 상황·날짜 / **유형별로 묻는 항목이 다름**(§6) | v2 |
| **S5** | 상담 / 메시지 | 전문가와 1:1 채팅 · 예산·취향 등 세부 파악. (스타일링의 핵심은 결국 대화다) | v2 |
| **S6** | 추천 / 예약 결과 | 유형별 결과물(§6) — 온라인이면 코디·상품·구매링크, 오프라인이면 예약 확정·세션 안내 | v2 |
| **S7** | 결제 / 예약 | 온라인이면 매칭 수수료·패키지, 오프라인이면 예약금·세션비. 상황(면접·결혼식 등)이 과금 단위 | v2 |
| **S8** | 완료·후기 | 만족도·후기·평점·"또 부르고 싶다" 의향 → 이 점수가 다음 사용자의 S2 목록 정렬에 반영됨(후기가 좋은 전문가가 위로) | v2 |

**지금(MVP) 만드는 것은 S1까지.** S1에서 "알림 신청"을 받고 끝난다. S2~S8은 나중(v2)에 실제로 작동하게 만든다.

### 3. 전문가 측 E0~E11 (전문가 앱)

**배경:** 전문가는 사용자와 다른 별도 앱을 쓴다. 일감을 받고, 일하고, 돈을 받는 흐름이다. 사용자 앱 상단의 '전문가 지원' 버튼을 누르면 E0으로 온다. (지금은 전문가가 적어서 운영자가 직접 손으로 모집한다.)

```
E0 랜딩 → E1 지원/등록 → E2 심사/승인 → E3 정산계좌·세금
                                       ▼
E4 대시보드 ─ E5 요청상세 ★ ─ 수락 ─ E6 상담 ─ E7 결과작성
  (가능/마감 스위치)  (사이즈 결과 받음)
[언제든] E8 상품관리 · E9 프로필/포트폴리오 · E10 리뷰 · E11 수익/정산
```

| # | 화면 | 무엇을 하나 |
| --- | --- | --- |
| E0 | 전문가 랜딩 | 수수료·정산·기대 수익 안내 · '1기 지원' 버튼 (사용자 앱 모집 링크가 여기로 옴) |
| E1 | 지원/등록 | 경력·전문분야·잘하는 상황·포트폴리오 제출 |
| E2 | 심사/승인 대기 | 플랫폼이 품질을 거른다(운영자 쪽 O1과 짝). 승인 전에는 사용자에게 안 보임 |
| E3 | 정산계좌·세금 | 돈 받을 계좌·사업자/원천징수 정보 등록 |
| E4 | 대시보드 | 들어온 요청·진행 중인 일 · **"지금 일 받을 수 있음/마감" 스위치** |
| **E5** | **매칭 요청 상세 ★우리만의 강점** | 사용자 체형 정보 + **★사이즈 번역 결과를 받는 화면** · 수락/거절. 전문가가 안 만나도 정확히 고를 수 있게 하는 곳. 숨고·크몽 같은 단순 매칭과 갈리는 지점 |
| E6 | 상담/메시지 | 추천 전에 사용자와 대화로 취향 파악 |
| E7 | 추천/결과 작성 | 유형별(§6) — 온라인: 상품·코디·링크 / 오프라인: 예약·세션 노트 |
| **E8** | **서비스 상품 관리** | 내가 파는 서비스를 진열(메뉴판처럼) · 유형별 범위·소요시간·가격 정하기. 사용자 S3에 이대로 보임 |
| E9 | 프로필/포트폴리오 | 사용자에게 보일 내 소개 편집(= S3의 원본) |
| E10 | 리뷰/평점 | 받은 리뷰 보기·답하기 (평판이 곧 다음 일감) |
| E11 | 수익/정산 | 정산 내역 · 어떤 유형·상황이 돈이 되는지 통계 |

**전문가 앱 중 지금(MVP) 만드는 것은 E8(상품관리)뿐이다.** 사용자에게 보일 메뉴판이라 가볍게 만들 수 있어서다. 나머지는 흉내만 낸 화면(목업)이다.

---

### 4. 운영 어드민 O1~O5 (중개)

**배경:** 사용자와 전문가만 있으면 사고가 난다(전문가 자격 미달, 노쇼, 환불 분쟁 등). 운영자가 중간에서 관리하는 화면들이다. 지금은 다 나중(v2)에 만들고, 초기엔 스프레드시트로 손수 처리한다(§8).

| # | 화면 | 무엇을 하나 |
| --- | --- | --- |
| O1 | 전문가 승인/심사 대기줄 | E2의 운영자 쪽 · 승인·반려로 품질을 거른다 |
| O2 | 분쟁·환불·신고 처리 | 노쇼·불만족 환불 · 신고 대응 → **믿고 쓸 수 있는 장터의 핵심** |
| O3 | 정산 집행·수수료 정책 | 전문가별 지급 · 수수료율·유형별 요금 결정 = **돈 버는 조절 손잡이** |
| O4 | 리뷰 관리 | 가짜·악성 리뷰 정리 |
| O5 | 매칭 성사율·거래액 대시보드 | 거래액·성사율·활동 중인 전문가 수 |

---

### 5. 핵심 다리 — 진단 결과가 전문가에게 자동으로 넘어간다 ★

**배경:** 이게 우리 서비스가 다른 스타일링 매칭과 다른 **딱 하나의 진짜 차별점**이다. 그냥 사람만 연결해 주는 게 아니라, "이 사람 사이즈 데이터"를 함께 넘겨준다.

**핸드오프(handoff — 앞 단계에서 만든 정보를 다음 단계로 그대로 넘겨주는 것. 예: 병원에서 검사 결과지가 다음 진료실로 넘어가듯)** 라고 부른다. 진단에서 만든 체형카드가 전문가 화면(E5)으로 그대로 넘어간다.

```
[진단 결과카드]                            [전문가 E5 요청상세]
  · 체형 프로필(캐릭터)      ──넘겨줌──▶      · 고객 체형·핏 인사이트
  · 핏 인사이트(주의할 부위)                   · ★브랜드별 추천 사이즈
  · 브랜드별 추천 사이즈                        → "안 만나도 정확히 고를 수 있다"
```

이게 왜 세 쪽 모두에게 좋은가:

- **사용자에게:** 요청할 때 키·몸무게를 다시 안 적어도 됨(귀찮음↓). 그래서 전환이 잘 된다.
- **전문가에게:** 줄자 없이도 사이즈를 확신 → 비대면(온라인) 스타일링인데도 반품이 준다. 도구가 전문가 실력을 올려준다.
- **플랫폼에게:** 이 데이터가 쌓일수록 매칭이 정확해지고, 나중엔 B2B 사이즈 API(v3) 자산이 된다.

기술적으로는 요청 데이터에 `body_profile_ref`(진단 결과를 가리키는 참조 값)가 붙는다. 진단은 **불변 스냅샷(찍은 뒤 안 바뀌는 사진 같은 기록)** 으로 저장되므로, 요청한 그 시점의 체형이 그대로 고정돼서 넘어간다(나중에 살이 쪄도 그때 요청은 안 바뀜).

### 6. 서비스 유형 3종과 유형별 분기

**배경:** 전문가가 파는 서비스는 3종류인데, 종류마다 **요청할 때 묻는 것(S4)·받는 결과물(S6)·돈 내는 방식(S7)이 다르다.** 그래서 한 흐름 안에 세 갈래가 생긴다. 참고로 "퍼스널 쇼퍼"는 서비스 이름이 아니라 전문가의 직함이다.

| 항목 | 💻 온라인 스타일링 | 🛍️ 동행 쇼핑 | ✨ 이미지 컨설팅 |
| --- | --- | --- | --- |
| **요청 폼(S4)에서 묻는 것** | 취향·예산·좋아하는 브랜드 | 일정·지역·만날 장소 | 목적(면접 등)·일정·지역 |
| **진행 방식(S5·S6)** | 안 만나고 채팅으로 큐레이션 | 오프라인에서 같이 쇼핑 | 오프라인 세션 |
| **결과물(S6)** | 코디·상품·**구매링크** | 현장 구매·세션 노트 | 이미지 가이드·세션 노트 |
| **과금(S7)** | 매칭 수수료/패키지 | 예약금 + 세션비 | 세션비 |
| **사이즈 결과가 얼마나 중요한가** | ★가장 중요(안 보고 원격 구매) | 보조(현장에서 입어봄) | 보조(체형 이해용) |

흐름: 전문가가 E8에서 유형별로 진열·가격을 정함 → 사용자가 S1에서 관심 유형을 고름(이때 어떤 유형이 인기인지 측정) → S2 목록에서 유형 탭으로 탐색 → S4에서 유형에 맞는 폼을 채운다.

### 7. 상태(state)와 예외 상황

**배경:** 행복한 경우만 설계하면 안 된다. 전문가가 응답을 안 하거나, 후보가 아무도 없거나, 노쇼가 나는 상황도 미리 정해 둬야 사용자가 막다른 길에 갇히지 않는다.

- **요청 상태 흐름:** 대기 → `수락` → 진행 → 완료 / 중간에 `거절`(→ 다른 전문가 자동 추천) / `만료`(전문가가 무응답)
- **매칭 실패:** 조건에 맞는 전문가가 0명이면 → "열리면 알림" 등록 · "예산·지역을 넓혀 보세요" 제안
- **취소/노쇼(no-show — 예약해 놓고 안 나타남):** 취소 시점에 따라 환불 규칙이 갈림 → O2에서 운영자가 조정
- **분쟁:** 신고 → 운영자 중재 → 환불/제재
- **품질 게이트:** 심사(E2) 통과 전 전문가는 사용자에게 안 보임
- **전문가가 아직 0명일 때(콜드스타트 — 양쪽이 다 없어 서비스가 안 도는 초기 상황):** S1/홈에 "오픈 예정·1기 모집"으로 보여준다. 같은 화면이 사용자에겐 "수요 측정(알림 신청받기)", 전문가에겐 "모집 공고"로 동시에 작동한다. 닭이 먼저냐 달걀이 먼저냐 문제를 한 화면으로 푸는 방법.

### 8. 지금(MVP) vs 나중(v2) — 어디까지 진짜로 만드나

**배경:** 처음부터 매칭 시스템을 다 만들면 돈·시간이 크게 든다. 그래서 먼저 "이 서비스에 수요가 있는지"만 싸게 확인한다. 이때 쓰는 방법이 두 가지다.

- **페이크도어(fake door — 아직 없는 기능의 '문'만 만들어 두고, 사람들이 문을 두드리는지(클릭하는지)로 수요를 재는 방법. 문을 열면 "준비 중이에요"가 뜬다)**
- **위저드 오브 오즈(wizard of oz — 자동인 척하지만 실제로는 뒤에서 사람이 손으로 처리하는 방법. 오즈의 마법사에서 커튼 뒤 사람이 조종하듯)**

| | 지금 (MVP) | 나중 (v2, 진짜 작동) |
| --- | --- | --- |
| 사용자 측 | **S1 진입+유형선택 → 페이크도어 알림신청** | S2~S8 전체 흐름 |
| 전문가 측 | 없음(운영자가 손으로 1기 모집), E8만 진열용으로 | 전문가 앱 E0~E11 |
| 운영 | 스프레드시트로 손수 | 어드민 O1~O5 |
| 매칭 | **위저드 오브 오즈:** 알림 신청자 ↔ 1기 전문가를 사람이 카톡으로 연결 | 자동 매칭·결제·정산 |

**MVP에서 확인하려는 것 두 가지:**
1. **유형별 전환신호(conversion signal — 사용자가 실제로 관심을 행동으로 보이는 비율. 여기선 S1에서 유형을 고르고 알림 신청하는 비율) ≥25%** — 어떤 유형에 진짜 수요가 있는지.
2. **전문가 수락률** — 손으로 연결해 보며, 우리 회사의 가장 큰 가정인 *"따뜻한 리드(사이즈 정보까지 딸려오는 좋은 고객)라면, 수수료가 낮아도 전문가가 들어올까?"* 를 코드 한 줄 없이 확인한다.

### 9. 데이터 모델 (매칭 부분에서 저장하는 정보)

**배경:** v2에서 실제로 작동시키려면 아래 정보들을 저장해야 한다. 각 상자(테이블)가 무엇을 담는지 본다.

```
expert            id · 전문분야 · 상황태그[] · 평점 · 가능상태 · 심사상태
service_offering  expert_id · type(online|shopping|image) · 범위 · 소요 · price
match_request     user_id · expert_id · service_type · occasion · date
                  · body_profile_ref(진단 결과 참조 = 핸드오프) · form{유형별} · status
consultation      request_id · messages[]
result            request_id · type · items[](구매링크) | booking(예약·세션노트)
review            request_id · rating · text · 재호출의향
settlement        expert_id · request_id · amount · fee · status
```

읽는 법:
- `expert` = 전문가 한 명의 정보.
- `service_offering` = 전문가가 파는 서비스 하나. `type`이 §6의 3종(online/shopping/image).
- `match_request` = 사용자가 넣은 요청 하나. `body_profile_ref`가 §5 핸드오프가 실제로 구현되는 지점(진단 결과를 가리켜 요청 시점에 고정). `form`은 §6 유형별로 다른 입력 내용을 담는다.
- `consultation` = 요청에 딸린 대화 기록.
- `result` = 결과물. 온라인이면 `items[]`(상품·링크), 오프라인이면 `booking`(예약·노트)으로 갈린다.
- `review` = 후기·평점. `settlement` = 전문가에게 얼마 주고 수수료 얼마 뗐는지 정산 기록.

### 10. 측정 지표

**배경:** 단계마다 "잘 되고 있는지"를 재는 숫자다. 특히 MVP의 두 숫자가 서비스를 계속할지 말지를 가른다.

| 단계 | 무엇을 보나 |
| --- | --- |
| MVP(페이크도어) | 유형별 클릭/알림신청률 **≥25%** · 수동 매칭 **전문가 수락률** |
| v2 사용자 | 요청 전환율 · 결제 전환율 · 재호출 의향 |
| v2 전문가 | 가동률 · 수락률 · 평점 |
| v2 운영 | 매칭 성사율 · 거래액(GMV) · 분쟁율 |
