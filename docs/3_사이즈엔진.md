# 핏팅 엔진 — 정본 스펙

> **무엇인가:** 사용자가 "평소 입던 옷이 어땠는지"를 답하면 → 그 사람의 체형을 거꾸로 계산하고 → 다른 브랜드에서 맞을 사이즈를 추천하고 → 주의할 점을 경고하는, **사이즈 번역 엔진**의 모든 규칙·데이터·약속을 담은 **하나의 정본 문서**. (구 `engine/reverse-inference.md` · `engine/diagnosis-questions.md` · `engine/hidden-fit-dimensions.md` · `engine/build.md` · `통합/모듈화-sangmin.md`를 합침)
> **읽는 사람:** 개발·통계를 몰라도 이해할 수 있게 썼다. 전문용어는 처음 나올 때 괄호로 풀이한다.
> **한 문장 원칙:** 이 서비스의 심장은 사이즈 번역 엔진이고, 그 정확도가 검증 안 되면 나머지는 전부 무의미하다.
> **연결:** 화면 흐름은 [2_화면과흐름.md](2_화면과흐름.md), 디자인은 [4_디자인가이드.md](4_디자인가이드.md). 실제 코드는 `src/lib/schema/*`(데이터 정의)·`src/lib/engine/rules/*`(계산 규칙). 여긴 그 내용을 말로 설명하고, 중복은 링크로 넘긴다.

## 0. 원칙 (모든 판단의 뿌리)

> **왜 필요한가:** 엔진을 만들다 보면 "이건 AI한테 시킬까? 사람이 정할까? 상의만 받아도 될까?" 같은 갈림길이 계속 나온다. 아래 6개는 그때마다 흔들리지 않기 위한 고정 규칙이다.

1. **숫자는 규칙으로, 말은 AI로.** 추천 사이즈·여유 계산 같은 **숫자는 정해진 규칙(정해진 공식으로 계산해 언제나 같은 답이 나오는 방식)** 으로 낸다. 검증할 수 있어야 하기 때문이다. 반대로 "당신은 어깨가 넓은 편이에요" 같은 **자연스러운 문장 설명만 AI(Claude)** 가 쓴다. 추천 숫자를 AI에 맡기지 않는다 — 맞았는지 확인할 방법이 없어서다.
2. **진단은 옷 종류(카테고리) 하나씩.** 상의로 알아낸 정보로 상의 사이즈를, 하의로 알아낸 정보로 하의 사이즈를 낸다. 상의 답으로 하의를 추측하지 않는다. 몸의 부위가 다르기 때문이다(상의=어깨·가슴, 하의=허리·엉덩이).
3. **상의+하의 둘 다 있어야 체형이 완성된다.** 상의 하나만으론 상체만 아는 "반쪽 체형". 둘 다 채워야 상체+하체가 잠기고, 그때 파생 종류(아우터·원피스·치마)가 열린다. 파생은 실제 착용으로 체형을 다시 알아내지 않고, 선호하는 핏·레이어링 여유만 묻는다.
4. **진단 결과는 그때그때의 스냅샷으로 남긴다(덮어쓰지 않음).** 체중이 바뀌어 다시 진단하면, 예전 결과를 고치지 말고 **날짜를 붙인 새 결과를 하나 더 추가**한다. 과거 기록을 살려야 "이 추천이 맞았나"라는 성적표(뒤의 킬 메트릭)가 오염되지 않는다.
5. **배움은 양쪽으로.** 사용자의 착용경험은 그 사람의 체형만 알려주는 게 아니라, **그 브랜드 옷이 실제로 어떻게 나오는지**도 알려준다. 그래서 데이터를 한 방향으로만 고정하지 않는다.
6. **소재는 묻지 않고 알려준다.** 비침·무게·수축·신축은 옷 자체의 속성이라, 사용자에게 묻지 않고 브랜드 데이터에서 찾아 **결과 화면에서 경고**만 한다. 사용자에겐 "몸으로 느낀 것"만 묻는다.

**★ 킬 메트릭(성패를 가르는 단 하나의 성적표):** 사용자가 **이미 잘 아는 브랜드**로 추천을 돌렸을 때 "실제로 내가 입는 사이즈랑 맞네" 하고 동의하는 비율이 **65% 이상**이어야 한다(옷 종류별로 측정, MVP는 상의 기준). 이 숫자가 모든 우선순위와 튜닝의 기준이다. 미달이면 엔진을 다시 설계하거나 멈춘다.

## 1. 데이터 모델 / 스키마 (엔진이 다루는 정보의 형태)

> **왜 필요한가:** 엔진은 결국 두 종류의 정보를 맞대 본다 — ①브랜드 옷의 실제 치수, ②사용자가 입어보고 느낀 것. 이 정보들을 **어떤 칸에 어떻게 담을지**를 처음에 정확히 정해두지 않으면 나중에 못 고친다. 여기서 그 "칸의 모양"(스키마)을 정의한다.

두 개의 축을 기억하면 된다:
- **A축 = 브랜드가 파는 옷의 실제 치수** (물건 쪽. 예: "이 셔츠 M은 가슴 단면 52cm")
- **B축 = 착용경험으로 거꾸로 알아낸 사용자 신체 치수** (몸 쪽. 예: "이 사람 가슴둘레는 약 96cm")

이 둘의 차이가 엔진의 핵심 개념인 **여유(ease)** = 옷이 몸보다 얼마나 큰가(단위 cm). 코드는 `src/lib/schema/*`에 있다.

### 1.1 카테고리와 부위 (category.ts) — 옷 종류가 모든 것의 1급 기준

> 옷 종류마다 신경 쓰는 몸 부위가 다르다. 그래서 "옷 종류(카테고리)"를 가장 위의 기준 축으로 둔다.

- 옷 종류는 5가지: 상의·하의·아우터·원피스·치마. 다 표현하되 **실제 작동은 상의부터** 시작한다.
- 부위마다 **재는 방식(Comparison)** 이 다르고, 이게 계산법을 가른다:

| 재는 방식 | 뜻 | 계산 |
|---|---|---|
| 둘레(circumference) | 가슴·허리처럼 빙 두르는 치수 | 브랜드 표는 보통 '단면'(반으로 접어 한쪽만 잰 값)으로 적으니 **×2 해서 둘레로** 만든 뒤 몸 둘레와 비교 |
| 너비(width) | 어깨처럼 평평한 가로 폭 | ×2 안 함. 옷 폭과 몸 폭을 그대로 비교 |
| 길이(length) | 소매·총장·기장 | 여유 개념이 없다. "짧다/길다"라는 기장감으로만 본다 |

- 각 부위는 **어떤 몸 치수와 맞댈지**(BodyDimension: 어깨너비·가슴둘레·허리둘레·엉덩이둘레·허벅지둘레·팔길이·키·밑위기준)를 지정한다.
- **부위 집합:** 상의 = 어깨·가슴·배·소매·총장 / 하의 = 허리·엉덩이·허벅지·밑위·기장. 파생 3종은 자리만 잡아둠(작동은 나중).
- 부위를 쓰임새로 3분류한다:
  - **역산 대상 부위(accuracyParts):** 둘레·너비 부위. 여기서 체형을 거꾸로 알아낸다.
  - **기장 선호 부위(lengthParts):** 길이 부위. 체형이 아니라 취향·키의 문제라 역산에서 뺀다.
  - **병목 플래그 부위(PAIN_FLAGS):** 브랜드가 사이즈표에 잘 안 적지만 실제로 불편을 만드는 부위(상의=팔·목, 하의=종아리). 표에 숫자가 없으니 체형 역산은 못 하고, 대신 "꼈다/안 꼈다" 경험을 모아 **브랜드 치수를 거꾸로 추정**하는 데 쓴다. 사용자들이 자유서술에서 반복해서 말하는 새 페인은 이 목록으로 승격된다(엔진이 스스로 커짐).

### 1.2 브랜드 실측 (sizespec.ts) — A축 데이터

> 엔진의 "연료". 브랜드 옷이 실제 몇 cm인지를 담는다.

한 칸 = `{ 브랜드, 카테고리, 핏라인, 사이즈, 부위별 치수, 출처메타 }`.
- **핏라인(FitLine):** 옷의 여유 스타일. skinny(꽉)·slim·regular·loose·oversize(헐렁) 5단계. 핏라인의 정체가 곧 "이 옷이 의도한 여유량"이다.
- **부위별 치수(garmentCm):** 브랜드 표에 적힌 **원본 그대로** 저장한다(단면이면 단면 값 그대로). ×2 환산·여유 계산은 필요할 때 규칙 코드가 하지, 미리 가공해 박아두지 않는다.
- **출처메타(Provenance, "이 숫자 어디서 언제 어떻게 가져왔나"):** 출처 URL · 수집 날짜 · 방식(수기/HTML/OCR) · 신뢰도(0~1)를 함께 붙인다. 브랜드는 시즌마다 실측을 바꾸므로, 날짜가 없으면 낡은 값으로 잘못 추천하게 된다. 판매량은 비공개라 리뷰 수·인기순 같은 **대리지표**로 대신하고 "이건 대리지표"라고 표시한다(isProxy).

### 1.3 착용경험 (wearexperience.ts) — B축 입력, 4개 층

> 사용자가 입어보고 답한 것. 이걸 재료로 체형을 거꾸로 계산한다. 답을 성격별 4개 층으로 나눠 담는다(섞으면 계산이 꼬임).

앞부분(브랜드·핏라인·사이즈)은 "어떤 옷인지" 특정하는 열쇠고, 답변은 4층:

| 층 | 담는 것 | 쓰임 |
|---|---|---|
| ① 착용감(fits) | 부위별 "꽉/딱/여유/큼" | 체형 역산의 핵심 신호 |
| ② 병목 플래그(painFlags) | "팔 꼈다/괜찮다" 같은 걸림 | 체형 역산엔 안 씀. 브랜드 치수 거꾸로 추정에만 씀 |
| ③ 기장 선호(lengthPrefs) | "짧음/딱/긺" | 취향·키 → 추천 보조에만 |
| ④ 자유서술(openNote) | 예상 못 한 불편 한 줄 | 반복되면 ②로 승격 |

- **소매·다리 길이 구분(facet):** 긴팔/반팔/민소매, 긴바지/반바지. **긴팔·긴바지를 기본 권장** — 신호가 가장 많고, 팔 경험을 항상 같은 부위(이두)에서 받아 브랜드 비교가 뒤섞이지 않는다. 이 값은 질문을 켜고 끄는 데 쓰고, 체형 역산 자체엔 영향 없다.
- 시점·체중은 옷 한 벌마다가 아니라 **진단 전체(스냅샷)에 한 번** 붙인다(최근 입은 옷으로 유도하므로).
- **설문 자동 생성:** `buildCategorySurvey`(카테고리를 주면 그에 맞는 질문 묶음을 만들어 주는 함수)가 층별 질문 그룹을 만든다. 부위를 코드에 일일이 박지 않고 카테고리에서 자동으로 뽑는다. 민소매면 팔 질문, 반바지면 종아리 질문, 반팔이면 소매기장 질문을 자동으로 뺀다. 화면은 이 결과를 그대로 그리기만 한다.

### 1.4 응답 어휘 (fitrating.ts) — 설문과 엔진이 같은 단어를 쓰게

> 사용자가 고른 보기와 엔진이 계산에 쓰는 값이 어긋나면 안 된다. 그래서 어휘를 한 곳에 정의한다.

- ① 착용감 4단계: 끼임 / 딱맞음 / 여유 / 큼.
- ② 병목 판정 2단계: 꼈어요 / 괜찮았어요. — **"괜찮았어요"(문제없음)도 꼭 받는다.** 불평만 모으면 "어디까지가 괜찮은 선인지"를 못 찾기 때문이다.
- ③ 기장 선호 3단계: 짧음 / 딱 좋음 / 긺. 취향이라 역산엔 안 넣는다.

### 1.5 핏라인지도 · 브랜드순위 · 앵커브랜드 (보조 데이터)

- **핏라인지도(fitlinemap):** 브랜드가 쓰는 원본 핏 이름("오버사이즈", "511 Slim")을 우리 표준 5단계로 잇는 사전. ⚠️ **이름과 실제 여유는 다르다** — 폴로 'Classic Fit'은 이름과 달리 그 브랜드에서 제일 넉넉하다. 그래서 사람이 검수한 신뢰도를 함께 붙인다. 치마·원피스는 핏라인이 아니라 실루엣(A라인 등)이 1차 정보다.
- **브랜드순위(brandranking):** 판매량 대신 쓰는 대리지표(리뷰·인기순) 순위. 어떤 상품부터 수집할지 우선순위에만 쓰고, 추천 숫자의 근거로는 쓰지 않는다.
- **앵커브랜드(anchorbrands):** 착용경험을 **물어볼 때** 기준으로 삼는 대표 브랜드 7종. 뽑는 최우선 기준은 **오프라인 매장에서 입어보기 쉬운가**(실측 없이도 경험을 만들 수 있는 가장 넓은 입구라서). 이건 **입력용 기준 목록**이지, 추천 결과로 내놓는 브랜드 목록이 아니다(혼동 금지).

### 1.6 인체 데이터 시드 (사이즈코리아 8차) — 0벌 출발점 + 체형 라벨

> **왜 필요한가:** 착용경험이 **한 벌도 없을 때도** 출발점이 있어야 한다. "성별·키·몸무게·나이"만으로 몸 치수를 대충 깔아두고, 착용경험이 들어오면 그 위를 덮어써 좁힌다. 이 출발점을 **국가 인체치수 데이터(사이즈코리아 제8차, 2020~24)** 를 정규화해 만든다. (`sangmin/data/body/derived/`, 단위 cm/kg 통일·결측 제거. **출처표기 의무** — 국가기술표준원 사이즈코리아.)

세 개의 시드 파일이 있다:

**① 기초신체 추정기 (body-base-model) — 0벌 디폴트**
성별별 **회귀식**으로 몸을 추정한다:
```
부위 cm = a·키(cm) + b·몸무게(kg) + c·나이 + 기본값(intercept)
```
- 7부위: 젖가슴·윗가슴·허리·엉덩이 **둘레**, **어깨너비**, **팔길이**, **위팔둘레**.
- 각 부위에 **정확도 지표**(r²·오차 rmse·표본수 n)를 붙여 "이 추정이 얼마나 믿을 만한지"를 안다.
- 경향: **둘레는 몸무게가 좌우**(r²≈0.8로 잘 맞음) · **어깨는 키가 좌우**(r²≈0.4로 느슨) · **나이는 특히 허리**에 유효(10년당 +1.7cm).
- 이게 **0벌 기본값**(부위당 ±2~3.5cm). 착용경험 역산(§3-[2])이 그 위를 덮어써 **±2cm로 좁히는 게 목표**.

**② 분포 (body-distribution) — 신뢰도·위치 표시**
성별×부위의 평균·표준편차·백분위(p5·p50·p95). 추정치가 **모집단 어디쯤인지(P##)** 를 알려줘 신뢰도 배너·스펙트럼 막대의 위치 표기에 쓴다.

**③ 체형 라벨 (archetypes) — 숫자를 캐릭터로**
표준체형 38 + 비만체형 32 = **70종**의 실측 부위벡터(비만 32종엔 설명문 — AI 캐릭터 서술의 근거). **역산이 끝난 뒤** 사용자의 부위벡터를 가장 가까운 아키타입으로 **스냅**해 결정론적 체형 라벨을 붙인다.
- ⚠️ **8유형 카드(`web/data/bodytypes.json`)와는 다른 층이다** — 저건 스타일링 표현용 캐릭터 8종, 이건 역산 결과를 맞대볼 실측 라벨 70종.
- ⚠️ **0벌 단계 분류엔 못 쓴다**(체형이 키·몸무게만으론 안 갈린다). 반드시 역산 뒤에.
- ⚠️ 체형명("역삼각/삼각")은 어깨↔엉덩이 실루엣이 아니라 **드롭·부피·신장체중 비례** 분류다 → 이름을 곧이곧대로 실루엣에 매핑 금지(어깨/엉덩이 비율은 38종 전부 0.37~0.43로 거의 일정).

## 2. 입력 스펙 — 무엇을 어떻게 묻나

> **왜 필요한가:** 좋은 역산은 좋은 질문에서 나온다. 자유서술을 받으면 계산할 수 없으니, 전부 **보기에서 고르는 형태**로 받고, 몸을 위에서 아래로 훑으며 묻는다. 질문이 많아 보이지 않게 티어(단계)로 나눈다.

**3티어:** T1=모두 답(필수) · T2=신경 쓰인 부위만 더 답(정확도) · T3=원하면 더(심화, "답하면 정확도 +N%").

- **Step 0 — 옷 종류 선택:** 여기서 고른 카테고리가 이후 질문·부위·추천을 전부 좁힌다.
- **Step 1 — 기준옷 특정 + 기본값:** 브랜드(앵커 7종) · 핏라인 · 하위타입(긴팔/반팔/민소매, 긴바지/반바지/크롭) · 사이즈 · 전체 느낌(작음/딱/큼) · 키(기장·이상치 판단 전용, 둘레 판단엔 안 씀) · 체중(선택, 재진단 판단용) · 착용시점(자동, 안 물음).
- **Step 2 — 위→아래 몸 스캔:**
  - **상의:** 목 → **어깨(필수)** → **팔·겨드랑이(필수)** → (낀 곳이면) 팔뚝/겨드랑이 → 소매길이(긴팔만) → 가슴 → 앞뒤품 → 배 → 밑단.
  - **하의:** **허리↔엉덩이(필수)** → 앞주머니 벌어짐 → 뒤밑위(앉을 때 파임) → 앞밑위 → 허벅지 → 밑단(긴바지만).
  - 하위타입에 안 맞는 질문은 자동으로 건너뛴다(반팔이면 소매길이 skip). 덤으로 브랜드 실측 해석도 정확해진다(반팔 소매 스펙 ≠ 긴팔).
- **Step 3 — 선호핏:** 원하는 전체 핏(타이트/정핏/루즈/오버) · 원하는 기장.
- **Step 4 — 결과:** 부위별 번역(추천 사이즈+예상 핏) · 옷속성 경고 · **"맞았나?" 피드백**(킬 메트릭 수집). 질문이 많아 보이지 않도록: "전체 편했어요" 한 번으로 통과하는 지름길, 원탭 보기, 답할 때마다 작은 인사이트 보상, 앵커 재사용으로 반복 제거(불편 없던 사용자는 4번 탭이면 끝).

## 3. 역산·추천·경고 규칙 (엔진의 두뇌)

> **왜 필요한가:** 여기가 엔진의 심장이다. "이 옷이 이렇게 느껴졌다"는 신호를 → 체형 숫자로 바꾸고 → 다른 옷 사이즈로 번역하고 → 위험을 경고하기까지의 **정해진 계산 절차**다. 전부 규칙(정해진 공식), 자연어 한 줄만 AI다.

전체 흐름을 말로 풀면: **①안 맞는 질문 걸러내기 → ②부위마다 신체 추정 → ③체형 카드 조립 → ④충돌 부위 정리(병목) → ⑤목표 옷에 맞는 사이즈 찾기 → ⑦옷 속성 경고**, 옆에서 **⑥브랜드 치수 보정 · ⑧시점 관리**가 돕는다.

**모든 계산의 기본 공식(말로):** 여유 = 옷 치수 − 몸 치수. 뒤집으면, 옷 치수를 알고 "얼마나 여유 있었는지"를 경험으로 알면 → **몸 치수 = 옷 치수 − 경험한 여유**로 몸을 거꾸로 알아낼 수 있다. 이게 **역산(逆算, 결과에서 원인을 거꾸로 계산하기)** 이다.

### [1] 하위타입 걸러내기 (subtype 게이팅, "안 맞는 질문은 계산에서 뺀다")
성립 안 하는 신호는 역산에서 빼고, 대신 **§1.6 기초 추정기(성별·키·몸무게·나이 → 부위 cm)로 채운다**(prior). 착용경험이 0벌이면 이 prior만으로 카드를 만들고 신뢰도를 "낮음"으로 표시한다.

| 하위타입 | 무시할 신호 | 대체 |
|---|---|---|
| 반팔 | 소매길이 | 팔길이 ← 키로 추정 |
| 민소매 | 소매길이·팔뚝 걸림 | 겨드랑이(암홀)만 유효 |
| 반바지 | 밑단 상태 | 기장 ← 키로 추정 |

### [2] 부위별 역산 규칙 (신호 → 신체 추정)

> **δ(델타)는 "한 칸 조정폭"이다.** 예를 들어 "가슴이 꼈다"면 그 사람 가슴이 옷보다 δ만큼 크다고 본다. δ의 정확한 값은 초기엔 모르니, 실제 데이터로 맞춰가는 **튜닝 값**이다.

말로 읽는 예: "어깨 봉제선이 어깨 끝을 넘어갔다"= 옷 어깨가 내 어깨보다 크다 → 내 어깨너비 = 옷 어깨 − δ. "겨드랑이가 자주 꼈다"= 필요한 겨드랑이 공간이 실제보다 크다(자주=2칸, 가끔=1칸).

| 신호 | 규칙(말풀이) | 알아내는 값 |
|---|---|---|
| 전체 느낌 | 작았으면 몸 +δ, 컸으면 −δ | 전역 기준선 |
| 어깨 봉제선 | 넘어감→내 어깨 = 옷−δ / 안쪽→+δ / 딱→같음 | 어깨너비 |
| 겨드랑이 걸림 | 가끔=1·자주=2 → 필요 공간 = 옷+계수k×정도 | 겨드랑이(암홀) 요구치 |
| 가슴 | 꼈으면 가슴 = 옷+δ, 남으면 −δ | 가슴둘레 |
| 배 | 같은 방식 | 배둘레 |
| 소매길이(긴팔) | 짧았으면 팔 김(+δ), 길었으면 −δ | 팔길이 |
| 앞뒤품 당김 | 당겼으면 품 부족 표시 | 앞뒤품 |
| 허리↔엉덩이 | 엉덩이 꼈으면 엉덩이가 표준보다 큼 / 허리 남으면 낙차 큼(반대) | 허리-엉덩이 낙차 |
| 앞주머니 벌어짐 | 벌어졌으면 하이힙 볼륨 큼 | 앞판 여유 요구 |
| 밑위(앉을 때 파임/처짐) | 파임·처짐→밑위 요구 큼 | 앞·뒤밑위 |
| 허벅지 | 꼈으면 +δ, 남으면 −δ | 허벅지둘레 |
| 밑단(긴바지) | 끌리면 기장 김, 짧으면 짧음 | 총장 보정 |

**가슴과 배 조합으로 체형 유형 읽기(말로):** 가슴은 맞는데 배만 꼈으면 → 중동(윗배) 볼륨형. 가슴만 꼈으면 → 상체 발달형. 둘 다 남으면 → 마른형/루즈 선호. 둘 다 맞으면 → 표준.

**"꽉/딱/여유/큼"을 cm로 바꾸는 표 (EASE_BANDS, "느낌↔센티 환산표"):**
사용자는 cm를 모르니 "딱맞음" 같은 말로 답한다. 이 표는 그 말을 **여유 cm 구간**으로 바꿔준다. 예를 들어 상의 가슴은 여유 2cm 이하면 "끼임", 8cm까지 "딱맞음", 16cm까지 "여유", 그 이상은 "큼"으로 본다. 반대로 사용자가 "딱맞음"이라 답하면 "여유가 2~8cm였구나"로 되짚어 몸을 계산한다.
- **부위마다 폭이 다른 이유:** 둘레 부위(가슴·배)는 몸을 빙 둘러서 숫자가 커 폭이 넓고, 어깨(너비)는 절반 스케일이라 폭이 좁다. 그래서 표의 키가 `(옷종류:부위)`다.
- ⚠️ **이 숫자들은 전부 초기 짐작값이다.** 사용자 피드백이 쌓이면 고친다(킬 메트릭이 판정관). 길이 부위는 여유 개념이 없어 이 표에서 뺀다.
- 현재 짐작값(참고): 상의 가슴 2/8/16 · 배 0/8/18 · 어깨 0/1.5/4 / 하의 허리 0/4/10 · 엉덩이 0/6/14 · 허벅지 0/5/12 (하의는 아직 검증 안 된 임시값).

### [3] 체형진단 카드 조립 (상·하의 결과를 한 장에)

상의 진단 결과(어깨·가슴·배·겨드랑이 등)와 하의 결과(허리-엉덩이 낙차·허벅지·밑위 등)를 한 카드에 모은다. 카드엔 이런 게 들어간다:
- **효력일자(effective_date):** 이 진단이 언제 것인지. 체중 바뀌면 고치지 말고 **새 날짜로 한 장 더** 만든다.
- **체중 버전(weight_version):** 체중이 바뀔 때마다 올라가는 번호.
- **upper/lower:** 상·하체 부위별 판정과 각 신뢰도.
- **completeness:** 상·하의가 각각 완료됐는지 → 파생 종류를 열지 판단하는 문.
- **체형 라벨:** 역산된 부위벡터를 §1.6 아키타입(70종)에 **스냅**해 결정론적 라벨을 얻는다. `character` 한 줄은 이 라벨(+비만군 설명문)을 **근거로만** AI가 쓴다 — 숫자를 지어내지 않게.
- **character:** "어깨 넓고 하체 볼륨형" 같은 **한 줄 설명. 여기만 AI가 쓴다.** 나머지 숫자는 전부 규칙.
- 상의만 끝났으면 상체 칸만 채우고, 상체 계열 파생(아우터)만 부분적으로 열린다.

### [4] 병목 판정 (충돌하는 부위 정리)

> **왜 필요한가:** 어깨는 딱 맞는데 겨드랑이가 부족한 옷처럼, 부위마다 요구가 엇갈릴 때가 많다. 다 만족시키는 사이즈는 없으니 결정이 필요하다.

**규칙(말로):** 각 부위에서 "필요한 치수 − 그 사이즈의 실제 치수"를 계산해 **가장 크게 모자란 부위(=병목)** 를 찾는다(argmax = 값이 가장 큰 걸 고른다는 뜻). 그 병목 부위를 만족시키는 **가장 작은 사이즈**를 추천하고, 나머지 부위는 "여기는 여유/부족"이라고 경고로 알린다. 이건 사람들이 실제로 쓰는 요령("제일 걸리는 데 맞추고 허리는 벨트로")과 같다.

### [5] 목표 옷 추천

사고 싶은 옷 T의 실측을 놓고: 부위별 **목표 = 내 몸 + 원하는 여유**를 잡는다. 겨드랑이 같은 **꼭 지켜야 할 제약**을 만족하는 사이즈만 후보로 남기고, 그중 목표에 가장 가까운 걸(병목 우선) 고른다. 출력은 추천 사이즈 + 부위별 예상 핏 + 경고. **추천은 항상 같은 옷 종류 안에서만** 한다(상의 입력→상의 사이즈).

### [6] 브랜드 치수 거꾸로 보정 (양방향 학습)

> **왜 필요한가:** 브랜드 사이즈표가 틀리거나 빠진 경우가 많다. 사용자 경험으로 그걸 고칠 수 있다.

스펙상 정사이즈인데 "겨드랑이가 자주 꼈다"는 사람이 여럿이면 → 그 브랜드·핏라인의 **표에 없는 부위(겨드랑이·앞뒤품) 치수를 거꾸로 추정해 갱신**한다. 즉 사람 체형(B)만이 아니라 브랜드 실측(A)도 배운다. 이게 쌓이면 §4의 "숨은 치수" 칸이 데이터로 채워지고, 이게 경쟁자가 못 따라오는 **해자(방어벽)** 가 된다.

### [7] 옷속성 경고 (묻지 않고 알려줌)

옷의 속성에서 자동으로 경고를 띄운다:

| 속성 | 경고 조건 | 문구 예 |
|---|---|---|
| 비침 | 밝은색·얇은 원단 | "비침 있음 — 이너 권장" |
| 신축 없음 | 스판 없고 타이트 추천 | "스판 없어 여유 빠듯할 수 있음" |
| 무게·부피 | 롱패딩 등 | "무게로 활동성 제약 가능" |
| 수축 | 면100%·생지 | "세탁 후 수축 가능 — 재진단 권장" |

기준옷 소재도 데이터에 있으면 조회하고, 없을 때만 묻는다.

### [8] 시점·재사용 관리

착용 시점은 자동으로 지금 시각. 체형 카드는 **체중이 그대로일 때만** 재사용하고, 바뀌면 끊고 새 버전을 만든다. 단 브랜드 치수 보정(A)에는 과거 입력도 계속 쓴다(옷 치수는 내 체중과 무관하니까).

### 튜닝 파라미터 (실측정으로 맞춰가는 값들)
δ(부위별 조정폭)·k(겨드랑이 요구 계수)·원하는 여유 기본값·병목 임계값. 시드 3브랜드 × 상·하의로 "신호→추천"이 서는지 보며 조정한다. **조정의 기준은 항상 킬 메트릭(≥65%).**

## 4. 숨은 치수 레지스트리 (거꾸로 추정해야 할 부위 목록)

> **왜 필요한가:** 브랜드 사이즈표에 없어서 "입어보기 전엔 모르는" 부위가 있다. 바로 여기서 불편이 생긴다. 이 목록이 엔진이 공략할 진짜 기회이자 방어벽이다.

**핵심 정의:** 맞춤옷은 재는데 기성복 사이즈표엔 없는 치수 = 숨은 치수. 표에 없으니 여유를 계산할 수 없고 → 예상 못 한 불편이 생긴다. (표에 있는 숫자의 "크다/작다"는 이미 풀린 문제라 제외.)

**추정 난이도 3등급:**
- **(A) 사용자가 직접 답할 수 있는 것** — 온라인 맞춤 폼도 받는 것(소매통·목둘레). → 바로 MVP 입력에 넣을 수 있음.
- **(B) 전문 재단사만 아는 것** — 겨드랑이·화장·앞뒤품·중동·어깨경사·자세. → 브랜드 치수를 **거꾸로 추정해야 하는 진짜 해자 구간.**
- **(C) 아무도 안 재는 것** — 무릎·종아리둘레. → 불편은 실재하나 어디서도 못 구해 추정 난이도 최상.

**상의 우선순위:** ①겨드랑이(암홀) ②어깨경사·봉제선 ③원단 두께·비침·신축. 배둘레는 맞춤 폼은 받는데 기성복이 거의 항상 빼는 항목이라 **가성비 좋은 1급 후보**.
**하의 핵심:** ①앞뒤 밑위 분리 ②허리-엉덩이 낙차·밴드 곡률 ③스판·소재.

코드에서는 이 목록이 §1.1의 **병목 플래그(PAIN_FLAGS)** 로 표현되고, 자유서술에서 반복되는 페인이 정식 부위로 승격된다(엔진이 스스로 커지는 성장 구조).

**이게 왜 방어벽인가:** 기성복 사이즈표는 신뢰도 자체가 낮다(표기 73cm인데 실측 67cm 같은 사례, 법적 기준도 없음). 그래서 브랜드 표만 믿으면 위험하고, 착용경험 역산(B축)이 꼭 필요하다. 사용자들은 이미 실측을 못 믿고 "나는 어깨가 있으니 한 치수 크게" 식으로 체형 번역을 하고 있다 — 우리 서비스 논리와 같다. (상세 후기·출처 근거는 프로젝트 리서치 아카이브에 있음.)

## 5. 엔진 계약 (엔진과 화면 사이의 약속)

> **왜 필요한가:** 엔진 속을 몰라도 화면이 엔진을 쓸 수 있으려면, "이렇게 넣으면 이렇게 나온다"는 **딱 하나의 약속(계약)** 이 필요하다. 이 약속만 지키면 안쪽 엔진을 나중에 통째로 바꿔도 화면은 그대로 돌아간다.

**진단 함수(diagnose)의 약속(말로):**
- **넣는 것:** ①기본정보(성별·나이·키·체중) ②카테고리별 선호핏 ③착용경험 목록.
- **나오는 것:** ①체형 카드(§3-[3]) ②추천 목록(브랜드·사이즈·병목·경고, §3-[4~5]) ③신뢰도 등급(입은 옷 0벌=낮음 / 1벌=보통 / 2벌 이상=높음).
- **철칙:** 숫자·판정은 규칙, `character` 한 줄만 AI. **화면은 이 약속만 알고, 엔진 내부(느낌↔센티 환산표·δ 같은 튜닝값)는 들여다보지 않는다.**

**피드백 로그(킬 메트릭 수집기):** 결과 화면에 항상 붙는다 — "실제 내 사이즈랑 맞나?(맞음/애매/틀림)" + "안 사본 브랜드도 사보겠나?" + 자유 의견. 이게 없으면 65%를 잴 수 없으므로 처음부터 심는다.

**엔진 출력 → 화면 대응** (화면 상세는 [2_화면과흐름.md](2_화면과흐름.md)):

| 엔진 출력 | 화면 |
|---|---|
| 상체 판정 × 하체 판정 | 8가지 체형 유형 코드 → 체형 카드 |
| 추천 목록 | 브랜드별 추천 사이즈표 + 경고 문구 |
| 신뢰도 등급 | 신뢰도 배너(0벌 낮음/1벌 보통/2+ 높음) |
| 부위별 수치 | 4축 스펙트럼 막대 |
| 완료 여부 | 진단현황(옷장)·파생 해금 표시 |

상의만 끝났으면 8유형은 아직 미확정 → "상체 부분"으로 표시한다.

**프로토타입에서의 임시 대체:** 지금은 HTML 프로토타입이라 엔진 계산을 **목업 어댑터(가짜 엔진, `app/js/engine-mock.js`)** 가 대신한다 — 위 약속과 똑같은 모양으로 미리 정해둔 카드·추천·신뢰도를 돌려준다. 화면은 진짜 엔진과 똑같이 호출하므로, 나중에 이 어댑터만 진짜 엔진으로 갈아끼우면 된다.

## 6. 빌드 순서 / 현재 상태

> **왜 이 순서인가:** UI를 먼저 예쁘게 만들고 싶은 유혹을 눌러야 한다. 엔진이 틀리면 아무리 예쁜 화면도 소용없다. 그래서 **데이터 → 엔진 → 화면** 순으로, 가설의 심장부터 친다.
> 비유: A(브랜드 실측)=연료, B(번역 엔진)=심장, C(프론트)=엔진이 맞는지 재는 계기판.

| 단계 | 만드는 것 | 통과 조건 |
|---|---|---|
| Phase 0 | 타당성 실험(간단 스크립트, 상의 1종 × 브랜드 3) | 역산 정확도(±1 사이즈)가 의미 있게 나오나 |
| Phase 1 | 스키마 + 시드 데이터 + 변환규칙 | 브랜드 3~5개 정규화 완료 |
| Phase 2 | 번역 엔진 + 자동 테스트 | Phase 0 사례가 계속 통과 |
| Phase 3 | MVP 화면 + 검증 루프 | 킬 메트릭을 실제로 잴 수 있음 |

**데이터 파이프라인(A축 채우기, 브랜드 치수 모으기):** 진짜 어려운 건 "모으기"가 아니라 **"제각각인 표들을 하나의 통일된 형태로 정리하기"**(작업의 80%). 순서: 수집(브랜드별 맞춤 수집기 + robots.txt 확인) → 추출(이미지면 OCR로 글자 인식) → 정규화(명칭·단위 통일 + 여유 계산) → 검증(이상치·사람 눈검수) → 저장(신뢰도·시점 함께) → 분석(표준체형과 비교). "완전 자동 크롤러"는 환상이고, 표적은 봇 차단이 심한 플랫폼(무신사 등)이 아니라 **브랜드 자사몰**이다. 유니클로 상의 하나만 끝까지 뚫어본 뒤 넓힌다(브랜드와 옷 종류를 동시에 벌리지 않는다).

**현재 성숙도(솔직한 상태):**
- ✅ 단면↔둘레 환산, 여유 계산, 느낌↔센티 변환의 **기본 산수는 동작한다.**
- ✅ **인체 데이터 시드(§1.6)는 정규화 완료** — 사이즈코리아 8차 → 0벌 회귀 추정기·분포·아키타입 70종(`sangmin/data/body/derived/`). ⚠️ 단 아직 목업 카드에 **연결 안 됨** — garmentCm 수집을 기다리지 않고 **0벌 체형카드를 실계산으로 바꿀 수 있는 가장 이른 지점**이다(회귀 추정기만으로 가능).
- ⚠️ §3-[1~8]의 역산·추천 절차는 **문서 설계만 됐고 아직 구현 안 됨** → 진짜 연결은 Next.js 단계에서.
- ⚠️ **브랜드 부위 실측(garmentCm)이 아직 안 모였다** → 추천 실계산 불가(그래서 지금은 목업). 실계산 전 유니클로 상의부터 수집.
- ⚠️ **하의 설문·느낌↔센티 표는 아직 검증 안 된 임시값** → 하의 정식화는 별도 작업.
- 정확도 분석은 체형 버전(체중)별로 나눠서 본다 — 체중 변화가 정확도 신호를 오염시키지 않게.

**아직 못 정한 질문:** 옷 몇 벌·어느 부위까지 받아야 오차가 실용 수준(±2cm)에 드나 / 실착 보정을 언제부터 섞나 / 느낌↔센티 표 초기값을 어떻게 잡나 / 페이크도어 클릭의 허수 보정.
