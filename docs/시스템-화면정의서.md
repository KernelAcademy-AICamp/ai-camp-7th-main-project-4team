# 핏팅 — 시스템 화면 정의서 (전체)

> **이 문서는 무엇?** 시스템의 **모든 화면**(사용자앱·전문가앱·운영어드민·공통)을 한 곳에서 **풀뎁스로** 정의합니다 — 화면별 목적·와이어·요소·**진입/이탈·데이터 입출력(저장소 키 계약)·상태 분기·소유·SEO**.
> 상위 정본: 화면 목록·URL·IA코드는 [2_정보구조.md](2_정보구조.md), 화면 사이 이동은 [3_사용자플로우.md](3_사용자플로우.md), 앱 화면의 "왜"는 [4_화면정의.md](4_화면정의.md), 쇼퍼가입 딥다이브는 [쇼퍼가입-화면정의서.md](쇼퍼가입-화면정의서.md), 진단 질문 정본은 [6_사이즈엔진.md](6_사이즈엔진.md), 8유형 데이터는 `web/data/bodytypes.json`.
> **구현상태 정본은 [README.md](README.md)** — 이 문서는 화면 설계·데이터 계약을 담습니다.
> **시각 정본(화면·순서·게이트)은 [스토리보드](html/스토리보드.html)·[와이어프레임](html/와이어프레임.html)** — 이 문서는 MVP 구현 화면의 데이터 계약 중심이고, 실매칭 전주기(요청→입찰/지명→**결제(에스크로)→진행·결과물→완료·정산→분쟁**)와 **고객지원**(고객센터 1.9·알림 G.6·개인정보 1.4E·FAQ G.2·양방향 신고)·관리자 거래운영(3.B.4~7)은 v2 타깃으로 스토리보드에 확정돼 있습니다. 흐름은 [3_사용자플로우.md](3_사용자플로우.md) §4·§5.

## 0. 저장소 키 계약 (전 화면 공통 · SSOT)

모든 화면은 브라우저 저장소(`fitting.*` 프리픽스)로만 상태를 주고받습니다(백엔드 없음 — 마일스톤1에서 Supabase로 승격). **한 화면이 쓴 키를 다른 화면이 읽는 지점이 곧 데이터 계약**입니다.

| 키 | 저장소 | 쓰기(소유) | 읽기 | 내용(shape) |
|---|---|---|---|---|
| `fitting.basic` | session | diag-basic | diag-fit · result | `{gender, age, height, weight}` |
| `fitting.dx` | session | diag-fit | result | `{basic, prefs{cat:fitLine}, experiences[]}` |
| `fitting.consent` | session | result | result | `{engineImprove, ageAttested, at}` |
| `fitting.feedback` | **local** | result | *(admin-diagnostics)* | `[{ts, bodyType, verdict, confidenceTier, engineImprove, ageAttested}]` |
| `fitting.auth` | local | index | index | 로그인 상태(lazy) |
| `fitting.favs` | local | index | index | 찜한 쇼퍼 목록 |
| `fitting.reqs` | local | index | index · *(admin-members)* | 코디 요청 = **매칭 수요 신호** |
| `fitting.reqsVer` | local | index | index | 데모 시드 마이그레이션 버전 |
| `fitting.pro.profile` | local | pro-signup · pro | pro · pro-quote | 쇼퍼 프로필 `{registered, name, services[], tagline, bio, specialties[], portfolio[]}` |
| `fitting.pro.reqs` | local | pro · pro-quote | pro · pro-quote | 쇼퍼가 받은 요청 `{cust, type, bodytype, cm, kg, occ, budget, status, offer?, review?}` |
| `fitting.pro.proposed` | local | pro | pro | "먼저 제안" 발송 목록 |
| `fitting.admin` | session | admin-login | admin-common(가드) | 관리자 세션 `{email}` |

> **주의 — 계약의 빈틈:** ① 진단이 수집한 `fitting.dx`/`feedback`은 현재 **어드민에서 읽히지 않음**(진단응답 화면은 샘플 렌더). ② 사용자 `fitting.reqs`와 쇼퍼 `fitting.pro.reqs`는 **분리된 저장소**라 아직 연결 안 됨(데모). 실배선은 마일스톤1.

```iatree
★ | 앱 | 사용자 앱 (탭 4개 · index SPA + 진단 플로우)
  ★ | 탭1 | 🏠 홈 | 랜딩·진단 유도
  ★ | 탭2 | 🔍 체형진단 | basic→fit→loading→result→(card)
  ★ | 탭3 | 👔 쇼퍼찾기 | 페이크도어(목록·상세·견적)
  ◐ | 탭4 | 👤 마이 | 프로필·진단·요청·찜·계정
◐ | 전문가 | 공급자 앱 (별도)
  ◐ | 2.0 | 쇼퍼 가입 위저드 | pro-signup SP-0~5
  ◐ | 2.1 | 쇼퍼 포털 | pro (대시·인박스·제안·프로필·후기·정산)
  ◐ | 1.3.7 | 견적서 | pro-quote (체형 상세+제안)
★ | 어드민 | 운영 어드민 (3.A~3.E · 프로토타입)
  ★ | 3.A | 사이즈·데이터 | admin/brands/garments/collect/engine
  ★ | 3.B | 회원·거래 | members (회원·전문가·매칭신호)
  ★ | 3.C | 사업현황 | business·diagnostics
  ○ | 3.D | B2B API | api
  ○ | 3.E | 내부운영 | ops
★ | 공통 | about·privacy·terms·_reset·admin-login
```

## 1. 사용자 앱 — 탭1~4 (`index.html` SPA + 진단 플로우)

> 탭 전환은 `go(id)`(#home/#shop/#my `.on` 토글) · `goMy(panel)`. 로드 시 `location.hash`와 `?from=CODE`(초대 배너)를 읽어 자동 진입. 진단 플로우(basic·fit·loading·result)는 **별도 페이지**.

### 탭1 · 🏠 홈 (`index.html#home`) ★ 🔍

- **목적:** 히어로 줄자 애니메이션으로 3초 안에 "쓸모"를 전달하고 진단을 시작시키는 랜딩.

```
┌───────────────────────────────────────────┐
│ Fitting▪   홈 쇼퍼찾기 마이     👤 쇼퍼지원 │  ← .menu / .navr
├───────────────────────────────────────────┤
│  [줄자 히어로]  #hMarker #hCm #hKg #hMtag   │
│  "브랜드마다 다른 내 사이즈, 한 번에"        │
│        [ 내 사이즈 진단 시작 ] → diag-basic │
│                                           │
│  작동 방식 3스텝  (.reveal/.hcount)         │
│  결과 미리보기  #hTypes .ty (card iframe)   │
│  상황별 쇼퍼 매칭 데모  #hOccRow #hShopNode │
└───────────────────────────────────────────┘
```

| 요소 | 표시 | 동작 |
|---|---|---|
| 히어로 CTA | [내 사이즈 진단 시작] | `startDiag()` → `fitting.dx/basic/consent` 삭제 후 `diag-basic.html` |
| 헤더 쇼퍼 지원 | 링크 | → `pro-signup.html` |
| 결과 미리보기 | 8유형 카드 iframe | `card.html?type=..&g=..` |
| 쇼퍼 매칭 데모 | 하드코딩 `SH`/`FLOW` | 시연용(비상호작용) |
| 초대 배너 `#inviteBanner` | `?from=CODE` 있을 때 | 친구 초대 유입 표시 |

- **데이터:** R `fitting.auth`·`fitting.favs`·`fitting.reqs` / fetch `data/bodytypes.json`(아바타 색)
- **상태:** `prefers-reduced-motion` → 애니메이션 건너뛰고 종료상태 / `?from` → 초대 배너
- **이탈:** diag-basic · `go('shop')` · pro-signup · about/terms/privacy(푸터)

### 탭2 · 🔍 체형진단 — 서비스의 심장

진단은 **동의 게이트 없이** 바로 시작(서비스 제공 최소 수집). 위저드: 2-1 → 2-2 → 2-3 → 2-4 → (2-5 피드백/2-6 전환/2-7 공유).

#### 2-1 · 기본 정보 (`diag-basic.html`) ★ 🔒

- **목적:** 성별·나이·키·몸무게 수집. 성별 선택 시 평균값 프리필.

```
┌──────────────────────────────┐
│  기본 정보                    │
│  성별  ( 남성 ) ( 여성 )       │  radio
│  나이  [20대 ▾]               │
│  키    [173cm ▾]              │
│  몸무게 [72kg ▾]              │
│              [ 다음 ]         │ → diag-fit
└──────────────────────────────┘
```

| 요소 | 입력 | 검증·동작 |
|---|---|---|
| 성별 | radio(필수) | 선택 시 나이·키·몸무게 평균 프리필 |
| 나이/키/몸무게 | select | 원문자열 저장(`'20대'`,`'173cm'`,`'72kg'`) |
| [다음] | 주 버튼 | `saveBasic()` → **W `fitting.basic`** `{gender,age,height,weight}` → diag-fit |

- **진입:** 홈/about CTA(진입 시 이전 세션 진단키 초기화) · **이탈:** diag-fit

#### 2-2 · 착용경험 보정 (`diag-fit.html`) ★ 🟢실연결 🔒

- **목적:** 정확도의 핵심. 대상 카테고리·선호핏 + 입어본 옷(브랜드·핏·사이즈)×1~3 + 부위별 느낌. **질문 정본은 엔진**([6_사이즈엔진.md](6_사이즈엔진.md)).

```
┌────────────────────────────────────────┐
│ 무엇을 진단할까요?                        │
│ [상의][하의] [아우터][치마][원피스]        │  카테고리
│ 선호핏  #prefseg (스키니…오버)            │
│ ── 입어본 옷 1 ──                         │
│ 브랜드[유니클로▾] 핏/품목[▾] 사이즈[M▾]   │  앵커(#brand1/item1/size1)
│ 부위별 느낌: 어깨[딱맞음] 가슴[여유]…      │  fits/painFlags/lengthPrefs
│ 자유메모 [………]         (+ 옷 더하기)     │
│              [ 진단하기 ]                │ → diag-loading
└────────────────────────────────────────┘
```

| 요소 | 입력 | 동작 |
|---|---|---|
| 진단 대상 | 카테고리 택1 | 기반(상·하의)=실측 역산 / 파생=선호핏만 |
| 선호핏 `#prefseg` | 세그먼트 | 카테고리별 `prefs`에 저장 |
| 앵커 옷 ×1~3 | 브랜드+핏+사이즈 | 브랜드는 앵커 6종 datalist |
| 부위별 느낌 | fits/pain/length | `SNUG/RELAXED…`, `OK/TIGHT`, `GOOD/LONG…` |
| 하의 밴드 | 토글 | `waistband:'none'|'banded'` |
| [진단하기] | 주 버튼 | **R `fitting.basic`, R/W `fitting.dx`** → diag-loading |

- **데이터:** R `fitting.basic`(성별→사이즈/브랜드 렌더) · R/W `fitting.dx` `{basic, prefs, experiences[]}`
- **상태:** 0벌=건너뛰기(신뢰도↓) / 1벌=보통 안내 / 브랜드 없음=직접입력 / 파생 카테고리=선호핏만
- **experiences[] shape:** `{category, brandId, brandName, fitLine, item, sizeLabel, subtype, gender, silhouette?, waistband?, fits{}, painFlags{}, lengthPrefs{}, openNote}`

#### 2-3 · 진단 로딩 (`diag-loading.html`) ★ 🔒

- **목적:** 역산+매칭 계산 대기(줄자 로딩 애니메이션). URL 파라미터(`skip/g/derived/cat`)만 읽고 result로 리다이렉트. **저장소 미접근.**
- **이탈:** result.html

#### 2-4 · 결과 카드 (`result.html`) ★ 🟢실연결 🔒

- **목적:** 제품의 얼굴. ① 8유형 체형 프로필 ② 핏 인사이트 ③ 브랜드별 추천 사이즈 ④ 옷장(진단 현황). 상의만 실계산, 파생은 선호핏 폴백/게이트.

```
┌───────────────────────┬───────────────────┐
│  8유형 카드(어두운 배경) │  브랜드별 추천 사이즈 │
│   실루엣 + 시그니처색   │  핏지수% · 병목부위  │
│   핏 인사이트           │  체형 측정 스펙트럼  │
├───────────────────────┴───────────────────┤
│ 이 추천 맞아요?  ✅맞아요 ➖애매 ❌틀려요    │ ← 정확도 검증(킬 메트릭)
│ ☐ 익명 엔진개선 동의(만14세)                │
│ [옷장: 상의✓ 하의○]  [전문가에게 받기 →]    │
└─────────────────────────────────────────────┘
```

| 요소 | 표시 | 동작 |
|---|---|---|
| 8유형 카드 | `bodytypes.json`(성별 축) | `btResolve(t,g)` |
| 브랜드 추천표 | 엔진 계산(FitEngine) | 핏스코어·병목 |
| **정확도 검증** `.rseg` | 맞음/애매/틀림 | `fb()` → **W `fitting.feedback`**(local) = 킬 메트릭 원천 |
| 엔진개선 동의 `#eiConsent` | 체크박스(선택) | **W `fitting.consent`** `{engineImprove,ageAttested,at}` |
| 전문가 전환 | CTA | → `index.html#shop` |
| 재시작 `resetDiag()` | — | `fitting.dx/basic/consent`(session)+`feedback`(local) 삭제 |

- **데이터:** R `fitting.dx`·`fitting.basic`·`fitting.consent` / W `fitting.consent`·`fitting.feedback` / fetch `bodytypes.json`
- **상태(신뢰도 3단계):** 0벌=낮음(주의 배너)·회귀 추정 / 1벌=보통 / 2~3벌=정확 · 데이터 없음→`renderNoData()` · 로드실패→`renderLoadError()` · 8유형 상세는 상·하의 완료(cardReady) 시에만
- **이탈:** `index.html#shop`(전문가) · `index.html`(저장) · card(iframe) · diag-fit(재진단)

#### 2-7 · 결과 공유 카드 (`card.html`) ◐ 🔒

- **목적:** 프로필·진단결과·요청내역·찜·계정. MVP는 "저장하려는 사람"만.
- **사이드 패널:** `mp-profile · mp-diag · mp-req · mp-fav · mp-acct` (`myNav`)

## 2. 전문가(공급자) 앱 — 별도 앱 ◐

> 사용자 앱과 화면·로그인이 완전히 분리. 홈 "쇼퍼 지원"으로 진입. 딥다이브: [쇼퍼가입-화면정의서.md](쇼퍼가입-화면정의서.md), [7_전문가매칭.md](7_전문가매칭.md).

### 2.0 · 쇼퍼 가입 위저드 (`pro-signup.html`) ◐ 🔍

- **목적:** 쇼퍼 온보딩 6단계(SP-0 소개 → 1 계정 → 2 기본정보 → 3 이력 → 4 포트폴리오 → 5 완료) → 포털 진입.
- **게이트:** `fitting.pro.profile.registered` 있으면 `pro.html`로 즉시 리다이렉트.
- **단계별 요소:** 1 계정(`#email/#agreeReq`, 검증) · 2 서비스가격(온라인35k/동행120k/이미지90k 토글) · 3 이력(태그 min1) · 4 포트폴리오(dataURL max8, "나중에" 허용)
- **데이터:** **W `fitting.pro.profile`** `{registered:true, name, services[{type,label,price}], tagline, bio, specialties[], portfolio[]}`
- **이탈:** `index.html`(닫기) · `pro.html`(완료)

### 2.1 · 쇼퍼 포털 (`pro.html`) ◐ 🔒

- **목적:** 쇼퍼(데모 페르소나 소희)가 요청 수신→제안→정산→후기를 관리하는 대칭 포털.
- **섹션:** `dash`(대시보드·긴급/최근/후보) · `inbox`(신규/진행/완료/보낸제안) · `propose`(먼저 제안) · `profile`(view/edit) · `reviews` · `settle`(정산)
- **데이터:** R/W `fitting.pro.profile`(null=`DEFAULT_PROFILE` 데모) · R/W `fitting.pro.reqs`(시드 5건) · R/W `fitting.pro.proposed`
- **상태:** PROFILE null → 데모 기본값 · 요청 상태(신규/제안발송/수락됨/완료)가 `drawerAction` 결정 · `dir:'out'`=먼저 제안
- **이탈:** `pro-quote.html?req=i`(견적서) · `index.html`(로그아웃/고객화면)

### 1.3.7 · 견적서 (`pro-quote.html`) ◐ 🔒

- **목적:** 요청 클릭 시 고객 **체형진단 상세**(부위 스펙트럼) + 요청 내용 + 제안 처리.
- **데이터:** `?req=` 인덱스 · R `fitting.pro.reqs`·`fitting.pro.profile` · fetch `bodytypes.json`(`btResolve`) · W `fitting.pro.reqs`(수락/거절/완료/취소)
- **상태:** 요청 없음→"찾을 수 없어요" · `dir:'out'`=제안 대상(체형 placeholder) · 상태가 액션 버튼 결정

## 3. 운영 어드민 (3.A~3.E) — 프로토타입 ★ 🔒

> 목업 로그인([admin-login](../web/admin-login.html)) 뒤에서 도는 프로토타입. 데이터는 실파일(garments/body-*) 또는 **샘플**로 렌더, 상태 변경은 비영속. 각 JS 상단에 정본 레코드 계약 주석. 실 인증·저장은 마일스톤1(Supabase). 전 화면 검색 차단(robots + `X-Robots-Tag`).

**공통 셸:** 상단 `.anav`(5개 섹션 탭) · `admin-common.js`(세션 가드 `fitting.admin`) · 표=`table.dt` · KPI=`.kpi`.

### 3.A 사이즈·데이터 — 해자의 실행 도구

| 화면 | 목적 | 주요 섹션 | 데이터 소스 |
|---|---|---|---|
| **admin** (3.A) | 데이터 콘솔 | 탭① 브랜드 실측 커버리지·필터 · 탭② 신체 사이즈(회귀 r²/RMSE·분포·추정 미리보기) | `garments.json`·`body-base-model.json`·`body-distribution.json` |
| **admin-garments** (3.A.1) | 실측표 뷰어 | 제품별 사이즈×부위 cm 드릴다운 | `garments.json`(생성물) |
| **admin-collect** (3.A.1) | 이미지→CSV 수집 | 붙여넣기/드롭/파일 → 수동전사·OCR(베타)·AI 브릿지·CSV 가져오기 | 산출 CSV → build-sizespec.py |
| **admin-engine** (3.A.3) | 엔진 시뮬·튜닝 | 몸→사이즈 파이프라인 분해·목표여유 what-if·파라미터 | BodyModel·FitEngine |

- **3탭:** 회원(3.B.2, 진단이력 기준·정지/차단) · 전문가 심사(3.B.1, pro-signup 대기열 승인/반려) · **매칭 수요 신호(3.B.3)** — 페이크도어 `fitting.reqs` 집계(서비스/상황/예산 수요·오픈견적 응답률·오픈알림)
- **소스 토글:** 이 브라우저 `fitting.reqs` ↔ 샘플. 분쟁·정산·리뷰(3.B.4~6)는 v2.
