## 0. 저장소 키 계약 (전 화면 공통 · SSOT)

데이터 계약은 **두 모드**입니다(D-15 어댑터 스왑 · `web/js/data.js` seam). **proto**(로컬 데모)=브라우저 저장소 `fitting.*`. **api**(app/ 배포)=서버 라우트 `/api/*`→**Supabase(RLS)**. 아래 `fitting.*` 표는 proto 계약, api 계약은 그 아래 DB 표.

### 탭2 · 🔍 체형진단 — 서비스의 심장

진단은 **동의 게이트 없이** 바로 시작(서비스 제공 최소 수집). 위저드: 2-1 → 2-2 → 2-3 → 2-4 → (2-5 피드백/2-6 전환/2-7 공유).

#### 2-1 · 기본 정보 (`diag-basic.html`) ★ 🔒

- **목적:** 성별·나이·키·몸무게 수집. 성별 선택 시 평균값 프리필.

#### 2-2 · 착용경험 보정 (`diag-fit.html`) ★ 🟢실연결 🔒

> **입력 옵션 소스 = `size-catalog.json`**(cm-free 공개 카탈로그 — 브랜드·사이즈 라벨만, `gen-catalog`가 garments에서 실측치 제거해 생성). 실측 cm는 클라 미노출(해자). 부위별 `fits`(체형역산)·`painFlags`(미표기∩페인)·`openNote`(자유서술)를 수집 → 엔진 강화 원천(admin 3.C.4).

- **목적:** 정확도의 핵심. 대상 카테고리·선호핏 + 입어본 옷(브랜드·핏·사이즈)×1~3 + 부위별 느낌. **질문 정본은 엔진**([6_사이즈엔진.md](6_사이즈엔진.md)).

```
┌────────────────────────────────────────┐
│ 무엇을 진단할까요?                        │
│ [상의][하의] [아우터][치마][원피스]        │  카테고리
│ 선호핏  #prefseg (스키니…오버)            │
│ ── 입어본 옷 1 ──                         │
│ 브랜드[유니클로▾] 핏/품목[▾] 사이즈[M▾]   │  앵커(#brand1/item1/size1)
│ 부위별 느낌: 어깨[딱맞음] 가슴[여유]…      │  fits/painFlags/lengthPrefs
│ 자유메모 [………]         (+ 옷 더하기)     │
│              [ 진단하기 ]                │ → diag-loading
└────────────────────────────────────────┘
```

| 요소 | 입력 | 동작 |
|---|---|---|
| 진단 대상 | 카테고리 택1 | 기반(상·하의)=실측 역산 / 파생=선호핏만 |
| 선호핏 `#prefseg` | 세그먼트 | 카테고리별 `prefs`에 저장 |
| 앵커 옷 ×1~3 | 브랜드+핏+사이즈 | 브랜드는 앵커 6종 datalist |
| 부위별 느낌 | fits/pain/length | `SNUG/RELAXED…`, `OK/TIGHT`, `GOOD/LONG…` |
| 하의 밴드 | 토글 | `waistband:'none'|'banded'` |
| [진단하기] | 주 버튼 | **R `fitting.basic`, R/W `fitting.dx`** → diag-loading |

- **데이터:** R `fitting.basic`(성별→사이즈/브랜드 렌더) · R/W `fitting.dx` `{basic, prefs, experiences[]}`
- **상태:** 0벌=건너뛰기(신뢰도↓) / 1벌=보통 안내 / 브랜드 없음=직접입력 / 파생 카테고리=선호핏만
- **experiences[] shape:** `{category, brandId, brandName, fitLine, item, sizeLabel, subtype, gender, silhouette?, waistband?, fits{}, painFlags{}, lengthPrefs{}, openNote}`

#### 2-3 · 진단 로딩 (`diag-loading.html`) ★ 🔒

- **이탈:** result.html

#### 2-4 · 결과 카드 (`result.html`) ★ 🟢실연결 🔒

> **api(배포):** 추천 계산을 **서버(`/api/diagnose`)에서** — garment 테이블 실측표로 역산·추천(garments 클라 미노출=해자), 브랜드 노출순서 반영, `diagnosis` 저장 + 피드백 `/api/feedback`(킬메트릭)→Supabase. 인증 표면 제거(계정 저장 숨김, '스타일리스트 찾기'만). **proto:** 클라 엔진(garments.json). `diagnoseSpecs()`가 모드 분기.

- **목적:** 제품의 얼굴. ① 8유형 체형 프로필 ② 핏 인사이트 ③ 브랜드별 추천 사이즈 ④ 옷장(진단 현황). 상의만 실계산, 파생은 선호핏 폴백/게이트.

```
┌───────────────────────┬───────────────────┐
│  8유형 카드(어두운 배경) │  브랜드별 추천 사이즈 │
│   실루엣 + 시그니처색   │  핏지수% · 병목부위  │
│   핏 인사이트           │  체형 측정 스펙트럼  │
├───────────────────────┴───────────────────┤
│ 이 추천 맞아요?  ✅맞아요 ➖애매 ❌틀려요    │ ← 정확도 검증(킬 메트릭)
│ ☐ 익명 엔진개선 동의(만14세)                │
│ [옷장: 상의✓ 하의○]  [전문가에게 받기 →]    │
└─────────────────────────────────────────────┘
```

| 요소 | 표시 | 동작 |
|---|---|---|
| 8유형 카드 | `bodytypes.json`(성별 축) | `btResolve(t,g)` |
| 브랜드 추천표 | 엔진 계산(FitEngine) | 핏스코어·병목 |
| **정확도 검증** `.rseg` | 맞음/애매/틀림 | `fb()` → **W `fitting.feedback`**(local) = 킬 메트릭 원천 |
| 엔진개선 동의 `#eiConsent` | 체크박스(선택) | **W `fitting.consent`** `{engineImprove,ageAttested,at}` |
| 전문가 전환 | CTA | → `index.html#shop` |
| 재시작 `resetDiag()` | — | `fitting.dx/basic/consent`(session)+`feedback`(local) 삭제 |

- **데이터:** R `fitting.dx`·`fitting.basic`·`fitting.consent` / W `fitting.consent`·`fitting.feedback` / fetch `bodytypes.json`
- **상태(신뢰도 3단계):** 0벌=낮음(주의 배너)·회귀 추정 / 1벌=보통 / 2~3벌=정확 · 데이터 없음→`renderNoData()` · 로드실패→`renderLoadError()` · 8유형 상세는 상·하의 완료(cardReady) 시에만
- **이탈:** `index.html#shop`(전문가) · `index.html`(저장) · card(iframe) · diag-fit(재진단)

## 3. 운영 어드민 — MVP 실배선 ★ 🔒

> **Google OAuth 로그인**([admin-login](../web/admin-login.html), Supabase Auth + `is_admin`=admin_user 멤버십) 뒤에서 도는 **실배선**. 데이터는 **Supabase(RLS admin 전용)**. 로그아웃은 Supabase 세션까지 종료. 전 화면 검색 차단(robots + `X-Robots-Tag`).
>
> **MVP 메뉴 = 4개**: `사이즈·데이터` · `진단·정확도` · `엔진 강화` · `전문가 수요`. **v2(3.B 회원·거래 / 3.C.1 사업현황 / 3.D B2B API / 3.E 내부운영)는 `admin-common.js`가 네비 숨김 + `gen-app` 미배포** — 화면은 구현돼 있으나 MVP 밖. 측정 2축(진단 정확도=킬메트릭 / 매칭 수요)을 보고 엔진 데이터를 직접 관리한다.

**공통 셸:** 상단 `.anav`(MVP 4메뉴 — admin-common.js가 v2 숨김·진단정확도/엔진강화/전문가수요 주입) · `admin-common.js`(세션 가드 `fitting.admin` + 네비 스코프 + 로그아웃) · `admin-auth.js`(Supabase 읽기/쓰기 헬퍼) · 표=`table.dt` · KPI=`.kpi`.

### 3.A 사이즈·데이터 — 해자의 실행 도구 · **DB 실배선** (MVP)

| 화면 | 목적 | 주요 섹션 | 데이터 소스 |

- **정확도×신뢰도 캘리브레이션 히트맵**(high-confidence 틀림=튜닝 1순위) · 8유형/앵커브랜드/페인 분해 · 응답 로그 · **정확도 추세**(날짜별·엔진버전별 맞음율 → 엔진이 강해지는지 확인).
- **소스:** 실 DB(Supabase `feedback`+`diagnosis` 임베드·RLS admin) ↔ 로컬 `fitting.feedback` ↔ 샘플 토글. 킬 메트릭(맞음율 ≥65%) 실측.

### 3.C.3 전문가 수요(페이크도어) — `admin-leads` ★ MVP

- 스타일리스트 매칭 수요(웨이트리스트 오픈알림 이메일·견적요청) 집계. KPI(총 수요·오픈알림·견적요청·**이메일 확보**) · **진단→수요 전환**(`lead.session_id`↔`diagnosis.session_id` 조인) · 서비스·상황별 · 수요 목록(이메일·유입 진단후/직접).
- **소스:** `lead` 테이블(db/02·RLS admin). 페이크도어 측정축.

### 3.C.4 엔진 강화(성장구조) — `admin-improve` ★ MVP

- 진단 로그(`diagnosis.input` painFlags·openNote + `feedback`)를 **개선 작업목록**으로(docs/6 §1.1·자동학습 아님·사람 액션 우선순위): ① **미표기∩페인**(브랜드×병목부위 끼임)→A축 역산 실측보정 ② **새 페인**(자유서술)→PAIN_FLAGS 승격 ③ **확신 틀림**→규칙 튜닝 ④ **수요 대비 데이터 부족**→수집. 각 행이 실측표·수집으로 연결.
- **소스:** `ADMINAUTH.diagnoses/feedbackJoin/garments`(RLS admin).

### (v2 · MVP 미노출) 3.B 회원·거래 / 3.C.1 사업현황 / 3.D B2B API / 3.E 내부운영
