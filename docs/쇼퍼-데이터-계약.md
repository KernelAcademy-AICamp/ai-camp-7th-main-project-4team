# 쇼퍼 데이터 계약 — 쇼퍼찾기 ↔ 쇼퍼 지원 포털

> 사용자앱 **쇼퍼찾기**(`web/js/index.js`의 `EX`)와 **쇼퍼 지원 포털**(`web/js/pro*.js`의 `fitting.pro.profile`)의
> 쇼퍼 데이터를 통일하기 위한 합의. 담당: 디자이너(쇼퍼찾기) · 개발자(포털). **이 문서가 정본.**

## 배경
포털에서 쇼퍼가 입력한 프로필이 쇼퍼찾기의 카드·상세·필터로 그대로 흘러가야 한다.
현재 두 쪽의 필드명·코드값이 어긋나 있어 아래처럼 맞춘다.

---

## ✅ 확정 사항

### 1. 서비스 코드 = `image` 로 통일
- 3종: `online`(온라인 스타일링) · `shopping`(동행 쇼핑) · `image`(이미지 컨설팅)
- 포털의 `imaging` → **`image`** 로 변경.

### 2. 서비스는 다중(`services[]`) + 서비스별 속성을 쇼퍼에게 입력받음
쇼퍼가 여러 서비스를 등록할 수 있고, 서비스마다 아래 값을 받는다.

| 서비스 | 제공 방식 | 활동 지역 |
|---|---|---|
| 온라인 스타일링 | **비대면**(타입 고정) | — |
| 동행 쇼핑 | **대면**(타입 고정) | 쇼퍼 **선택** |
| 이미지 컨설팅 | **대면**(타입 고정) | 쇼퍼 **선택** |

- 구조: `services: [{ type, price, mode, regions? }]`
  - `mode` = 타입 고정(온라인→**비대면** / 동행·이미지→**대면**)
  - `regions` = **대면 서비스**(동행·이미지)만
  - **예상 기간(`dur`)은 제거** — 받지 않기로 확정.

### 3. 상황 = 전문분야 (통일) · 7가지 고정
쇼퍼 편집(포털)에선 **"전문분야"**, 고객 화면(쇼퍼찾기)에선 **"상황"** 으로 부르되 **같은 필드**를 쓴다.

| 코드 | 라벨 |
|---|---|
| `date` | 소개팅·데이트 |
| `interview` | 면접·발표 |
| `wedding` | 결혼식 하객 |
| `travel` | 여행 |
| `daily` | 데일리 스타일링 |
| `personal` | 퍼스널 스타일링 |
| `bodycover` | 체형 커버 스타일링 |

### 4. 스타일 (별도) · 프리셋 8가지
상황(전문분야)과 **별개로 "스타일" 태그**를 둔다. **아래 8가지 프리셋 중 선택**:

`캐주얼` · `미니멀` · `시크` · `클래식` · `스트리트` · `빈티지` · `스포티` · `걸리시`

### 5. 소개(bio)
- 포털이 **경력 소개를 빼고 소개(bio) 하나만** 남김 → 그대로 쇼퍼찾기 `bio` 로 사용.

### 6. 포트폴리오
- 포털 구조로 통일: **사진 + 착용 모델 키·몸무게(cm·kg)**.

---

## 필드 매핑 (포털 → 쇼퍼찾기)

| 포털 `fitting.pro.profile` | 쇼퍼찾기 `EX` | 상태 |
|---|---|---|
| `name` | `nm` | ✅ |
| `services[] {type, price}` | `services[] {type, price, mode, regions?}` | ✅ 구조 통일(기간 제거·대면 지역) |
| 전문분야(7가지) | `occ[]` = 상황(7가지) | ✅ 통일(라벨만 다름) |
| 스타일 태그(프리셋 8) | `tags[]` (스타일) | ✅ 확정 |
| `bio` | `bio` | ✅ (경력 소개 제거) |
| `portfolio[] {src, height, weight}` | 상세 갤러리(사진 + cm·kg) | ✅ cm·kg 표기 |
| `regions[]` | 대면 서비스(동행·이미지) 속성으로 | ✅ 서비스별 속성 |
| — | `rating` · `review` · `match` | 파생값(입력 아님) |

---

## 반영 담당
- **포털(개발자)**: `imaging`→`image` · `services` 다중(대면 서비스에 `regions`, **기간 제거**) · 전문분야=상황 7가지 · 스타일 태그 · 경력 소개 제거
- **쇼퍼찾기(디자이너)**: `EX`를 `services[]`로 전환 · 목록·필터·상세·요청 화면 개편 · 상황 7가지 · 제공방식(비대면/대면)·활동지역 · 포트폴리오 cm·kg ✅ 반영 완료

> 참고 · 화면 변화 예시: 다중 서비스 도입 시 목록/필터/상세/요청 4화면이 어떻게 바뀌는지는 별도 목업으로 공유.
