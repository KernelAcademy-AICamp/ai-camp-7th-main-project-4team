# 핏팅 — 사용자 플로우 (User Flow)

> **이 문서는 무엇?** 사용자가 화면 사이를 **어떤 순서로, 어디서 갈라지고, 어디서 되돌아오는지**를 그린 흐름도입니다. 화면 목록·URL은 [2_정보구조.md](2_정보구조.md), 화면별 상세 요소·상태는 [4_화면정의.md](4_화면정의.md), 용어는 [0_용어집.md](0_용어집.md).

---

## 1. 흐름을 짤 때 지키는 3원칙

세 원칙 모두 "사용자를 중간에 놓치지 않으려는" 목적에서 나옵니다.

1. **가입은 나중에 (lazy 가입)** — 비회원 그대로 진단~결과~피드백까지 다 하게 두면 이탈이 줄고 측정 표본도 커집니다. 가입은 딱 **두 순간**에만 요청: ① 결과를 저장할 때(2-4), ② 전문가 매칭을 실제로 쓸 때(3-1). (상세 §3-2, 정책은 [5_정책·데이터.md](5_정책·데이터.md))
2. **진단은 동의 없이 시작** — 진단은 서비스 제공에 필요한 최소 수집이라 **진입 동의 게이트 없이** 시작합니다(구 G.5 필수 게이트 폐기). 대신 **엔진 개선 목적의 데이터 활용만** 결과화면(2-5)에서 **opt-in**으로 받고, 이때 만 14세 확인을 접목합니다.
3. **막다른 길 금지** — 실패하거나 비어 있는 화면에도 항상 되돌아갈 길을 둡니다(추정 실패 → 옷 추가, 신뢰도 낮음 → 재진단, 링크 만료 → 홈).

## 2. 메인 흐름 (MVP)

> MVP에서 실제로 만드는 화면(★)만 이어 그린 "가장 잘 풀리는 경로 + 갈림길". ○(껍데기)는 도착점으로만 표시.

```flow
legend | ac 체형진단(MVP 핵심) | shop 쇼퍼찾기(페이크도어) | warn 가입 트리거 | green 킬 메트릭
phase | ink | 🏠 탭1 홈 | — 입구
node |  | 홈 랜딩 | ★ | "이게 뭐 하는 서비스" 3초 전달 + [내 사이즈 진단 시작] CTA · 상황별 쇼퍼 리스트(오픈예정/1기 모집) · 작동 3스텝
conn | 진단 시작 (동의 게이트 없음)
phase | ac | 🔍 탭2 체형진단 | — 무료 퍼널 · 제품의 얼굴
node | 2-1 | 기본 정보 입력 | ★ | 성별 · 나이 · 키 · 몸무게 (성별 선택 시 평균값 미리 채움 · 선호핏은 2-2에서)
node | 2-2 | 착용경험 보정 | ★ | 옷 종류(상의/하의)·선호핏 · 입어본 옷 [브랜드+핏+사이즈] ×1~3벌 · 부위별 느낌(큼/여유/딱맞음/끼임) | @★ 체형 역산 정확도의 핵심 입력 · 기반=상의+하의 / 파생(아우터·원피스·치마)=선호핏만
loop | [state] 입어본 옷 0벌 → 건너뛰기(신뢰도 낮음) · 브랜드 목록에 없음 ◇ → 직접 입력
conn | 제출
node | 2-3 | 진단 로딩 | ★ | 엔진 처리 — 체형 역산 + 브랜드 사이즈 매칭 + AI 설명 문장
conn | ◇ 엔진 판정
split | 확신 못 함 · 신뢰도 낮음 | 2-4 [상태] 추정 실패 — "옷을 더 추가해 주세요" | ⟶ 2-2 착용경험으로 되돌아감 :: 잘 나옴 | 결과 카드 표시 ↓
hero | 2-4 | 결과 카드 | ★ 제품의 얼굴 | ① 체형 프로필(8유형 캐릭터) ② 핏 인사이트 ③ 브랜드별 추천 사이즈 · 옷장 · 신뢰도 3단계(0벌 낮음/1벌 보통/2~3벌 정확) · 비회원=URL 임시보존 → 가입 시 영구
loop | 재진단(결과 불만족) · 옷장 → 이어서 다음 종류 진단 → 2-2로 되돌아감 (기본정보·체형 재사용)
conn | 결과 후 분기
branch | 📝 | 2-5 피드백 | ★ | "실제 내 사이즈와 맞나요?" 동의/비동의 · "안 사본 브랜드도 사보겠어요?" 의향 · 엔진개선 opt-in(만14세 확인) | metric:킬 메트릭 · 사이즈 정확도 ≥ 65%
branch | 🔖 | 결과 저장 | ★ | 결과를 영구 보존하려면 → 가입 모달(트리거 ①) | 🔒 lazy 가입 트리거 ① → 마이 · 저장 안 해도 2-5로 계속
branch | 📱 | 2-7 결과 공유 | 후속 | 카드 이미지 / 링크 공유 — 입소문 유도 | v2 (여력 시)
branch | 👔 | 2-6 전문가 전환 | ★ | "스타일까지 전문가에게 받기" — 무료→유료 퍼널의 다리 | lead | arrow:↓ 탭3 쇼퍼찾기로 이동
conn | 2-6 누름
phase | shop | 👔 탭3 쇼퍼찾기 | — 전문가 매칭 · 페이크도어
node | 3-1 | 매칭 진입 (페이크도어) | ★ | 관심 서비스 선택(온라인 스타일링 / 동행 쇼핑 / 이미지 컨설팅) · 상황 선택 · 클릭 기록 · 오픈 알림 신청 + 1기 모집 | color:shop | metric:전환신호 · 유형별 클릭·알림신청 ≥ 25%
conn | 알림 신청 시
gate | 가입 게이트 | 실제 이용(알림신청/요청) 시점에 모달로 진입 | lazy 가입 트리거 ②
node | 3-2 | 알림 완료 / 준비중 | ★ | "오픈하면 알려드릴게요" 확인 — 여기가 MVP 흐름의 끝(도착) | color:shop
```

## 3. 화면별 진입/이탈 · 가입 시점

### 3-1. "들어오는 조건 → 나가는 곳" 표

| 화면 | 언제 들어오나 | 갈림길 | 어디로 가나 |
|---|---|---|---|
| 탭1 홈 | 첫 방문 / 링크 만료 후 복귀 | 진단 시작 | 2-1 |
| 2-1 기본정보 | 홈에서 진단 시작 | 입력 완료 | 2-2 |
| 2-2 착용경험 | 기본정보 완료 / 재진단·옷추가로 복귀 | 제출 / 옷 없음◇ | 2-3 / (직접입력 분기) |
| 2-3 로딩 | 착용경험 제출 | 엔진 판정◇ | 2-4 결과 / 추정 실패 |
| 2-4 결과카드 | 잘 나옴 | 저장◇① · 재진단 · 공유 · 다음 | 가입창 / 2-2 / 2-7 / 2-5 |
| 2-4 [추정 실패] | 엔진 신뢰도 낮음 | 옷 추가 | 2-2 |
| 2-5 피드백 | 결과 확인 후 | 응답 완료 | 2-6 |
| 2-6 전문가 전환 | 피드백 후(또는 결과에서 바로) | 버튼 누름 | 3-1 |
| 3-1 페이크도어 | 전환 버튼 | 유형 선택·알림신청 / 실이용◇② | 3-2 / 가입창 |
| 3-2 알림 완료 | 알림 신청 | — | **끝(도착)** |

### 3-2. 가입은 정확히 언제 요구하나

본선(핵심 흐름)은 **비회원 그대로** 끝까지 갑니다. 가입은 아래 두 지점에서만 *옆으로* 잠깐 갈라집니다.

| 가입 지점 | 비회원은 | 가입하면 | 참고 |
|---|---|---|---|
| **① 결과 저장** (2-4) | 결과를 임시 링크로 잠깐 보관(만료 있음) | "결과 저장하기" → 결과가 계정에 **영구 보관** | 저장 안 해도 흐름은 2-5로 계속(가입은 선택) |
| **② 전문가 매칭 실제 이용** (3-1) | 구경·유형 선택까지는 가능 | "요청/알림신청"을 누르면 **가입 관문** → 3-2 도착 | MVP는 "알림신청"까지가 페이크도어, 실제 매칭은 v2 |

> 매칭 쪽(전문가 앱·운영자)의 흐름은 [7_전문가매칭.md](7_전문가매칭.md) §2~4.
